高吞吐DAG-BFT系统中的因果证据热路径
本文针对高吞吐DAG-BFT系统（如Narwhal/Tusk、Bullshark、DAG-Rider、Shoal、Mysticeti、Cordial Miners等）中**热路径(causal evidence hot-path)**的存储问题进行深度调研。我们重点考察这些系统在区块/顶点、头、证书、父引用、交易批（批次）等证据对象上的管理方式，以及它们如何建立前向/后向引用、数据物化假设、修复与消费路径、保留/垃圾回收策略等。下文结合论文和实现文档，详细分析每种系统的设计特点和缺陷，并通过对比矩阵总结异同，最后指出最关键的研究空白。
Narwhal & Tusk
对象(证据)：Narwhal将交易打包为Batch，主节点构造包含交易摘要和父节点证书指针的Header；收集到2f+1个验证者签名后，产生对应的Certificate。每个**顶点(Vertex)**实际上是经认证的Header（含证书）。这形成一个有向无环图(DAG)，用于共识排序。
前向引用：每个Header显式引用其父轮次（round）中所有“强边”(≥2f+1个)对应的父顶点证书，以及最多f个“弱边”引用前几轮无法通过强链到达的顶点；此外，Header还包含指向工作节点提交的交易批(Batch)的摘要。Certificate可以视作对Header的聚合签名，也隐含对Header及其引用的认可。
后向引用：实现中通常不显式维护后向指针。Narwhal的存储接口只提供GetHeader、GetCertificate、GetBatch等前向访问方法，不提供倒序指针。如果需要查找某块被哪些后续块引用，只能通过遍历DAG结构或索引来间接推断，没做持久化优化。
数据物化假设：Narwhal允许Header+证书进入共识流程，而对应的完整交易批体(区块体)可能尚未到达。例如，主节点广播Header时，仅包含交易摘要，不包含完整交易；其他节点需要验证数据可用性再投票。如果节点收到Header但缺少Batch体，系统通过网络层的FetchBatch接口主动拉取缺失数据。因此，摘要先行，完整数据后补，是Narwhal的常见模式。
缺失修复路径：Narwhal实现了一个“fetcher”组件：任何节点若收到Header却没本地Batch，就调用FetchBatch(from,digest)向其他节点请求完整交易。这相当于基于DAG拓扑的按需数据可用修复机制。
共识后消费路径：在Tusk共识（Narwhal上的零开销共识）达成后，Narwhal通过调用MarkCommitted(certs)标记已提交的证书顶点，然后用GetTransactions(certs)获取对应交易提交给执行层。也就是说，在证书提交后，Narwhal会将其关联的交易批体交由执行。共识协议本身并不负责事务执行，它只提供已认证的顶点列表，执行由上层驱动。
持久化与GC策略：Narwhal的存储接口支持持久化，但保留策略较为简单：以“轮次”为单位进行垃圾回收。例如实现中有配置WithGCInterval，默认每50轮清理一次老数据；底层存储也提供DeleteBeforeRound(round)方法。这意味着Narwhal倾向于按轮次滚动清除所有早于某轮的数据，而不区分谁已经消费或依赖了这些数据。简言之，Narwhal采用基于轮次的扁平删除，而不考虑具体因果关系。
消费/档案状态跟踪：Narwhal本身不维护关于证据是否被修复、执行或归档的元数据。它只知道哪些证书已提交(通过MarkCommitted调用)，并假定执行层随后处理交易。但并未记录交易是否真正执行、是否被包含在快照/归档中或是否作为后续恢复的依据保留。因此，从协议设计角度看，Narwhal对证据的生命周期知之甚少，仅靠简单GC可能导致关键数据过早丢弃，出现执行缺失或检查点覆盖不全的风险。
Bullshark
对象(证据)：BullShark是建立在Narwhal之上的共识层协议，将Narwhal的Header视为顶点(Vertex)。每个顶点结构包括所属轮次(round)、创建者(source)、交易块(block)、以及引用边集。这与Narwhal的Header类似，但废除了外部证书概念：顶点的“已认证性”由至少2f+1个验证者间接保证（即“强边”）。
前向引用：正向引用关系与DAG-Rider相同：每个顶点至少包含2f+1条指向上一个轮次顶点的“强边”(主父引用)和最多f条指向更早轮次顶点的“弱边”，以确保DAG连通性。这些引用就构成了DAG的有向边：强边用于确定共识领导块，弱边用于遗留依赖。此外，每个顶点还携带一个完整的交易区块(block)，包含具体交易数据。
后向引用：BullShark不显式存储反向依赖指针。顶点间的DAG连接只通过前向引用隐式维护，不保留子依赖列表。顶点被插入后，如果后续顶点引用它，这些信息存储在后续顶点的前向引用字段中，而不会反向索引到前驱顶点。
数据物化假设：BullShark基于可靠广播(RBC)构建DAG：每个顶点在创建时都通过可靠广播机制传播。因此，顶点的内容（交易区块）在广播过程中就已经被传送给所有正确节点，不需要事后补充。也就是说，顶点本体和摘要是一起传播的，节点不会见到缺少区块体的顶点。如果RBC丢包，RBC协议会保障最终送达。因此，BullShark假设任意进入DAG的顶点都已完整物化。
缺失修复路径：由于使用RBC，节点通常无需单独拉取缺失区块；RBC协议自身在一定轮次后保证每位正确节点都接收到相同的顶点内容（即至少有2f+1个签名的顶点体）。BullShark文档并未描述额外的数据拉取逻辑，因为顶点本体在初始广播时即完成同步。
共识后消费路径：BullShark仍然是DAG-DRB(BAB)协议，提交路径通过本地重构来完成：一旦协议达到共识（即按照协议的提交规则确定领导块），各节点本地遍历DAG以确定最终的提交序列。提交后的顶点块（即交易）可直接交由执行层。BullShark自身并无显式的“交付”调用，它假定一旦领导块确定，相关交易即可安全执行。
持久化与GC策略：BullShark的论文明确提出了加时间戳的垃圾收集机制以保持内存有界。具体来说，每个顶点包含一个广播时间戳v.ts，各节点维护一个垃圾收集轮次GCround。只有当一个轮次中所有顶点的中位时间戳超过阈值（如3Δ）时，该轮及更早轮次才会被垃圾回收。这一做法确保在全局稳定之后才丢弃老顶点，以保证所谓的同步公平性。然而，在无界异步执行前（GST之前），系统仍可能丢弃部分内容，BullShark仅保证GST后公平性。早期的BullShark实现首先尝试借鉴Narwhal的轮次共识GC方法，但发现会导致低吞吐者的区块被过早清理，从而牺牲了公平性。因此BullShark最终引入了上文的时间戳方案以平衡公平性和GC需求。
消费/档案状态跟踪：BullShark协议本身并未显式跟踪“证据是否执行或归档”状态。它的垃圾回收依赖于时间戳和轮次，而不是是否消耗或快照。从执行角度看，一旦顶点被提交，它就可以丢弃原有的等待信息；但BullShark并未在协议中为执行完成、已归档等事件记录任何标志。这意味着，即便某些顶点已被提交并执行完毕，协议并不在元数据中标记这些顶点，以此决定是否清理。这种做法与普通区块链的按照高度归档（checkpoint）不同，不考虑复杂的因果依赖，只靠时间条件来收集。
DAG-Rider
对象(证据)：DAG-Rider是一个理论上的异步BAB协议，节点间广播“消息”(可视作区块)并构建一个结构化DAG。每条消息即为顶点，包含交易内容和对前几轮消息的引用。顶点没有额外的证书或签名证据，而是通过可靠广播集成。
前向引用：DAG-Rider顶点采用类似Narwhal/BullShark的强/弱边结构：每个顶点引用至少2f+1条指向上一轮顶点的边（强边），以及最多f条指向更早轮次的边（弱边）。这些引用形成DAG的边关系，用于后续的提交规则。
后向引用：同样地，协议没有显式的后向指针结构。顶点之间的关系只通过前向引用建立，子节点信息需在DAG中向前搜索获得。
数据物化假设：DAG-Rider使用**可靠广播(RBC)**作为构建DAG的传播机制：每个顶点从发布者到达所有节点时即包含完整内容，协议设计假设正确节点最终接收到顶点及其内容。因此，顶点在逻辑上进入共识前已完成广播，节点无需像Narwhal那样后续拉取。RBC的多轮传播保证了这一点（实际执行时，DAG-Rider预期6轮RBC来生成一个提交结果）。
缺失修复路径：RBC提供可靠性，理论上保证了最终交付；协议设计并未引入额外的修复步骤。若节点暂时未收到某顶点，RBC协议会在内部多轮交换（如Bracha协议）后完成投递。
共识后消费路径：DAG-Rider的提交规则是本地化的：一旦DAG构建完毕，所有节点根据DAG拓扑和共识规则各自确定一个全序输出，将顶点逐一交付。协议不需要专门的执行调用，节点自己维护的DAG即视为“已提交区块链”。提交后，顶点和交易可以交由应用执行。
持久化与GC策略：DAG-Rider是学术协议，没有具体实现指导持久化或垃圾收集。理论上，为保证安全性（即总有办法提交所有正确顶点），它使用弱边(weak links)引用至所有未提交的早期顶点，从而保证即使丢弃局部数据，仍有途径保留访问这些顶点。但这本质上要求保留整张DAG而不做削减，否则安全性受损。文献没有给出实际的清理机制，只说明RBC本身成本高昂（需要多次通信）。因此，DAG-Rider依赖于无限保存DAG历史，无垃圾回收，留待后续理论扩展。
消费/档案状态跟踪：协议不维护任何“已提交”、“已归档”或“已执行”的元数据。节点只通过DAG本地结构确定提交顺序，不对数据消耗做标记。由于没有拆分共识与数据可用层，不存在“修复后消耗”这一概念。简言之，DAG-Rider完全没有引入与证据生命周期管理相关的机制。
Shoal
Shoal是一种**协议无关(agnostic)**的优化框架，旨在改进基于Narwhal的共识（如Bullshark）的延迟和健壮性。它引入领导者信誉(leader reputation)和流水线支持，以减少时间戳和等待。
对象(证据)：Shoal本身使用底层共识的证据对象——基本上与Narwhal/Bullshark相同，即顶点(Vertex)/区块(Block)、头、交易批次等。它并不增加新的证据类型。
前向/后向引用：Shoal保留了底层DAG协议的引用结构（如Bullshark中的强/弱边）。它引入领导者信誉度指标，但这只是额外的状态信息，不改变区块内的链指引用方式。Shoal的论文主要关注并行领导块和跳过超时的机制，没有详细描述额外的引用或元数据管理。
数据物化假设：Shoal使用Narwhal/Bullshark的数据可用层，因此假设与Narwhal相同：区块先广播其摘要，再补足交易数据，或者像Bullshark那样通过可靠广播立即传播完整数据。Shoal不改变底层的物化流程，只在收到节点超时之前优化领导者决策。
缺失修复路径：依赖Narwhal/Bullshark的现有机制（如Narwhal的FetchBatch或Bullshark的RBC），Shoal没有引入新的修复逻辑。
共识后消费路径：Shoal保留原协议的消费方式——例如在Shoal+Bullshark组合中，提交的顶点按照Bullshark的提交规则交付，无需额外步骤。
持久化与GC策略：Shoal论文并未提及任何额外的GC或存储策略；它沿用底层共识的策略（如Bullshark的时间戳GC或Narwhal的轮次GC）。基于已有文档，没有证据表明Shoal提供新的持久化元数据或垃圾回收技术。
消费状态跟踪：Shoal也未介绍任何执行/归档跟踪。它只关注减少共识延迟和超时次数，对证据存储生命周期的管理没有特殊支持。
Mysticeti
Mysticeti是一个面向部分同步网络的DAG共识协议系列，包括Mysticeti-C（只使用共识路径）和Mysticeti-FPC（带有快速路径和检查点机制）。这两个协议都使用“签名区块”作为单一消息类型，每个区块自包含交易数据和签名。
对象(证据)：Mysticeti协议中的核心证据是区块(Block)，每个区块包含交易批和提议者签名。协议中不使用像Narwhal那样的多签证书，每个区块的单签名即为验证其合法性的证据。多个区块按协议被序列化输出。
前向引用：与其它DAG协议类似，每个区块可能包含指向先前区块的引用，以保证DAG连通性（文中未详细列出，但显然存在）。这些引用形成了DAG拓扑供共识使用。与Narwhal等不同，Mysticeti不需要区块引用证书，区块自身就是最小的提交单位。
后向引用：协议未提及持久化后向引用；所有依赖关系通过区块中的前向边隐式建立。
数据物化假设：每个区块在广播时都携带完整交易和签名信息。Mysticeti使用可靠广播机制向所有节点分发区块，因此区块体在到达时即为完整数据。没有单独的数据可用层或后续取回路径；一旦区块进入协议流程，它的所有数据就已经物化。
缺失修复路径：若节点在广播时未收到区块内容，可靠广播协议将处理缺失。没有额外的节点间补取逻辑需要说明。
共识后消费路径：Mysticeti-C的提交规则设计为“每个区块独立提交”，无需等待一波完成即可提交。也就是说，只要协议条件满足（例如领导块被确认），节点就将区块提交给应用层。Mysticeti-FPC扩展了这一点：它对不需要共识的快速交易路径也采用可靠广播，快速路径交易一经收到即可在本地提交；需要共识的交易走共识路径，作为普通区块提交。同时，每次共识提交都被视为检查点(Checkpoint)。与许多其他协议需要额外的检查点子协议不同，Mysticeti将已提交的区块集直接作为检查点来使用。因此，协议本身通过“区块提交即检查点”的方式简化了快照和存档流程。
持久化与GC策略：论文并没有专门讨论数据持久化或GC。逻辑上，由于每个区块经提交后就成为区块链的一部分，以后只需依赖已完成的检查点继续执行，理论上可以安全丢弃更早的中间数据。Mysticeti中，检查点由已提交区块组成；一旦一个检查点明确，位于该检查点之前的区块可视为可移除。实际上，文档指出共识提交本身就充当了检查点，不需要额外收集步骤。这为GC提供了线索：可按检查点批量删除旧数据，但具体实现细节留待实践。
消费/档案状态跟踪：Mysticeti通过使用单一的签名区块，简化了管理：每次共识提交立即置为最终，不需要二次证明或归档。协议通过检查点机制推断哪些交易已完成：所有已提交的区块即认为“已归档”。对执行和快照的状态跟踪由协议的检查点定义隐式完成，因此协议不需要额外元数据来记录“已修复/已执行”的状态。
Cordial Miners
Cordial Miners提出了一套区块链的偏序化形式——Blocklace（区块花环），并在此上完全避免使用可靠广播的共识协议。每个区块包含交易和指向先前区块的多个哈希指针。
对象(证据)：核心对象是Block (区块)，和其他协议类似，每个区块携带一批交易(payload)及指向前代区块的指针集。Blocklace结构没有外部证书：区块本身连同其指针即构成协议的状态机达成证据。
前向引用：每个区块在Blocklace中指向它所“观察到”的先前区块集合。例如，协议规定在轮次$r$中，节点只能引用轮次$r-1$的所有区块（若该轮成为“友好轮(cordial)”），以及必要的之前轮次的区块来保持连通。这种指针集定义了DAG的边。由于没有可靠广播，所谓“友好轮”指的是有超过2f+1个区块的轮次（超过简单多数）。正确节点会等到至少有2f+1个区块(超多数)才出下一个块。
后向引用：Cordial Miners同样不存储子依赖列表。后向关系隐含在每个区块的前向指针内；系统不维护倒序索引。
数据物化假设：协议假设每个区块通过**最佳努力广播(best-effort broadcast)**机制传播。也就是说，区块被广播出去时已包含完整交易；如果有丢失，其他区块的指针会间接承诺所遗漏的区块。Cordial Miners放弃可靠广播，而是利用区块互相“可达性”的结构：一个新区块所指向的前序区块集代表了其创建者所知的全部先前信息。因此，理论上广播即完成数据传播，缺失内容通过后续区块的指针自然填补（以区块相互指向的形式）。
缺失修复路径：由于省略了可靠广播，若个别节点未能接收到某个区块，其信息仍可通过其他区块的指针路径间接获知。协议设计要求，每个新区块都“引用”它目前已见到的所有Tip区块，因此缺失的区块会在后续轮次被引用确认。这相当于用区块作为“ACK/NACK”消息，隐式指出节点对哪些区块已有了解。因此，不需要额外的显式拉取步骤。
共识后消费路径：Cordial Miners的共识最终通过本地排序实现：随着DAG不断增长，节点周期性地根据Blocklace定义的“波浪(wave)”和领袖块(fast leader block)规则，将看到的区块拓扑转化为最终序列。简言之，一旦节点观察到某轮的领导区块满足条件，节点即可确定该波浪内所有关联区块的提交顺序，提交这些区块所含交易。提交后，无需再额外调用；节点按照拓扑序将交易交付。
持久化与GC策略：论文未讨论明确的垃圾回收：实际上，Cordial Miners的核心优势在于无需可靠广播而非持久化管理。Blocklace中的所有区块理论上都需保留，直到被本地排序输出为止。没有指定当区块可安全删除的检查点或epoch，因此实现上若要持久化，需要自行定义何时丢弃早期区块（例如可以在区块被所有节点确认后删除）。协议本身假定区块被不断引用，没给出自动GC规则。
消费/档案状态跟踪：由于协议中“区块提交就是拓扑排序输出”，没有额外机制记录哪些区块已执行或已归档。节点只在内存中建立Blocklace并按需排序，执行完成后区块可以丢弃，但这需要实现者自行管理。协议未指定持久化元数据，也未区分“已用于排序”的块和未使用块。

The Lifecycle of Causal Evidence in High-Throughput DAG-BFT Systems: A Hot-Path Storage Analysis
Introduction and Critique of the Causal Evidence Hypothesis
The evolution of Byzantine Fault Tolerant (BFT) consensus mechanisms has fundamentally shifted toward Directed Acyclic Graph (DAG) architectures. Systems such as Narwhal, Tusk, Bullshark, Shoal, Mysticeti, and Cordial Miners have successfully circumvented the performance bottlenecks of traditional leader-centric protocols (such as Practical Byzantine Fault Tolerance or HotStuff) by aggressively decoupling the logical ordering of transactions from the physical materialization and dissemination of evidence. This architectural separation routinely allows modern DAG-BFT networks to sustain throughputs exceeding 100,000 transactions per second (TPS) while maintaining sub-second latency.
The primary hypothesis under investigation—that this decoupling creates a severe hot-path storage problem requiring causal-aware retention—is not only theoretically sound but has recently been empirically validated by catastrophic failures in production environments. The hypothesis correctly identifies that high-throughput DAG-BFT validators frequently order or reference logical structures, such as cryptographic digests, block headers, and quorum certificates, significantly before the corresponding physical data payloads and execution dependencies are fully materialized, synchronized, or verified across local nodes. Consequently, the assumption that underlying storage layers can manage this causal evidence as a flat Key-Value (KV) history pruned by simple round or epoch watermarks is fundamentally flawed.
However, to provide a rigorous and critical evaluation, the framing of the hypothesis must be refined to distinguish between theoretical protocol state abstractions and the physical implementation realities of local storage state. In abstract protocol models, the dilemma between network fairness and garbage collection (GC) in DAGs is a well-documented mathematical challenge. For instance, DAG-Rider's reliance on "weak links" to guarantee asynchronous fairness famously necessitated unbounded memory, rendering it impractical for real-world deployment without architectural compromises. The profound novelty of the proposed hypothesis lies not in identifying the general necessity for garbage collection, but rather in diagnosing the acute cross-layer abstraction failure at the physical hot-path storage level. Modern production systems treat the underlying storage engine as a flat namespace governed by lexicographical bounds, completely ignoring the complex topological and causal dependencies required by asynchronous execution and sync pathways.
To properly situate this research direction, it is necessary to explicitly distinguish the hot-path causal storage problem from adjacent domains of distributed systems research. The hypothesis correctly points out that this issue is wholly distinct from ordinary blockchain historical pruning. Historical pruning (such as Ethereum's state trie pruning or EIP-4444) deals with state that has already been executed, finalized, and substituted by a cryptographically secure checkpoint or state snapshot. In stark contrast, the hot-path problem deals with active consensus evidence required for immediate liveness, quorum intersection, and current-epoch validation.
Furthermore, this problem cannot be solved by offline blockchain graph analytics or generic graph databases. While graph databases such as Neo4j provide ACID guarantees over complex topologies and natively understand forward and reverse edges, they introduce massive write-amplification and locking overhead. Imposing relational or graph-database transactional semantics on the consensus hot-path would obliterate the 100k+ TPS targets that DAG-BFT protocols are designed to achieve. Similarly, ordinary RocksDB or Log-Structured Merge (LSM) tree tuning is insufficient. RocksDB compaction and tombstoning operate strictly on lexicographical key spaces and are completely agnostic to the causal topology or execution-layer dependencies of the stored objects. Lastly, distributed file storage or Data Availability (DA) storage layers guarantee the eventual public retrieval of raw byte blobs but do not provide the synchronous, low-latency topological resolution required by a consensus execution engine to finalize a state transition in milliseconds.
Therefore, the hypothesis stands as a highly critical and urgently needed research vector. Flat KV pruning based merely on consensus round watermarks is dangerously ignorant of asynchronous execution paths, sync repair pathways, and accountability dependencies. As will be analyzed extensively in this report, this exact topological ignorance recently caused a multi-hour mainnet consensus failure in a major DAG-BFT network, proving that correct retention must inherently rely on causal awareness.

The Directionality Gap in Evidence Graph Maintenance
The physical storage of a DAG in an LSM tree database relies on maintaining references between the objects defined above. An analysis of these references exposes a fundamental gap in how modern implementations track dependencies, heavily favoring forward progression while remaining dangerously blind to reverse dependencies.
Forward references in DAG-BFT systems point backward in logical time (e.g., a vertex in round $r$ points to parent vertices in round $r-1$). These references are strictly explicit, durable, and immutable. They are embedded directly into the structural payload of the header or certificate. Because the parent hash is part of the cryptographically signed payload of the child vertex, forward references can never be altered once broadcast. This allows any node to deterministically trace the causal history of a given anchor block back to the genesis round.
However, reverse references—which are required to answer the critical retention question: "Who depends on this digest or vertex?"—are physically problematic and effectively absent in persistent storage. When Vertex A is written to persistent storage (e.g., RocksDB), the system inherently does not know which future vertices will eventually reference Vertex A. When Vertex B is later written referencing Vertex A, updating Vertex A's database entry to include a pointer to Vertex B would require a highly inefficient read-modify-write cycle. To maintain the massive IOPS throughput required by protocols like Shoal and Mysticeti (often requiring hundreds of thousands of writes per second), primary modules completely avoid maintaining on-disk reverse indices.
Instead, reverse references—such as tracking which child blocks are waiting on a missing parent, or which execution threads are waiting for a batch payload—are maintained only as implicit or transient structures in volatile memory. These are typically implemented as asynchronous hash maps within the consensus process. If a node crashes or restarts, the reverse dependency graph is instantly lost and must be expensively recomputed by scanning forward edges during the recovery phase. The absence of explicit, durable reverse references means the storage layer cannot utilize standard reference-counting techniques for garbage collection. The underlying storage system natively lacks the metadata to determine if a piece of causal evidence is still actively required by an uncommitted child, an incomplete synchronization task, or a pending asynchronous execution thread.
Materialization Assumptions and Decoupling Mechanisms
The core operational advantage of high-throughput systems like Narwhal, Bullshark, Shoal, and Mysticeti rests on the explicit assumption that logical evidence can securely enter consensus before physical materialization is strictly guaranteed at every local node. By heavily relying on cryptographic proofs of availability rather than immediate local possession, these networks drastically reduce the latency of the critical consensus path. However, this decoupling heavily complicates the hot-path storage lifecycle.
The Decoupling of Data and Metadata
In the Narwhal and Tusk architecture, traditional monolithic consensus operations are split. Mempool workers process raw transaction payloads, storing them locally and broadcasting them to peer workers. The workers then provide a cryptographic digest of these payloads to the primary consensus node. The primary node constructs a DAG consisting entirely of these lightweight metadata digests and forward references to previous headers.
Crucially, a primary node can receive a certificate from a peer, validate its cryptographic signature, and include it as a parent in its own next header without possessing the underlying batch payloads locally. The presence of a valid certificate (signed by $2f+1$ validators) guarantees that a quorum of workers somewhere in the global network holds the data. This allows the logical DAG to advance at network speed, unencumbered by the bandwidth constraints of large payload propagation.
Repair Pathways for Missing Bodies
Because the consensus logic advances independently of physical materialization, nodes must execute complex repair pathways when they eventually need to consume the evidence. If a node must execute a totally ordered sequence but lacks the underlying payloads, it relies on out-of-band synchronization protocols.
In Narwhal, a specific pull mechanism is implemented by the primary to seek missing batches. Upon receiving a block containing an unknown digest, the primary instructs its local worker to pull the batch directly from the remote worker that originally created the block. This ensures that the primary's bandwidth is conserved for consensus voting.
More sophisticated implementations, such as the Beluga block synchronizer integrated into the Sui blockchain's Mysticeti consensus, formalize this process into a dedicated abstraction layer. Beluga utilizes a deterministic pull strategy for "live" missing blocks, sending rapid pull requests to specific validators to minimize latency and ensure receipt within a $2\Delta$ round-trip time. For historical or bulk recovery, Beluga employs random pull strategies to prevent adversarial validators from triggering coordinated network stalls via redundant, uncoordinated pulls. Crucially, these repair pathways operate asynchronously and entirely outside the purview of the persistent storage layer's garbage collection logic.
Asynchronous Consumption Pathways
Once the DAG successfully orders a sequence of commit anchors, the evidence must be consumed. This consumption happens asynchronously via decoupled execution engines. After the consensus protocol outputs a total order, a separate execution thread traverses the causal history of the committed anchors, fetches the physical payloads from local worker storage or remote peers, and computes the final state transitions.
This creates a perilous, multi-layered race condition. If the primary consensus node garbage-collects its DAG vertices, or if the workers prune their stored payloads based solely on the fast advancement of the consensus commit frontier, slower nodes or lagging execution engines will encounter "Not Found" errors during their asynchronous catch-up routines. This leads to localized liveness failures, where a node cannot compute the state transition, or systemic failures if the pruning occurs globally before a critical threshold of nodes has executed the state.
Retention, Garbage Collection, and The Fairness Dilemma
Because fine-grained reference counting is too costly for high-throughput persistent storage, DAG-BFT systems rely heavily on heuristic or threshold-based Garbage Collection (GC) mechanisms to prevent state from growing infinitely. The literature reveals several distinct approaches to GC, each making specific mathematical compromises between memory boundedness, network fairness, and execution safety.
Round and Watermark-Based Pruning
Narwhal introduced a highly structured, round-based GC mechanism. The primary and workers collaboratively track a "low watermark" and a "high watermark." The high watermark represents the latest committed consensus round, while the low watermark represents the oldest round still topologically required—often determined by calculating the commit progress of the slowest honest validator in the network. Periodically, the system sweeps volatile memory alongside the underlying batches_table or blocks_table in RocksDB to aggressively delete any object belonging to a round below the low watermark.
The fundamental flaw in this approach is that the logic is purely topological and consensus-centric. It rigidly assumes that if the primary consensus logic no longer mathematically requires a round to build the active DAG frontier, the downstream execution and sync engines also no longer require it. This abstraction break is the root of the hot-path storage problem.
Asynchronous Fairness vs. Unbounded Memory
The earliest DAG-BFT protocols faced a mathematical dilemma between providing fairness in asynchronous networks and managing memory. DAG-Rider utilized "weak links"—references to older, uncommitted vertices—to ensure that blocks proposed by slow honest replicas would eventually be included in the total order. However, because a weak link might reference a vertex from an arbitrarily old round, DAG-Rider required the system to maintain an unbounded memory space. It was impossible to safely garbage collect old rounds because the protocol could never mathematically guarantee that a future block wouldn't legally reference a heavily delayed vertex.
To make DAG-BFT practical for deployment, protocols like Tusk explicitly removed weak links entirely, trading asynchronous fairness for the ability to implement aggressive, bounded garbage collection. If a transaction was orphaned due to aggressive pruning, Tusk relied on clients to re-inject the transaction into subsequent rounds.
Timestamp and Commit-Synchronized Pruning
Bullshark attempted to bridge this gap by restoring a degree of fairness during synchronous network periods while tightly binding garbage collection to the commit process. Bullshark establishes a specific GCRound threshold, calculated based on timestamp differences between rounds or the causal history of committed anchors.
However, even commit-synchronized pruning introduces severe implementation hazards. An audit of Aleo's implementation of the Bullshark consensus protocol revealed a critical vulnerability: the GC logic assumed regular commit progression. If the network stalled and commits temporarily ceased, the local DAGs would grow ad infinitum because the GCRound threshold was never updated. Aleo was forced to patch the implementation to ensure that memory thresholds were strictly bounded by commit events rather than assumed round progression. This highlights how fragile GC mechanisms are when they rely on single-metric heuristics rather than true causal dependency tracking.
Time-Bounded Weak References
Protocols such as Cordial Miners and Black Marlin eliminate the massive overhead of reliable broadcast primitives in favor of best-effort bare-message communication. To preserve fair transaction inclusion without breaking garbage collection, they introduce the concept of "time-bounded weak references." In these systems, blocks include a secondary set of references to previous rounds, but the protocol mathematically bounds the maximum temporal distance of these references. This inherently bounds the retention window required by the storage layer, completely preventing the unbounded memory issues that plagued DAG-Rider, while still providing a predictable window for fair inclusion.
The Uncertified DAG Pruning Model
The most complex retention dynamics exist in uncertified DAGs, such as the Mysticeti protocol powering the Sui blockchain. Mysticeti drops the explicit certification step used by Narwhal and Bullshark, dramatically lowering protocol latency. By advancing the DAG based on receiving $n-f$ messages rather than explicitly verifying $n-f$ cryptographic signatures per block, CPU overhead is massively reduced.
However, to maintain safety and execution capabilities without explicit certificates, the system must implicitly certify sub-DAGs by observing commit patterns. This complex topological requirement forces Mysticeti to implement two distinctly scoped and heavily disjointed garbage collection routines:
DAG-GC: Operates directly on the structural topology of the DAG. When a leader block is committed, all blocks not included in the committed subDAG are immediately dropped, and the structural round threshold is updated. This prevents unbounded memory growth during rapid block synchronization.
Vote-GC: Mysticeti-FPC (Fast Path Consensus) embeds fast-path transactions for owned objects directly within the DAG structure to bypass total ordering overhead. Finalizing these transactions requires tracking inherited votes down a block's self-ancestry path. Thus, voting metadata—stored in structures known as the vote_cache and descendant_graph—cannot be pruned concurrently with the structural DAG. It must be retained until the fast-path transactions are fully finalized by the execution layer.
The Knowledge Gap: Evidence State Tracking
To validate the hypothesis that flat KV history pruning is inadequate, one must ask: Does the persistent storage system inherently know the state of the evidence it holds? An exhaustive review of high-throughput DAG-BFT repositories (including Narwhal, Bullshark, Shoal, and Mysticeti) reveals a profound knowledge gap.
Repaired State: The knowledge of whether a missing block payload has been successfully repaired is managed solely by transient, volatile synchronizer logic (e.g., the Beluga protocol state machine). The persistent RocksDB storage layer is entirely unaware of active sync requests.
Executed State: The DAG consensus layer assumes a strict "fire and forget" architecture. It streams ordered commit signals to an execution engine via asynchronous channels but rarely blocks its internal garbage collection routines to wait for execution confirmation. Therefore, the system frequently does not know if a payload has actually been executed before it issues a deletion command based on round progression.
Checkpointed / Archived State: While advanced protocols like Hydra or Orthrus emit epoch-boundary checkpoints to facilitate state pruning , these mechanisms operate on macro-timescales (epochs). Inside an active, ongoing epoch, hot-path storage operates independently of archival storage and lacks real-time awareness of checkpoint handoffs.
Catch-up and Accountability: Byzantine Fault Tolerance relies on equivocation evidence—storing conflicting signatures from malicious validators to trigger slashing mechanisms or auto-unlock procedures. However, if the round containing the conflicting blocks falls below the consensus GC watermark, the evidence is blindly deleted by the storage layer, narrowing the critical window for accountability enforcement.
Is there any evidence that existing systems maintain enough metadata for causal-aware retention? The definitive answer is no. Modern implementations store data as isolated (round, digest) -> payload blobs. The critical gap is the complete absence of persistent reverse-dependency graphs or cross-subsystem retention oracles. RocksDB is told to delete a round, and it executes the deletion, mathematically oblivious to whether an active execution thread is midway through traversing a vote cache that relies on those exact blocks.
The Danger of Causal Ignorance: The Mysticeti 2026 Outage
The hypothesis posits that decoupling ordering from materialization without causal-aware storage implementation will inevitably lead to systemic failure. This is not merely a theoretical vulnerability isolated to academic papers; it manifested catastrophically in a production environment on January 14, 2026, when the Sui Mainnet suffered a prolonged six-hour consensus stall.
An in-depth post-mortem analysis of this incident provides the ultimate empirical proof of the hot-path storage problem. The global outage was triggered by a divergence in how honest validators managed causal evidence under the highly optimized Mysticeti-FPC protocol.
The Root Cause: Desynchronization of Abstractions
To achieve industry-leading low latency for transactions involving "owned objects" (assets strictly governed by a single private key), Mysticeti-FPC bypasses the traditional total ordering consensus path. Instead, users are responsible for ordering their own transactions, and validators merely need to detect and block conflicting transactions (double-spends) using a local lock table and a system of "invalid votes".
To minimize network overhead, Mysticeti-FPC was designed so that blocks natively inherit transaction votes from their "self-ancestors." Therefore, to determine the cumulative transaction vote of any given block, a validator must recursively traverse the full causal history of that block down its self-ancestry path to the originally voted block.
Because transaction finalization via voting is an asynchronous execution task that inherently lags behind the topological commit of the leader blocks, the Sui core developers implemented two distinct garbage collection abstractions: DAG-GC (which structurally drops physical blocks from the local DAG immediately after they are topologically committed) and Vote-GC (which delays the pruning of entries in the vote_cache and descendant_graph until the subDAGs are fully finalized by the execution layer).
The Causal Evidence Failure Cascade
This dual-GC architecture profoundly violated the fundamental requirement of causal evidence retention. Because the two GC mechanisms were decoupled and uncoordinated, the system created a fatal race condition between consensus progression and execution resolution.
The DAG-GC mechanism, acting as a flat, watermark-based pruner, prematurely deleted self-ancestry blocks from the hot path. The consensus logic successfully identified that the logical round had advanced past the required commit threshold and promptly instructed the storage layer to prune the underlying physical blocks. However, the fast-path execution logic (governed by Vote-GC) still actively depended on the causal history of those exact blocks to resolve complex vote inheritance chains.
Because the underlying physical blocks were deleted, the self-ancestry chain was irrevocably broken. Validators lost the "full ancestry up to the voted block" required to accurately resolve the transaction votes. Consequently, validators began to diverge on whether to accept or reject incoming transactions. Unable to consistently verify inherited votes, the honest validators failed to reach quorum, breaking the core agreement safety guarantee of the protocol. To prevent state corruption, the network algorithmically halted, resulting in an unexpected and massively disruptive six-hour blackout.
This incident provides irrefutable evidence that managing causal evidence merely as a flat KV history—where DAG-GC operates blindly to the execution layer's dependency graphs—is a fundamentally flawed paradigm. Correct retention inherently required the system to know "who depends on this evidence" (the vote_cache) and "whether it has been safely executed". The abstraction gap between the consensus structure and execution-level materialization resulted in a massive availability failure, highlighting the absolute necessity of the causal-aware retention policies proposed in the hypothesis.

Identification of the Strongest Research Gap
A critical synthesis of protocol literature, implementation architectures, and recent production failures reveals a profound and largely unaddressed research gap. The strongest research gap exists directly at the Systems-Storage Architecture Interface. Specifically, there is a fundamental and dangerous mismatch between the abstract causal graphs required by DAG-BFT consensus algorithms and the physical storage engines (like RocksDB) used to persist them.
This gap is explicitly a composite of storage-system-level and implementation-level deficiencies, which manifest directly as severe security/availability-level vulnerabilities. While the protocol-level mathematics of DAG-BFT are largely sound and well-verified (e.g., quorum intersections, commit rules), the translation of these mathematical abstractions into physical disk states breaks down under high-throughput conditions.
The Nature of the Abstraction Gap
Storage Engine Ignorance: Modern DAG-BFT nodes rely entirely on LSM-tree-based KV stores to persist vertices and batch payloads. An LSM tree manages data layout using lexicographical sorting and background compactions. It possesses absolute zero semantic understanding of a causal DAG topology. When a primary node executes garbage collection, it typically issues a generic DeleteRange command based on a round number (e.g., Delete all keys < Round 50). This operation is totally blind to cross-round causal dependencies and executing threads.
The Reverse Reference Void: Because building and maintaining a durable reverse-index in an LSM tree essentially doubles the write amplification of the database, systems simply do not maintain persistent data structures tracking which future objects depend on a given piece of historical evidence. Consequently, the GC module cannot perform atomic safety checks (e.g., "Is this specific batch payload still awaited by an asynchronous execution thread?") before issuing irreversible deletion commands to the disk.
Decoupled Lifecycle Hazards: As hypothesized, decoupling ordering from physical materialization dictates that a data object's consensus lifecycle (when it is mathematically ordered by the primary) operates on a completely different timescale than its execution lifecycle (when it is locally consumed by the state machine) or its sync lifecycle (when it is repaired by the Beluga synchronizer). Using a single, monolithic scalar metric—such as the committed consensus round frontier—to manage persistent storage is fundamentally unsafe. The Mysticeti vulnerability definitively proves that when storage is managed by decoupled, uncoordinated GC routines, the system will inevitably delete data required by lagging subsystems.
Why Existing Paradigms are Insufficient
One might assume that this gap could be bridged by adopting existing database or blockchain storage techniques, but these are uniquely ill-suited for the hot-path DAG-BFT problem:
Generic Graph Databases: While databases like Neo4j natively track causal forward and reverse edges and could easily answer retention queries, they are optimized for complex, offline analytical querying. They cannot sustain the raw, continuous $>100,000$ TPS write velocity required by systems like Shoal or Mysticeti. The locking overhead alone would stall the consensus pipeline.
Ordinary Blockchain Pruning: Linear blockchains (like Ethereum or Bitcoin) prune historical data only after it is finalized, executed, and safely buried beneath cryptographic checkpoints. The DAG hot-path problem involves active, uncommitted, or unexecuted data residing in the highly volatile edge of the mempool and consensus frontier. The retention logic must operate in real-time, not retroactively.
Volatile Memory Management: While systems programming languages like Rust (used by Narwhal, Bullshark, Shoal, and Sui) offer safe memory management via atomic reference counting (Arc), these elegant abstractions do not extend to the disk-backed LSM tree. If a validator node restarts, the RAM-based Arc counters are instantly lost, leaving the persistent state's safety guarantees completely reliant on fragile, forward-scanning recovery heuristics.
Defining the Research Direction Forward
This gap requires the creation of entirely new Causal-Aware Storage Abstractions for BFT Hot-Paths. Future research must systematically bridge the topological requirements of the DAG with the high-performance constraints of physical storage.
Specifically, this research vector should aggressively investigate:
Lightweight Durable Dependency Tracking: Developing novel data structures that can maintain asynchronous reverse-dependency metadata without suffering the severe IOPS penalties of standard relational or graph database indices.
Multi-Lifecycle Pruning Oracles: Designing GC controllers that aggregate state not just from the consensus round frontier, but concurrently from the execution engine's dependency graphs (e.g., Mysticeti's vote_caches), the repair synchronizer's waitlists (e.g., Beluga's pull requests), and accountability watchdogs.
Atomic Cross-Layer Deletion Protocols: Formalizing the safe deletion of causal evidence such that a storage backend will inherently reject a pruning command if the targeted vertex or payload is still topologically required for executing a fast-path consensus vote or resolving a pending synchronization request.
By formalizing the hot-path storage problem as an independent architectural layer with strict causal-retention guarantees, researchers can prevent the class of complex liveness and safety failures that currently threaten the stability and scalability of the world's most performant DAG-BFT networks.
Conclusion
High-throughput DAG-BFT protocols represent a fundamental paradigm shift in distributed consensus. By aggressively decoupling the topological ordering of events from the physical dissemination and execution of data, these protocols achieve immense scalability and near-optimal latency. However, as this comprehensive analysis demonstrates, this architectural decoupling introduces a severe and highly complex hot-path storage problem.
The hypothesis that causal evidence cannot be managed merely as flat KV history using simple round or checkpoint watermarks is unequivocally validated by both theoretical system analysis and stark empirical evidence. In asynchronous and partially synchronous environments, evidence must be retained based on its active topological utility to multiple distinct, asynchronously operating subsystems: the consensus primary, the execution engine, and the repair synchronizer.
When protocol implementations assume that evidence materialization is perfectly synchronized with logical ordering—or when they attempt to apply independent, uncoordinated garbage collection policies to deeply interdependent causal structures—they invite catastrophic failure. The January 2026 Sui Mainnet outage, directly caused by a desynchronization between DAG structural pruning and execution vote-cache dependencies in the Mysticeti protocol, stands as the definitive proof that causal ignorance at the physical storage layer is a fatal vulnerability.
Addressing the identified research gap—creating a high-throughput, causal-aware storage architecture that natively tracks the lifecycle of hot-path evidence across subsystem boundaries without destroying TPS performance—is absolutely critical. Solving this hot-path storage problem will mature DAG-BFT networks from highly performant academic models into the robust, fault-tolerant substrates required to safely manage global, high-stakes infrastructure.
DAG-BFT 系统：热路径证据对象与依赖
Narwhal/Tusk：每轮由每个验证器发布一个区块（含身份标识、轮次号、交易列表、指向前一轮至少2f+1个区块证书的引用，以及本地签名），并通过可靠广播（RBC）分发。当某个区块获得 ≥2f+1 验证器的签名后，就形成一个可用性证书（certificate），包含该区块的摘要、轮次、签名集合等。因此，前向引用有：区块→父证书（上轮块）、证书→区块摘要。区块中通常包含事务或事务批摘要（即Narwhal的“批”）。Narwhal 允许在部分数据缺失时先提交摘要/证书，其它节点后续通过点对点提取（pull）机制或RBC保证重新获取完整区块。证书和区块头构成DAG结构，为每个新区块提供依赖；但系统中不显式维护区块到其子项的反向指针。所有Narwhal节点只需持有当前轮次的区块和证书即可正确运行，较早轮次的数据（包括区块体和事务）可在共识确定了垃圾回收轮次后卸载到被动存储（如CDN或归档）。总之，Narwhal 采用轮次为粒度的回收策略：当共识协商并固定了一个 GC 轮次后，所有早于此轮的块/证书可丢弃。这种策略只保留“当前”轮次的因果历史（带有签名的可用证书），其余历史移交外部存储。若回收不当（不同节点选择不同GC轮次），可能导致DAG历史不一致。Narwhal/Tusk 并未额外记录执行或快照是否消耗了某块，仅依赖轮次和签名完成度；因此它不具备按“消费”情况做保留决策的元数据。
Bullshark：基于类似DAG-Rider的设计，但加入了同步时序保证。每轮每个节点广播一个包含交易的**顶点(Vertex)**结构，内含：源标识、轮次、事务数据，以及至少2f+1个“父顶点”引用（上一轮强引用）和最多f个早轮弱引用。顶点通过可靠广播达到一致传递；RBC机制保证相同内容但无集中式证书对象。前向引用为：顶点→前一轮（强/弱）父顶点。没有专门的反向链表，结构上的有向边隐含依赖。Bullshark对每个顶点附加时间戳$t_s$，并维护一个“垃圾回收轮次”指标。具体地，在一个轮次的领导者被排序（提交）后，一旦至少n-f个顶点的中值时间戳超过阈值，该轮次就被回收。也就是说，保留策略基于同步条件和领导者提交：完成共识后，旧轮次的顶点均可移除。这些顶点提交后，其事务会被交付执行；执行完成与否不影响回收决策（因为只有顺序确定轮次的完成才触发GC）。Bullshark假定遗漏的事务若未包含在提交顶点中，会被后续重新提案，所以即使某些顶点被GC，事务最终仍会被包含。协议没有对领导者投票过程或执行状态做特殊标记，仅使用顶点轮次和时间戳作元数据。若回收过早，尚未被新的强引用覆盖的弱引用可能丢失，影响公平性保障。
DAG-Rider：异步DAG协议，每轮每节点广播一个顶点（含事务、轮次编号），至少包含前一轮2f+1个父引用和最多f个弱引用。它避免领导者，顶点通过RBC传播。前向引用：顶点→前轮顶点（强/弱链接）。反向引用：无。重要区别是DAG-Rider的弱链接机制保证了最终所有事务公平性，但也意味着需要保留所有历史顶点，没有安全的垃圾回收策略；凡是收到的历史数据都可能最终被引用（no GC）。也就是说，它不删除任何历史顶点，否则会破坏公平性和一致性。
Shoal / Shoal++：这是在Narwhal/Bullshark之上改进的协议，基本对象同Narwhal/DAG（块和证书）或Bullshark（顶点），并无新增证据类型。Shoal引入负责人机制和流水线来减少延迟，但保留策略与上层协议相同——一旦交易被共识提交后，其包含的数据可在后续被回收。Shoal假定系统能够识别已提交事务的影响，但并未引入额外持久化标记。其前向引用和反向引用模式与底层Bullshark相同，没有针对“热路径”做特殊处理。
Mysticeti：一种新的DAG-BFT协议，采用“无证书DAG”模式，实现3消息轮次达成共识。每个顶点依旧包含事务和父引用，但不需要第2f+1签名的证书。前向引用为：块→使用阈值逻辑时钟指示的前驱块。由于没有传统证书，块通过阈值广播（或RBC）进行传递；如果节点缺少块内容，需要从签名该块的节点处拉取数据。Mysticeti论文中提出了检查点机制（每若干块设置检查点）以保留状态跨周期。具体GC方案未详述，但暗示通过Epoch/检查点提交来决定何时完成旧历史的持久化。该协议也只保留处理当前流水线所需的区块历史，不维持额外的反向引用。
Cordial Miners：提出基于“块链(blocklace)”的DAG算法，将广播、排除双签、排序集成在同一结构。基本对象是区块（含事务、前驱引用）。前向引用为：块→其在块链结构中的前驱；协议没有可靠广播构件，而是通过块链的因果顺序来实现。这意味着没有独立的证书对象，所有验证依赖签名和哈希链完整性。Cordial 没有在论文中明确垃圾回收策略；由于采用了Hashgraph式DAG，它可能需要保留大量历史以确保正确投票和去重。因为不是基于轮次的视图，因此难以清理历史——类似DAG-Rider，它并未提出自动GC方案，只保证安全性和效率。
区块链存储引擎与裁剪机制
数据种类：传统区块链节点存储的类别包括：块头、块体（交易列表）、交易收据、交易日志、状态树节点（或账户存储键值）及合约代码等。例如，以太坊 Geth 客户端的数据库显示，KV 存储分为“Headers、Bodies、Receipts、Transaction index、Log index、Contract codes、Trie nodes”等多种类别。状态数据通常以Merkle Trie形式存在或变种（如以太坊即将引入Verkle Trie），并可能定期生成快照。描述了Geth使用状态快照和布隆过滤器来确定应保留的Trie节点。Aptos也存储交易日志、事件、写入集(accumulator)、各版本的对象状态等。Sui 则以对象为中心，存储对象历史版本、交易、事件以及检查点元数据。比特币节点存储UTXO数据库及必要的区块信息。Celestia作为数据可用性层，存储所有块及其编织数据，但关注点在数据摘要和分发，不管理全局状态。
裁剪/保留策略：各系统采取不同策略平衡数据规模。一类是基于高度/轮次/版本窗口的保留：例如，以太坊Geth支持历史剪裁(geth prune-history)，可以删除早于某区块（如合并前区块）的块体和收据，只保留区块头和当前状态。Geth还可执行离线状态数据库修剪，利用区块快照确定目标Trie层并删除不在快照中的节点。Erigon客户端则在初次同步时确定修剪模式（full、minimal或archive），一旦同步完成无法更改；full模式丢弃旧状态数据以保持较小存储。Nethermind提供混合裁剪模式：内存裁剪不断舍弃过时状态，小幅控制增量存储；完全裁剪按需触发，将当前状态复制作新数据库，验证后删除旧数据库。历史数据导入导出功能（如Era1格式）允许用户重新填充或导出区块历史。Aptos默认保留最近1.5亿笔交易，可配置prune_window以不同版本窗口裁剪账本交易历史。如果关闭裁剪，节点会增长到TB量级；一般保持至少1000万笔以上以防重组。Sui根据区块纪元(epoch)裁剪：全节点通常只保留当前纪元内的对象与交易，前面纪元的历史被转移至检查点桶（云存储）。这样，节点上磁盘使用受限于一个或几个纪元；查询更早对象时可从云上的检查点恢复。比特币则可配置“Prune mode”，删除一定深度以前的区块，只保留UTXO集与最新数据。Celestia暂未提供本地裁剪机制，因为其设计本质为轻客户端依赖外部DA服务。
依赖/因果跟踪：现有区块链存储引擎通常只意识到线性块链或版本号的依赖，而不跟踪多维依赖“谁需要谁”。例如，以太坊的状态Trie保留父子指针用于Merkle验证，但不会对特定历史账户状态建立反向链；也没有概念“交易被多少后续事务引用”。以太坊节点通过快照或状态根确定保留边界，与执行是否完成有关但并不细粒度标记每笔交易是否被索引或已删除。Aptos内部使用版本号和写入集累加器（accumulator）记录状态变化，但只做单向累加历史，不维护事务到后续状态的显式引用。它的事务索引（txn_by_hash）不易按版本裁剪，需要专门清理。Sui存储对象的所有历史版本，有版本链指示前驱关系，但也不存任何全局反向依赖。大部分链都保留映射索引（如交易哈希→区块，日志索引），用于查询历史，但这些索引主要服务查询，不直接参与共识的热路径。简言之，这些系统不具备DAG-BFT那种“因果价值”标记：它们根据高度或状态快照裁剪，而非根据后续依赖关系来决定某条证据是否仍然必要。
反向引用：一些链通过索引表实现少量反向查询：以太坊中的“块哈希→块号”、“交易哈希→块索引”、日志索引行等。这些索引用于RPC查询（查历史Tx或事件），但它们针对的是外部查询需求，不用于共识安全。状态树本身没有维护父指针的倒排列表。Aptos的事务索引类似，以备通过哈希查交易，但其设计不利于分段裁剪。Sui对象ID可用于查找最新版本，但旧版本只能依赖检查点。总的来说，除非为特定查询专门建立索引，否则这些系统没有主动维护“谁依赖了这个对象”的持久反向链；这与需要保障热路径完整性的DAG-BFT要求存在差距。
执行感知保留：当系统判断可以回收时，是否考虑记录某证据是否已“消费”？ 绝大多数传统链节点不具备这一功能。它们的裁剪通常依赖于最终状态根或版本窗口，隐含执行已同步到某个高度。例如，以太坊在完成状态快照后再裁剪，确保状态已确定；Sui按检查点裁剪前一个纪元的数据，也意味着该纪元内所有交易均已执行并快照。但链存储本身并不记录“这笔交易是否已发送给执行层”或“这个合约码是否加载完毕”这类标记。换句话说，保留决策不特别区分“还在待执行队列”还是“已确认执行”的数据。仅仅因为状态被写入快照，就被认作已执行并可剪除旧数据。
线性链 vs. DAG-BFT 对比：线性链存储的保留机制与DAG场景存在本质差异。线性链拥有单一路径的因果关系——新区块只需依赖最近的已最终化区块；因此，只要保留从最新最终化高度开始的状态快照和区块信息，即可满足安全需求。依赖关系简单、单向（前继区块）。而在高吞吐DAG-BFT中，每个块或证书可能被多条未来指令引用（父子关系形成复杂DAG），因此裁剪必须考虑“哪些未决的共识或执行操作还依赖此证据”。传统链没有为这种多方向依赖设计元数据；它们认为只要最终状态根稳定，旧历史就是可删的。这种差异意味着：向DAG-BFT迁移时，必须引入针对因果依赖的新机制，而非沿用现有的线性快照思路。

先前研究与创新差异
现有区块链节点普遍采用线性存储模型，仅根据最终状态和时间窗口来裁剪数据。以太坊Geth等客户端通过状态快照和高度阈值删除过期数据；Aptos和Sui也提供按交易计数或纪元划分的保留窗口。这些机制关注的是“历史是否可删”而非“谁需要此证据”。例如，Nethermind提出的Era1格式只是为离线归档区块方便校验历史，无关热路径决策。此前研究（如EIP-4444等）也只涉及区块或交易的传输与存储约束，并未跟踪共识依赖。换言之，传统链存储注重状态有效性，而不维护证据的依赖计数或可达性元数据。因此“因果感知保留”这个问题在现有文献和实现中尚属空白。高吞吐DAG-BFT的独特之处在于：排序决策可以在证据具体物化前作出，各节点接收到不同“视图”的依赖，因而必须显式记录哪些历史数据仍是当前决策的根基。这更多是一个协议+实现层面的新课题，而非普通数据库问题。现有系统在热路径上并未保留足够元数据来支持因果依赖的精细回收——例如，他们不会记录每条交易在多少待执行中被引用或被某检查点覆盖。因此，高吞吐DAG-BFT的创新之处还在于如何设计额外的索引/标记机制，以确保只删去不再被任何共识/执行路径使用的证据，而这一点目前还需要新的方法来实现（可归类为存储系统级和协议级的研究缺口）。
Causal Evidence Storage in Storage-Backed DAG-BFT Architectures: A Critical Analysis
Introduction and Problem Context
The architectural evolution of high-throughput distributed ledgers has increasingly gravitated toward Directed Acyclic Graph (DAG) based Byzantine Fault Tolerant (BFT) consensus protocols. Protocols such as Narwhal, Tusk, Bullshark, and Mysticeti achieve extraordinary throughput by explicitly decoupling the dissemination of transaction payloads from the logical ordering of those payloads. In these systems, vertices in the DAG represent proposals that encapsulate batches of transactions, or more commonly, cryptographic digests of those batches. As validators vote on these vertices by including references to them in subsequent proposals, they organically construct a causally ordered graph. The protocol ultimately achieves consensus by committing a specific "leader" vertex and, by extension, its entire causal history.
This structural concurrency, while optimal for maximizing network throughput, fundamentally alters the assumptions surrounding blockchain storage engines. In traditional linear blockchains, the inclusion of a transaction in a block dictates its ordering, its execution, and its physical materialization simultaneously. In high-throughput DAG-BFT systems, however, logical ordering is decisively decoupled from physical evidence materialization. Validators may order or reference digests, headers, certificates, and parent references long before the corresponding object bodies, batch payloads, execution dependencies, or archive substitutes are fully materialized and consumed by the local execution engine.
This decoupling introduces a critical tension. The storage substrate is burdened with a complex hot-path storage problem: retaining the structural evidence of the DAG to guarantee liveness and correctness for lagging peers. The hypothesis under investigation posits that causal evidence cannot be managed merely as a flat key-value history subject to naive round-, epoch-, or time-based pruning. Correct retention theoretically requires knowing who depends on this evidence, whether it serves as an active protocol root, whether it has been consumed by execution and accountability paths, and whether it has been safely substituted by a state snapshot.
This report conducts an exhaustive evaluation of this hypothesis. By systematically analyzing the storage engines, data taxonomies, dependency tracking mechanisms, and pruning logic of prominent blockchain architectures—including the Ethereum ecosystem (Geth, Erigon, Reth, Nethermind), Aptos, Sui, Celestia, and Bitcoin—the analysis isolates prior art, identifies conceptual gaps, and delineates the genuine novelty in designing execution-aware, causality-driven storage engines for DAG-BFT consensus.
Taxonomical Analysis of Ledger Storage Categories
To contextualize the management of causal evidence, it is necessary to establish exactly what data categories are stored across modern ledger architectures, as the cryptographic and structural nature of the data dictates the retention policy. The presumption that blockchain storage systems are "naive KV stores" is historically inaccurate; modern clients employ highly specialized, tiered database architectures.
In the Ethereum ecosystem, the stored data is predominantly state-centric rather than consensus-centric. Legacy clients like Geth and Nethermind store blocks, headers, receipts, and a massive set of Modified Merkle Patricia Trie (MPT) nodes that represent the global state. Because the MPT deduplicates state across blocks, nodes are heavily interleaved, creating a complex web of shared cryptographic structures on disk. Erigon revolutionized this model by moving away from the MPT for state storage, introducing a flat key-value architecture backed by the MDBX storage engine. Erigon categorizes data into highly specific tables: Headers, Block Bodies, Receipts, PlainState (which stores accounts and storage together), History Of Accounts, Change Sets (recording the history of changes at each block), and HashedState for calculating the Merkle root on the fly when necessary. Reth further refines this by implementing a tiered storage model (Storage V2), isolating hot state in RocksDB and offloading historical account and storage changesets to append-only static files.
Aptos utilizes a Jellyfish Merkle Tree (JMT) optimized for Log-Structured Merge-tree (LSM-tree) databases like RocksDB. The JMT uses a monotonically increasing, version-based key schema designed specifically to mitigate write amplification. Aptos categorizes its core data into the ledger_db (containing transactions, transaction outputs, events, and write sets) and the state_merkle_db (the state authentication data structure). Furthermore, Aptos maintains specific index families such as stale_node_index and stale_node_index_cross_epoch to track data that has been overwritten.
Sui operates on an object-centric model rather than an account-centric one. Every object on-chain is referenced by a globally unique ID and a strictly increasing version number. The storage architecture is heavily bifurcated based on node roles. Validators maintain a consensus_db on high-performance NVMe drives to store the latest DAG consensus data (Mysticeti vertices, certificates, and causal links), while the authority_store (or authorities_db) maintains the execution state, object versions, effects, and events. Full nodes maintain unpruned RocksDB indexes for efficient querying. Sui also heavily utilizes checkpoints, which define the certified history of the blockchain and act as the definitive archive substitute, often offloaded to cloud buckets.
Conversely, Celestia, acting purely as a Data Availability (DA) layer, fundamentally eschews state execution storage. It stores blocks, headers, and raw data blobs organized by namespaces. The data category is purely sequential bytes and the namespace Merkle trees required for data availability sampling (DAS). Bitcoin, utilizing the Unspent Transaction Output (UTXO) model, stores sequential block files containing full transaction data, alongside a distinct levelDB database for the active UTXO set and block index.

Pruning Mechanics and Retention Policies Across Architectures
The mechanisms by which these systems discard data reveal profound differences in how they value historical dependencies. Pruning policies are largely dictated by the data structures they manage and the finality rules of their respective consensus algorithms.
In the case of legacy Ethereum clients like Geth, the reliance on the MPT necessitates a complex, in-memory reference counting garbage collector. When processing a block, Geth tracks all trie nodes whose reference count drops to zero. However, because states belonging to different blocks share the same data on disk to minimize storage, determining when an arbitrarily old trie node becomes entirely unreferenced is computationally exhaustive. Geth's historical pruning attempts to flush older nodes to disk first, which often results in dangling subtries that are exceptionally difficult to find and remove from the database. This highlights a fundamental limitation: tracking granular dependency metadata (such as reference counting) on persistent storage introduces intractable write amplification and metadata shuffling. Therefore, Geth relies on offline state pruning to rebuild the trie natively from a recent state root.
Erigon resolves the complexity of MPT pruning by utilizing a flat state model combined with an asynchronous pipeline. Erigon's architecture is separated into the Executor, the Committer, and the Pruner. Pruning is dictated by a distance metric (--prune.distance), which defines the maximum amount of committed state history the node is allowed to keep alive, measured in block height. This is a block-height-based moving window. Because Erigon uses ChangeSets to record reverse diffs at each block, it can simply drop ChangeSets that fall outside the prune horizon. Reth mirrors this distance-based approach but achieves extreme disk I/O efficiency by utilizing static file segments (e.g., 10,000 blocks per file) that can be unlinked from the underlying operating system filesystem instantly, completely bypassing LSM compaction overhead.
Aptos implements a strictly windowed pruning policy. The ledger_pruner is configured to keep only the most recent transactions, defaulting to a window of 150 million transactions for mainnet nodes. For state data, Aptos employs specialized state_merkle_pruner mechanics, divided into inner-epoch and inter-epoch pruners. If a state tree node is overwritten by a later transaction within the same epoch, it is aggressively pruned; if it is overwritten by a transaction in a different epoch, it is pruned according to inter-epoch configurations. This represents a highly structured, epoch-aware retention policy designed to maintain a perfectly lean JMT.
Sui's pruning policies are arguably the most sophisticated among smart contract platforms, heavily utilizing RocksDB-native features. Sui prunes object versions, historic transactions, effects, events, and checkpoint data. The retention policy is defined by the authority-store-pruning-config, which relies on specific epoch parameters such as num-epochs-to-retain and num-epochs-to-retain-for-checkpoints. Crucially, Sui relies on RocksDB Compaction Filters for object-table pruning. Rather than manually tracking and deleting keys on the hot path—which stalls the execution thread—a compaction filter allows the application to inspect a key-value pair during the background LSM compaction process and dynamically decide whether to drop it based on custom logic (e.g., checking if the object version is older than the retention threshold).
In contrast, Celestia utilizes a purely time-based pruning window. The pruning window was recently reduced from 30 days to 7 days plus a 1-hour buffer. This window is intrinsically linked to the "weak subjectivity period," ensuring that light nodes can sample data within a timeframe where the validator set is still subject to social consensus slashing conditions (the unbonding period). Bitcoin nodes using UTXO pruning operate on a space-bounded logic; they simply delete flat block files starting from the oldest once the configured local storage size limit is reached, retaining only the active UTXO set database and the chain of block headers to verify proof-of-work.
Dependency Tracking, Causal Value, and Reverse Referencing Mechanisms
The central tenet of the proposed hypothesis hinges on whether storage systems track dependency or causal value, and whether they maintain reverse references to execute retention correctly. The analysis indicates that while modern engines are highly sophisticated regarding the management of state dependencies, they are virtually blind to causal consensus evidence.
Reverse references are maintained strictly for state history, historical queries, and rollback mechanisms, not for hot-path consensus causality. As established, Geth’s reference counts act as reverse references for trie nodes, but this is purely an in-memory garbage collection mechanism to determine when a subtrie is no longer needed by any block. It is not used for hot-path correctness.
Aptos, however, utilizes an explicit reverse index mechanism. The stale_node_index explicitly tracks reverse diffs—it records the specific ledger version at which a state node was overwritten. This allows the state_merkle_pruner to identify exactly which keys to delete when the pruning window advances past that version, completely avoiding the need to re-traverse the Merkle tree to find obsolete nodes. This is an elegant reverse index, but it is utilized strictly for state garbage collection and offline database maintenance.
Erigon maintains deep reverse references in the form of ChangeSets and History Of Accounts. Whenever a block executes, Erigon records exactly what the previous value of the account or storage slot was before the mutation. This allows the node to rewind the state to any historical block without needing to store full snapshots at every height. These reverse diffs are used for historical RPC queries (like eth_call at a past block) and for handling chain reorgs.
Sui tracks object version dependencies intrinsically through its data model. Because every mutation increments the object's version number, the database naturally maintains a linear lineage of an object's history. Sui tracks transaction dependencies via transaction effects, logging exactly which object versions were consumed and produced. However, this is linear state execution tracking.
When examining the specific retention of DAG-BFT causal evidence, these systems display a conspicuous lack of dependency tracking. The DAG vertices contain parent references (cryptographic hashes of previous proposals), establishing the structural graph. Yet, the storage engines do not maintain a dynamic reverse index answering "which active vertices depend on this older vertex?" In Narwhal, Tusk, and Mysticeti, the underlying consensus_db treats these vertices largely as isolated blobs of data indexed by their digest or round number. There is no reference counting or reverse diff mechanism applied to the consensus causality graph itself.
Execution-Aware Retention and the Hot-Path Disconnect
Execution awareness defines whether a node's pruning mechanism communicates with its execution engine. If retention decisions are execution-aware, the system knows whether a transaction, object, or checkpoint has been successfully executed, indexed, archived, or substituted by a snapshot before authorizing its deletion.
Execution awareness is heavily embedded in Erigon's state architecture. The Pruner operates asynchronously but is strictly bounded by the Committer. The pipeline maintains a sequential flow: Execution, followed by Committing, followed by Pruning. A gap naturally exists between these stages due to batch sizes and disk I/O latency. If the prune horizon advances past the last_committed_state, a critical failure occurs because the system would delete nodes that were never safely flushed to disk, necessitating a massive replay of chain history. Erigon's retention decisions are thus exquisitely aware of execution progress; the pruner is hard-locked behind the committer.
Similarly, Sui's transaction pruning is strictly gated by checkpoint completion. The background pruner only removes historical transactions and their effects after the encompassing checkpoint has been completed, signed by a quorum, and finalized. A checkpoint serves as the permanent record used for state sync, proving that the causal structural evidence (the DAG) has been safely substituted by a state coverage artifact.
However, when examining the retention of DAG-BFT causal evidence directly within the consensus engine, existing protocols revert to execution-ignorant mechanisms. In Sui's Mysticeti consensus, garbage collection of the consensus_db is triggered during commits. The protocol takes the round of the leader block, and all blocks with lower rounds that have not been included in any committed sub-DAG are dropped from the database. Narwhal and Tusk similarly operate with a configured garbage-collection depth (e.g., 50 rounds). Fides also utilizes a strict depth-based DAG garbage collection.
This depth-based and commit-based mechanism is inherently execution-agnostic. The DAG drops vertices based purely on the advancement of logical rounds and consensus commits. If the execution engine is heavily backlogged and has not yet materialized the payload digests contained within a committed vertex, a round-based garbage collection will prematurely delete the causal evidence required for the execution node to trace the payload dependency. If the network experiences severe partial synchrony, a node relying on hot-path evidence might find that the parent references it needs to construct a valid proposal have been pruned from the consensus_db simply because the global DAG advanced too rapidly. The retention mechanism has no concept of whether the payload was "consumed by repair/execution/catch-up paths."
Delineating Causal Evidence from Conventional Storage Paradigms
The user's hypothesis specifically demands distinguishing this causal evidence problem from adjacent storage paradigms. This is crucial to establishing the academic boundary of the research direction.
Distinction from Ordinary Blockchain Historical Pruning: Ordinary pruning (like Geth's trie pruning or Erigon's block distance pruning) assumes that older data is universally less critical to immediate protocol execution. Once a block is finalized in a linear chain, its historical state has zero "hot-path" correctness value. Causal evidence pruning, however, deals with data that might still be actively required to resolve ordering ambiguity among concurrently committed units. Pruning a DAG vertex too early breaks the local node's ability to verify the precedence constraint of the current frontier.
Distinction from Offline Blockchain Graph Analytics: Offline analytics tools (such as Dune Dashboards indexing Sui checkpoints) operate strictly out-of-band. They reconstruct the graph after all data has been safely archived and checkpointed. They do not operate under the latency, memory, or disk I/O constraints of a live validator. The hot-path causal storage problem requires making retention and dropping decisions in milliseconds while the consensus engine is continuously streaming thousands of vertices per second.
Distinction from Generic Graph Databases: Generic graph databases (e.g., Neo4j) track vertices and edges with extensive metadata and rich indexing. Applying a generic graph database to a high-throughput DAG-BFT consensus engine would instantly suffocate the protocol. The overhead of writing dependency arrays, updating reverse-edge indices, and executing complex Cypher-style queries on disk for every 512-byte transaction batch would introduce catastrophic write amplification, mirroring the failure of Geth's on-disk reference counting.
Distinction from Ordinary RocksDB/LSM Tuning: RocksDB tuning involves adjusting block cache sizes, bloom filters, write buffer sizes, and compaction styles (Level vs. Universal). While Sui utilizes a Compaction Filter , this is merely a delivery mechanism for custom logic. Ordinary tuning optimizes the engine for raw bytes. The causal evidence problem is fundamentally algorithmic: the storage engine must be taught when a key-value pair represents an active protocol root versus an obsolete, checkpointed structure, a logic that standard LSM parameters cannot express.
Distinction from Distributed File Storage or DA Storage: Systems like Celestia or IPFS provide data availability and retrievability. They operate on temporal pruning windows (e.g., 7 days). They do not care about the causal relationship between blobs. DAG-BFT storage must care about causality because a missing parent vertex invalidates the entire sub-DAG, stalling the local consensus instance.
Comparative Paradigm: Linear Blockchain Storage Versus DAG-BFT Causal Storage
The fundamental divergence between linear blockchain storage and DAG-BFT storage lies in the origin of finality and ordering. In linear architectures (Ethereum, Bitcoin), physical materialization and logical ordering are perfectly synchronized. A block is only valid if its parent exists, its payload is available, and its state transition is computable. Storage engines like Erigon and Reth can safely prune historical state because the lineage is cryptographically frozen in a single chain of headers.
In a DAG-BFT system, the committed DAG is itself the structural ordering substrate. Vertices are committed concurrently. The final linearization of transactions is derived post-consensus via deterministic graph traversal. Consequently, a vertex possesses intrinsic "causal value." It provides the verifiable structural precedence evidence necessary to reconstruct the shared history.
If a linear blockchain prunes block 1,000,000 while the tip is 1,000,064, node operation is entirely unaffected because block 1,000,064 contains the complete, finalized state root required to process block 1,000,065. However, if a DAG-BFT node prunes vertex A while vertex B (which causally depends on A) is still being processed by a lagging execution layer or is needed to resolve ordering ambiguity among concurrently committed atomic units, the node loses the ability to independently verify the precedence constraint.
Mechanisms that transfer well from linear systems include background LSM compaction filters (Sui) and stale node tracking (Aptos), as they efficiently manage the disk I/O of flat state without stalling the main thread. Mechanisms that do not transfer include simple block-distance pruning (Reth/Erigon) and time-based availability windows (Celestia). These linear mechanisms assume that older data is universally less critical to immediate hot-path protocol execution. In a partially synchronous DAG, an older vertex might be a crucial, unmaterialized parent reference required to stitch together a newly finalized sub-DAG.

Critical Evaluation of the Hypothesis: Prior Art and Genuine Novelty
The hypothesis posits that causal evidence storage is a distinct hot-path problem requiring a retention mechanism that tracks active protocol roots, repair consumption, and snapshot substitutions, effectively distinguishing it from ordinary KV history pruning. This framing is highly astute, yet it requires critical tempering against existing prior art in distributed systems and the realities of storage engineering.
Prior Art Weakening the Novelty
The concept of "causal-aware retention" is not without precedent. In the domain of pure distributed systems theory, distributed garbage collection based on causal dependencies is extensively documented. Algorithms utilizing vector clocks and vector-times to identify global roots and safely garbage collect causally obsolete events in a distributed computation trace back decades. In these systems, a node merges dependency vectors and computes whether a global root is no longer reachable from any active root, subsequently removing it. This mirrors the user's specific requirement to know "whether it is still an active protocol root."
Furthermore, existing DAG-BFT implementations are not entirely naive, and characterizing them as treating data merely as "flat KV history" underestimates systems like Narwhal, Mysticeti, and Fides. These systems maintain in-memory graph structures and execute garbage collection precisely by tracking committed sub-DAGs and dropping vertices that are excluded from the causal history of a committed leader block. Tusk specifically had to forbid "weak links" (a feature from DAG-Rider designed for theoretical fairness) because weak links made garbage collection impossible; every block ever received had to be stored in case it was pointed to by a weak link. This proves that DAG consensus designers are actively aware of the topological constraints of storage.
Additionally, the database mechanics required to implement such retention already exist. The user seeks to avoid ordinary RocksDB tuning, yet Sui explicitly uses RocksDB Compaction Filters to safely drop overwritten object versions in the background. A compaction filter could theoretically be fed an in-memory bitmap of "active protocol roots" to safely drop obsolete causal evidence from the disk during standard LSM compactions, neutralizing the "hot-path" storage bottleneck without needing a novel database architecture.
The Genuine Novelty: Bridging the Decoupling Gap
Where the hypothesis identifies genuinely novel territory is in the intersection of storage-backed causal evidence and execution-lag materialization in BFT systems.
Existing distributed GC algorithms and DAG round-pruning mechanics suffer from a fatal assumption: they assume that once structural consensus is reached, the data's immediate hot-path utility is exhausted. In a high-throughput DAG-BFT system where dissemination outpaces execution, a vertex might be logically committed by the consensus engine but its payload digests remain physically unmaterialized by a lagging replica. If the consensus engine strictly enforces a 50-round garbage collection depth or drops non-leader paths on commit , it will physically delete a committed vertex from the consensus_db before the trailing execution engine has requested the payload digest contained within it, or before the node has established the verifiable precedence evidence for Multi-Round Visibility ordering.
The resulting failure mode forces the replica out of the hot-path DAG traversal entirely and into a slow, out-of-band checkpoint state-sync, severely degrading node performance. Therefore, standard DAG-BFT garbage collection is consensus-aware but decidedly execution-ignorant.
The genuine novelty of the proposed research direction is designing a storage retention policy that is explicitly aware of the physical materialization lifecycle, without resorting to the catastrophic disk I/O of a generic graph database. A truly robust DAG-BFT storage engine must maintain causal evidence on disk not just based on its inclusion in a committed leader's sub-DAG, but based on multi-stage lifecycle tracking:
Ordering Finality: Has the vertex been included in a committed sub-DAG? (Current DAG-BFT GC stops here).
Payload Materialization: Have all payload digests referenced by this vertex been fully downloaded from peers?
Execution Consumption: Has the execution engine processed the state transitions mandated by this vertex?
Archive Substitution: Has the network generated a certified checkpoint that covers the effects of this vertex, rendering the causal structural evidence obsolete for catch-up paths?
If a system attempts to implement this by storing granular dependency metadata—e.g., maintaining a graph database on disk mapping every vertex to every dependent transaction—it will succumb to the exact same write-amplification failure that plagued Geth's MPT reference counting. The constant metadata shuffling on disk will destroy the high throughput the DAG was designed to achieve.
The optimal solution, and the most fruitful research direction, lies in synthesizing the Aptos stale_node_index paradigm with Erigon's asynchronous pipeline logic and Sui's LSM compaction filters. Rather than updating dependencies directly on disk, the system could maintain an in-memory watermark array representing the execution and materialization frontiers of the node. A background RocksDB compaction filter could then asynchronously reap DAG vertices from the database only when their logical round falls safely behind the lowest watermark of the consensus, execution, repair, and checkpoint materialization pipelines.
Concluding Synthesis
The storage of causal evidence in high-throughput DAG-BFT systems presents a complex architectural challenge that transcends traditional blockchain state pruning. Linear blockchains benefit from a strict synchronization between logical ordering and physical materialization, allowing them to aggressively prune historical state using block horizons or epoch windows without impacting hot-path execution correctness.
While the framing of the hypothesis must acknowledge that existing DAG-BFT protocols actively manage causal graphs in memory and that distributed garbage collection algorithms form robust prior art, the core proposition holds immense merit. Current DAG-BFT implementations rely on round-depth or leader-commit garbage collection that is blind to execution-layer materialization lag. When logical ordering dramatically outpaces physical data availability, this blindness results in the premature deletion of causal evidence, forcing lagging replicas into slow checkpoint-recovery fallback paths and threatening network liveness.
Developing a storage engine that retains causal evidence based on its active utility across the entire consensus, execution, and repair lifecycle—while explicitly avoiding the catastrophic write amplification of on-disk reference counting—represents a highly valuable and necessary frontier in maximizing the resilience of decoupled DAG architectures.
DAG-BFT系统：热路径因果证据分析
主要对象： 在高吞吐DAG-BFT协议中，热路径上通常存在多种对象：包括提案（proposal）、区块（block）、顶点(Vertex)、证书(Certificate)、父引用(parent references)、交易批次(Batch)的摘要与载体(payload)、验证者间同步状态、修复请求等待队列以及执行/检索依赖等。例如，Narwhal系统中节点通过可靠广播(RBC)生成并分发“区块”和对应的可用性证书(certificate-of-availability)；Bullshark中每个轮次由各验证者广播含交易区块的顶点(Vertex)，并附带“强边”与“弱边”指向前一轮和更早轮次的顶点；Shoal和Mysticeti等也以此类DAG顶点为核心。而区块头(Header)、证书(Certificate)、交易摘要、鉴证(anchors)、检察证据(如双花证据)等均是热路径中的关键证据对象。
Narwhal/Tusk： Narwhal将交易批次通过可靠广播(RBC)传播，形成一个按轮次分层的DAG。每个RBC区块含有交易数据和对前一轮已认证区块（RBC证书）的引用。当节点收到2f+1签名的可用性证书时，该区块视为“可用”且可供Tusk协议消费。节点仅在新轮次开始时停止传播和丢弃未完成的RBC请求，因此旧区块可通过签名者重新获取。Narwhal设计者指出，可通过共识来确定垃圾收集轮次，将前几轮的区块卸载到CDN或外部存储。也就是说，Narwhal有意移除DAG-Rider中的“弱链接”以支持GC。正向引用：每个区块指向前一轮的证书（或区块）作为父；反向引用：协议一般不存显式子孙指针，只依赖转发或请求机制从签名者获取丢失数据。
Bullshark： Bullshark协议中，每个节点每轮构造并可靠广播一个vertex结构，包含交易区块和引用前一轮所有顶点的强边，以及若干弱边指向更早轮次中与该顶点未连通的节点。Vertex结构示例：
rust
复制
struct vertex {
  round r;
  validator_id source;
  Block block;            // 交易区块
  Set<Vertex> strongEdges; // 指向轮次 r-1 的顶点集合
  Set<Vertex> weakEdges;   // 指向轮次 < r-1 的顶点集合
}
在强路径检测中，只要存在通过强边连接到先前顶点的路径，则该先前顶点不包含在weakEdges中。因此正向引用：当前顶点引用其父轮次的所有顶点（强边）和必要的旧顶点（弱边）；反向引用：协议本身未显式存储孩子或依赖者的指针，但实现中可通过DAG查询路径或维护待添加缓冲列表。物化假设：Bullshark假设在提交协议中，区块随同顶点一起广播（顶点包含完整块体），若部分节点未收到某区块，可向签名该顶点的验证者请求恢复（类比Narwhal策略）。消费路径：一旦确定了锚点(anchor)或块可提交，该块将被交付给上层状态机。垃圾收集：Bullshark引入“时间戳”和“GC轮”机制：基于2f+1个签名者的中位时间戳，当超过一定延迟阈值时，将旧轮次的顶点清除；即每隔若干轮通过共识推进GCround，从DAG中移除早期顶点。如果保留错误地丢弃了仍有后继的顶点，可能导致后继无法最终提交。
DAG-Rider 与 Tusk：原始DAG-Rider允许每轮广播一个含任意过去轮次弱引用的提案，以实现最终公平性。这意味着必须长期保留所有历史区块（弱链接覆盖所有未强连通的节点），因此无法安全GC。Tusk（Narwhal+HotStuff）简化了DAG-Rider：它采用2f+1可靠广播移除弱链接，仅保留指向前一轮的引用，从而可支持垃圾收集。物化假设：两者都允许在未收到完整块体时先签发摘要和引用，然后通过RBC修复缺失的块。
Shoal/Shoal++： Shoal协议基于Bullshark结构，增加了多个并行“锚点(anchor)”和快速提交规则。其对象与Bullshark类似（节点/块+引用），另维护weak_votes数组跟踪间接支持度。Shoal允许某些节点被标记为锚点，一旦足够多提案连接到锚点就可“快速提交”。引用结构：节点引用其父节点，同样有轮次分层；证书(certified node)也存在，用于普通提交路径。反向引用：协议跟踪弱投票计数（weak_votes）来辅助提交，但不显式链接子节点到父节点。存储假设：与Bullshark一样，节点（可能证书前）在完整RBC后加入DAG，没有特殊链外修复提到。GC/保留：Shoal论文未专门提出GC规则，推测继承Bullshark思想，对已提交锚点的DAG历史可逐步淘汰（通过共识或归档）。错误保留旧节点不会破坏正确性，但可能浪费资源；丢失未提交节点会丧失其所表达的事务共识权。
Mysticeti (Sui)：Mysticeti-C采用单消息“已签名区块(signed block)”的DAG，不使用显式证书。每轮每个验证者多播一个包含父引用的区块，不需要再额外的投票证书。引用结构：每个区块包含对前一轮区块的引用（类似Bullshark强边），并可能包含交易的Merkle根或状态摘要以防双花。反向引用：协议未定义，通常通过哈希查找或DAG遍历实现。物化假设：无可靠广播，多播保证消息到达。但无论如何，如果少数节点未收到区块，可通过网络重传获取。GC/保留：Mysticeti目标低延迟，论文未具体说明GC策略。一般而言，已共识的区块可在合适时由快照或检查点替代，不影响协议安全。错误删除未决区块会导致安全故障（丢失已广播的提案）。
Cordial Miners：提出“Blocklace”数据结构，将数据、投票与排序信息混合于块中。没有可靠广播或外部证书机制；每个**块(block)**直接引用其父块并隐式承载共识信息。引用结构：块带有指向先前区块的部分有序链接。反向引用：不存在专门的子引用；依赖关系通过块之间交叉链接来体现。描述，区块融合了传播、排除双花和排序三大功能，无需额外元数据。GC/保留：Cordial Miners也未详细阐述删除策略，但由于每个块自身包含全部必要信息，只要保留链上的最后若干层块及块链都能进行验证。过早丢弃区块会丢失其内含的共识信息和可能的多引用，导致协议中断。
物化与修复路径：综上，各协议通常允许摘要/引用先入后补：验证者可先接收块头或证书进入本地DAG，再异步拉取缺失的交易块体或先前引用的顶点。恢复机制多依赖：Narwhal利用签名者存储保证，可以向签名者请求；Bullshark/Shoal未明言但类似；Cordial和Mysticeti因广播确保最终可达，大多省略专门的修复逻辑。证明/责任路径（如双花检测、可证明公平）通常依赖完整DAG日志或额外签名事件，在协议完成后可以下线验证，不在热路径重点分析范围内。
生命周期元数据与管控：目前大多数DAG-BFT协议并未维护丰富的长期生命周期元数据。Narwhal/Tusk可通过共识协商垃圾收集轮次；Bullshark通过局部时钟确定GC阈值；而Shoal/Shoal++、Mysticeti等对GC几乎没有规定。协议通常只凭回合、锚点或检查点来隐晦决定清理时机（例如Narwhal使用检查点替代部分历史）。这意味着现有系统对因果依赖的管理较弱：它们主要依据线性进度（轮次或区块高度）、时间/窗口或状态快照来裁剪数据，很少知道某证据被哪些后续块依赖，也不跟踪“责任窗口”。因此，若要实现因果感知的精细保留，需要在协议或存储层额外维护如依赖图、引用计数或消费标记等元数据。
区块链存储引擎与修剪策略对比
链式区块链客户端（以太坊、比特币、Aptos、Sui等）的存储模型与DAG-BFT热路径存储有本质差异。这些系统主要围绕线性链、状态快照和高度来组织存储和修剪，而非追踪协议内在的因果关系。
数据类别： 以太坊客户端（Geth、Erigon等）保存区块头、区块体（交易列表）、交易回执（以及日志）、账户/合约状态（Merkle Patricia Trie节点或平坦KV）以及快照信息等。比特币节点管理区块头、交易列表和UTXO集；可选地可丢弃老区块数据，仅保留UTXO集以答复状态查询。Aptos/Sui等新链则存储交易历史（Ledger）与认证状态（Jellyfish Merkle Tree、对象数据库）、事件/索引、以及周期性检查点（Epoch/Ckpt）数据。Celestia作为DA层保存所有交易数据和Blob，支持轻节点不同级别的数据留存策略。
修剪/保留策略： 以太坊中，全节点通过快照同步产生最新状态后，可离线执行状态修剪，移除超过一定深度的Merkle Trie节点。Geth使用快照层决定保留节点：至少保留最近128块对应的状态节点，其余视为陈旧并删除。Erigon在“完整模式”下默认执行状态修剪，也只保留当前状态；归档模式才保留全历史。Nethermind类似，执行“完全修剪”后只存储当前状态，依据阈值或手动触发来重建并替换状态数据库。Aptos提供配置窗口保留最近N百万笔交易的账本历史，对旧交易和事件予以丢弃，其状态则通过最近快照保全。Sui鼓励验证者/全节点启动修剪模式：删除已提交交易和已删除对象的完整历史，仅保留最新状态；相应地，历史查询需要依赖云存储的检查点。Celestia则引入“滚动窗口”修剪：节点按7天窗口只保障近7天数据可用，超出窗口的块由归档节点负责。总体来看，传统区块链依赖线性进度（区块高度/轮次）和时间戳/窗口确定丢弃点，或开启“归档模式”彻底保存全历史。
依赖/因果跟踪： 普通区块链对状态依赖采取简单方式：以太坊依赖Merkle Trie结构，通过根哈希保证完整性，而不显式记录各个节点依赖哪些交易。修剪时依赖引用计数：Geth的离线修剪使用快照对Trie节点打标，保留最近版本所需节点；然后删除不在快照集内的节点。但它并不追踪事务或块之间的依赖关系；比特币不跟踪交易在UTXO之间的详细关联，只需UTXO集合即可验证。Aptos的状态存储是原子化的Jellyfish Merkle Tree，可通过块高度索引，但并不为历史快照做详细引用计数；它的“备份系统”纯粹保留全历史以外。Sui使用对象存储，保留活跃对象以及对检查点有效的对象快照，不记录对象在各事务/区块中被如何修改的依赖图。Celestia仅保证数据可用性，未做任何交易依赖管理。反向引用： 这些系统通常无持续维护反向索引（例如交易->区块、Trie节点->块或根）。Erigon甚至选择不存储日志而在查询时重演交易，表明历史信息并非持久保存。唯一区块链提供反向查询的机制是通过辅助索引或全节点功能（如EVM日志索引、树的历史版本），但它们主要用于方便查询和数据分析，不影响核心共识或修剪逻辑。
执行感知保留： 传统客户端的修剪策略通常并不考虑交易是否被执行或归档。例如Geth离线修剪仅根据状态快照存在性删除Trie节点，不关心那些节点是否已经在重组(rollback)或执行队列中使用。Nethermind的“完全修剪”触发条件依据数据库大小或手动请求，与事务执行状态无关。Aptos/Sui的节点保存最新状态快照，但不跟踪哪个交易被哪些合并快照覆盖，只是简单丢弃超出窗口的数据。只有启用归档模式时，才确保不丢弃任何已执行数据。可见，这些链上存储引擎缺乏对“已执行&提交”与“尚需传递”状态的细粒度管理。它们往往默认为“只要块被链上确认，就可以安全丢弃相关原始数据”，而不维护额外的执行路径标记。
热路径与离线需求： 区块链的存储机制针对离线查询和状态读取进行了优化，如日志/事件索引、合约状态快照等，用于跨重启快速恢复。但对共识热路径（如当前轮次的数据依赖）几乎不提供支持：链节点不需要时即刻丢弃过期数据，没有实时因果依赖缓存的概念。热数据（最新状态、最新若干区块）被保存在快速存储中，而旧数据则通过归档节点、快照或链下备份维持可用。任何依赖更历史块的恢复都被视作离线操作。

发现差距： 以上系统均未维护针对协议因果依赖的专门元数据。它们将数据保留策略简化为线性指标（高度、窗口）或状态快照，缺乏“哪些后继依赖此数据”“是否已被重放/执行”之类的追踪。例如，Geth用快照复原Trie节点后再离线删除旧节点；Nethermind只关心当前状态而忽略过去的执行链。这意味着链式存储引擎不具备“因果价值”管理——一旦数据超出保留窗口，即使后续验证器可能需要它进行修复或验证，也会被丢弃或卸载。因此，仅依靠现有区块链存储机制不足以满足高吞吐DAG-BFT的因果感知保留需求。
云存储分层与生命周期管理
主流云存储和大规模层级存储（如AWS S3、Azure Blob、Google Cloud Storage以及传统HSM方案）主要按访问频率、对象年龄、成本等信号进行数据层级划分，而不具备理解区块链协议语义的能力。
层级信号： 云存储根据对象创建/最后访问时间、访问频率、大小、成本或用户定义的策略规则来决定何时迁移。例如，AWS S3的生命周期策略可基于对象创建日期自动转移到较低成本的存储类别；智能分层(Intelligent-Tiering)则监控访问模式：30天未访问转至低频层，90天后转入归档层。这些决策关注年龄和访问行为，并对超小对象（<128KB）给予特殊处理。对象标签(tags)、元数据可以作为触发条件，但这些标签由用户自定义，与应用语义无关。传统HSM方案也主要利用文件时间戳、大小、路径/所有权等策略输入，现代加入标签、工作流状态或成本信号。
未知因素： 这些系统不理解协议依赖。它们无法知道一个对象是否为某个活跃DAG根节点、是否仍有后续顶点指向它、是否已被执行或在检查点中被覆盖，也不认识拜占庭容忍窗口或区块提交情况。例如，S3不知道某个对象是区块数据还是随机文件，也无法推断“这个区块的子区块尚未到达，共识尚未完成”。因此，协议级别的依赖性（如验证者间等待、修复需求）不在这些系统考量范围内。
索引与元数据： 云对象存储提供标签系统、存储类及清单(inventory)等功能，支持审计和管理。例如S3清单可以列出桶中对象及其元数据，生命周期规则可针对标签或前缀操作，访问日志记录对象下载行为。但这些索引仅用于管理和合规，不提供复杂的图结构查询或可达性分析。对象元数据基本静态，缺乏针对协议演进的动态信息（如“已经应用到区块链状态”标记）。
可转移性： 一些概念可借鉴到DAG-BFT。比如访问频度类似“请求频度”，即若很多节点频繁请求某些缺失区块，则可以将其视为“热数据”并预留更多资源。生命周期规则对应可配置策略，比如“达到某个锚点后安全丢弃原始提案”。但是，云层级通常是异步且低频的操作（天级触发），不适合共识热路径对低延迟和即时性的需求。策略引擎（HSM中的核心）概念表明可以根据多维信号（包括标签、成本、大小）决定迁移；若能为DAG-BFT数据赋予类似标签（如“处于活跃轮次”“已完成执行”）并实时更新，或许可实现精细保留，但当前系统缺乏此类动态因果信息。
热点/寒冷对比： 传统热/冷存储更多关注“频繁访问 vs 长期冷静”，而因果价值分层则关注“协议进程中的活跃度”。例如，在DAG-BFT中，一个在验证窗口内的提案即使最后访问次数不多，也不应被删，因为它可能影响安全。相似地，如果一个交易块已被确认并检查点覆盖，则可以认为其因果价值降低，可进归档。可见因果价值层级考虑因素包括：是否为当前活跃根；未满足的子依赖数量；是否已被状态快照包含；是否在责任期内等。**创新性判断：**这样的因果驱动策略超越了简单的缓存替换：它需要协议层的元数据与运行时信息的紧耦合。现有层级存储只提供部分启发（比如访问频度间接反映活跃度），但缺乏明确的协议语义。可部分借用策略引擎思路（如依据标签迁移数据），但需在DAG-BFT上下文下动态生成和使用标签，如“活跃锚点ID”、“未执行交易计数”等。

Causal Evidence Storage in Storage-Backed DAG-BFT Systems: A Critical Analysis of Causal-Value-Aware Tiering
1. Introduction and Architectural Context
The evolution of Byzantine Fault Tolerant (BFT) consensus protocols has undergone a paradigm shift, transitioning from monolithic, strongly coupled architectures toward modular, high-throughput Directed Acyclic Graph (DAG) based systems. Foundational protocols such as Tendermint and HotStuff achieved consensus by sequentially proposing and finalizing individual blocks, tightly coupling the dissemination of transaction data with the cryptographic voting process required for total ordering. While elegant in their simplicity and capable of achieving sub-second finality, these systems suffer from severe throughput bottlenecks because the primary node must broadcast the entire payload to the network and gather a quorum of signatures before the protocol can advance.   
High-throughput DAG-BFT systems systematically eliminate this bottleneck by introducing a fundamental decoupling of logical ordering from physical evidence materialization. In protocols such as Narwhal, Tusk, and Bullshark, the architecture is bisected into a data dissemination layer (comprising multiple worker nodes) and a consensus layer (the primary node). Workers continuously stream transaction batches to each other and form certificates of availability. The primary nodes then construct a continuously growing DAG where vertices represent these certificates (or the batches themselves) and edges represent causal relationships, effectively acting as implicit votes. Because the consensus logic interprets the topological structure of this DAG to achieve a total order without requiring additional communication overhead on the critical path, the network can achieve unprecedented throughput, often exceeding hundreds of thousands of transactions per second.   
This decoupling, however, introduces a profound and previously under-theorized hot-path storage problem. Validators in a DAG-BFT network frequently order or reference cryptographic digests, block headers, certificates, and parent references long before all corresponding object bodies, batch payloads, repair dependencies, execution dependencies, or checkpoint metadata are fully materialized and consumed across the network. The situation is further exacerbated by the advent of uncertified DAGs, such as Mysticeti, which eliminate explicit certification to reduce good-case latency to theoretical minimums. In uncertified DAGs, validators build upon uncertified proposals, relying on implicit certification patterns to commit sub-DAGs. Consequently, validators accumulate a massive influx of causal evidence—cryptographic proofs of ancestry, equivocations, and partial payload availability—that must be retained in fast storage to guarantee asynchronous worst-case liveness, censorship resistance, and eventual fairness.   
The central hypothesis under evaluation asserts that this causal evidence should not be managed merely as a flat key-value (KV) history utilizing simplistic round-based, epoch-based, or time-based pruning. Instead, correct retention and tiering dictate a need for "causal-value-aware tiering." This tiering model requires the storage layer to possess epistemic awareness of protocol dependencies, identifying who depends on specific evidence, whether it remains an active protocol root, whether it has been consumed by catch-up or accountability paths, and whether it has been safely substituted by aggregate state snapshots. This report provides a critical, exhaustive examination of this hypothesis. It contrasts the posited causal-value-aware tiering model against large-scale cloud tiering systems, hierarchical storage management (HSM), data lake formats, and tiered KV stores, ultimately determining the transferability of these mechanisms to the consensus hot path.
2. Demarcating Causal Evidence Storage from Legacy Storage Paradigms
To rigorously evaluate the hypothesis surrounding causal-value-aware tiering, it is imperative to first establish the boundaries of the problem by distinguishing it from adjacent storage optimization challenges within distributed systems and blockchain architectures. The hot-path storage of causal evidence in a DAG-BFT system is fundamentally distinct from conventional historical pruning, graph analytics, and distributed file storage.
The most common point of confusion is conflating causal evidence storage with ordinary blockchain historical pruning. Traditional state pruning in monolithic blockchains is strictly a post-execution optimization. It involves discarding historical state tries or blocks that are no longer required for current state derivation or immediate block validation. For example, Ethereum developers have continuously optimized state trie pruning, with modern node implementations utilizing "Snap sync" to download recent headers and keep only the most recent 128 block states in active memory, relying on parallel state-sync and healing phases to reconstruct the local trie. Similarly, CometBFT (formerly Tendermint) utilizes a configuration parameter to retain a minimum block height, safely pruning older blocks to reduce disk usage. In these legacy systems, the pruning logic is predominantly driven by temporal or height-based thresholds linked to unbonding periods; for instance, Cosmos SDK chains often set their trust period to slightly less than the 14-day unbonding period to ensure equivocation evidence is maintained for slashing purposes. Causal evidence storage in DAG-BFT, by contrast, is a pre-execution and concurrent-execution problem. It deals with uncommitted, partially committed, or recently committed DAG vertices that are actively required for network liveness and topological traversal, rather than merely historical auditing or lightweight syncing.   
Furthermore, this storage problem must be distinguished from offline blockchain graph analytics. Graph analytics applied to blockchains operate on static, immutable data extracted from the chain to analyze transaction flows, tokenomics, or network structures. These systems operate entirely offline, extracting data into external data warehouses, and are completely unconstrained by the sub-second latency and synchronous disk I/O requirements of a hot-path BFT consensus mechanism. Similarly, generic graph databases, while natively designed to manage nodes and edges, are optimized for complex, multi-hop queries, property graph traversals, and pattern matching across massive datasets. They do not natively understand cryptographic non-equivocation, Byzantine fault thresholds, or the specific rules of commit wave propagation inherent to protocols like Bullshark or Mysticeti.   
Another critical distinction must be drawn between causal-value-aware tiering and ordinary RocksDB or Log-Structured Merge-tree (LSM) tuning. LSM-trees are the default storage engines for a vast majority of modern blockchains, including high-performance networks like Aptos. LSM-trees manage data as flat key-value pairs, buffering writes in memory and sequentially flushing them to disk as Immutable Sorted String Tables (SSTables), moving data through progressively larger levels via background compaction to mitigate read, write, and space amplification. Aptos, for instance, has engineered immense optimizations around its RocksDB implementation, separating the Jellyfish Merkle Tree (JMT) into multiple physical shards across different RocksDB instances to parallelize CPU core usage and minimize write amplification during execution. While RocksDB supports native tiered storage by hinting the file system to migrate "cold" SSTables to slower media based on data temperature , the LSM engine remains entirely agnostic to the causal value of the keys it stores. It compacts based on file size, deletion tombstones, and level thresholds, lacking the semantic awareness to prioritize a specific vertex digest simply because it serves as an active repair root for a temporarily partitioned validator.   
Finally, the causal evidence storage problem is entirely distinct from distributed file storage or Data Availability (DA) storage solutions. Systems like IPFS, Walrus (deployed alongside Sui), or generic DA layers provide robust, erasure-coded blob storage intended for long-term archival, disaster recovery, or censorship-resistant data availability. They are designed to solve the problem of retrievability and static immutable storage off-chain, ensuring that historical snapshots or sequences of checkpoints can be accessed without bloating the primary chain's state. These architectures are fundamentally unsuited for the highly dynamic, structurally interdependent lifecycle of a hot-path consensus DAG, where the relevance of an object shifts drastically based on the topological advancement of the graph in real-time.   
3. Architectural Analysis of Hyperscale Cloud Tiering Systems
To evaluate the hypothesis that causal evidence requires a bespoke tiering mechanism, one must analyze how hyperscale cloud systems and massive object stores handle data lifecycles, and specifically, what deterministic or heuristic signals they employ to trigger the tiering of data. Cloud storage environments have pioneered the concept of storage classes—balancing cost, durability, and retrieval latency—but their triggering mechanisms operate on vastly different timescales and assumptions than BFT consensus.
3.1. Signals Driving Cloud Tiering
Cloud storage systems rely heavily on highly generalized, backward-looking, or statically deterministic signals to transition objects across hot, warm, cold, and archive tiers. The most ubiquitous and elementary signal is the age of the object, defined by the wall-clock elapsed time since its creation or modification. In Google Cloud Storage (GCS), lifecycle rules frequently utilize the age condition, defined strictly as the number of days since the object was created. Similarly, AWS S3 Lifecycle policies transition objects to Standard-Infrequent Access (Standard-IA) or S3 Glacier based on a predetermined number of days post-creation configured by the bucket administrator.   
Beyond static age, the most sophisticated cloud tiering mechanisms monitor access frequency and recency. AWS S3 Intelligent-Tiering abstracts manual lifecycle rules by actively monitoring access patterns at the object level. It operates as an automated lifecycle management system that utilizes a 30-day consecutive non-access period to transition objects to an Infrequent Access tier, yielding immediate cost savings. If an object remains untouched for 90 days, it is automatically migrated to the Archive Instant Access tier, and further down to Deep Archive Access after 180 consecutive days of inactivity. This is a strictly backward-looking statistical signal: if an object has not been accessed recently, its probability of future access is presumed low. Actions such as GetObject, PutObject, or CopyObject immediately reset this consecutive inactivity timer, bumping the data back to the frequent access tier.   
Object size and cost thresholds also serve as primary signals for cloud tiering engines. Because monitoring object access incurs computational overhead, cloud providers impose minimum size constraints to ensure economic viability. For example, S3 Intelligent-Tiering does not auto-tier objects smaller than 128 KB. For these small files, the system charges a monitoring fee (typically $0.0025 per 1,000 objects per month) but keeps the objects in the frequent access tier, as the metadata management cost would negate any fractional storage cost savings achieved by tiering them down.   
Furthermore, cloud storage systems utilize policy rules, user tags, and explicit storage classes to govern data placement. GCS utilizes complex conditional logic such as matchesStorageClass, matchesPrefix, and custom object tags to route specific data types through distinct tiering pipelines. An enterprise might configure a rule that transitions objects with the prefix logs/ to a Nearline storage class after 30 days, and to a Coldline class after 90 days. Additionally, versioning states such as isLive or DaysSinceNoncurrentTime allow these systems to differentiate between the active head of an object and its deprecated historical versions, effectively allowing users to prune old data without breaking current application state.   
The choice of tier inherently dictates the retrieval latency model, which serves as a secondary signal based on application tolerance. Transitioning data from S3 Standard to S3 Standard-IA maintains millisecond latency identical to the hot tier, whereas transitions to deep archive tiers (like S3 Glacier Deep Archive) impose retrieval times ranging from several minutes to upwards of twelve hours. Applications must explicitly opt into these latencies, meaning tiering signals are always calibrated against the maximum acceptable delay for data retrieval.   
3.2. Epistemic Limitations: The Semantic Blindness of Cloud Tiering
While hyperscale cloud tiering systems are highly efficient at managing petabyte-scale data lakes based on statistical access probabilities and economic thresholds, they suffer from profound epistemic limitations when evaluated against the strict, deterministic requirements of DAG-BFT consensus.
Specifically, cloud tiering systems operate with complete semantic blindness regarding the internal structure of the data they store. They do not know or understand protocol dependency. They are entirely unaware that a specific object (representing a DAG vertex) is cryptographically and logically bound to another object (its parent vertex or batch payload). If a client requests a specific vertex, the cloud system cannot preemptively warm the parent vertices based on a causal link; it treats each object as an independent, isolated entity within a flat namespace.
Cloud systems fundamentally cannot identify active repair roots. In a distributed consensus network, if a validator suffers a temporary crash or network partition and subsequently reconnects, it must traverse the DAG from its last known committed state to synchronize with the current network head. The specific, mathematically verifiable subset of older vertices required to bootstrap this lagging validator constitutes a repair root. To a cloud tiering system relying on access frequency, these older vertices appear entirely "cold" and may have been transitioned to an archive tier. The system has no mechanism to recognize that this specific cold data has suddenly become mission-critical for network liveness.
Similarly, hyperscale object stores are ignorant of descendant fanout. A DAG vertex may be physically old—a trait that cloud systems would aggressively penalize via lifecycle demotion—but possess a massive descendant fanout, meaning a large portion of the subsequent DAG graph references it. In protocols like Narwhal or Bullshark, an older vertex might serve as an anchor for a commit wave or a critical weak link necessary for ensuring eventual fairness and censorship resistance. Cloud systems treat highly referenced structural nodes identically to isolated, orphaned nodes of the exact same physical age.   
Furthermore, these systems lack the capacity to understand execution dependency and accountability windows. DAG-BFT systems require retaining data not just until it is ordered, but until it is safely executed by the underlying state machine or until a specific accountability window expires. For example, evidence of Byzantine equivocation—such as a validator signing two conflicting blocks at the same height—must be retained and accessible until the offending validator is actively slashed by the protocol or passes its unbonding period. Cloud systems cannot dynamically tie an object's retention policy to the progressive execution of a state machine or the fulfillment of cryptographic slashing conditions.   
Finally, cloud tiering cannot accommodate checkpoint and archive substitution dynamically. In high-throughput architectures like Sui, data can be safely pruned from operational full nodes the moment it is substituted by an archival store (such as Walrus) and a cryptographic epoch checkpoint is formed. Cloud storage lifecycle rules cannot link the deletion of a primary object to the successful, verifiable cryptographic ingestion of an aggregate snapshot in an external system. They rely on static time delays rather than deterministic state transitions.   
4. Indexing, Metadata Mechanisms, and Data Lake Formats
To effectively support intelligent tiering and object lifecycle management, cloud systems and large-scale big data architectures rely on extensive out-of-band indexing and metadata mechanisms. Understanding these mechanisms is crucial to determining whether their architectural patterns can be retrofitted for DAG-BFT storage.
In hyperscale object storage, AWS S3 maintains internal inventories and processes continuous object access monitoring logs to compute the access frequency required for Intelligent-Tiering. Google Cloud Storage utilizes complex metadata catalogs to track object tags, custom metadata key-value pairs, prefix structures, and precise creation timestamps. These catalogs act as secondary indices, allowing backend batch processes to scan the bucket namespace and execute retention policies without opening the underlying data payloads.   
In the realm of big data analytics and data lakes, table formats like Apache Iceberg, Delta Lake, and Apache Hudi utilize highly sophisticated manifest architectures to solve the problem of metadata at scale. Apache Iceberg, for instance, replaces the traditional directory-based layout of data lakes with a hierarchical metadata structure. It maintains a root metadata file that points to a manifest list, which in turn points to individual manifest files that track the actual underlying data files (often stored in Parquet format). These manifests include rich column-level min/max statistics that enable advanced partition pruning during query planning. To manage the lifecycle of this rapidly accumulating metadata, Iceberg employs specific maintenance operations. The expire_snapshots procedure removes older, unnecessary snapshots from the metadata tree, followed by an "orphan file cleanup" process that physically deletes the underlying data files that are no longer referenced by any active snapshot.   
Similarly, Delta Lake utilizes a transactional log to track changes to the table state. To prevent the storage layer from bloating indefinitely, Delta Lake relies on the VACUUM command. This operation identifies and permanently removes data files that are no longer referenced by the Delta transaction log and are older than a specific retention threshold (defaulting to 7 days, or 168 hours). Apache Hudi addresses metadata scaling by introducing a specialized LSM-based timeline. This timeline tracks all actions performed on the table, and Hudi utilizes a compaction process to periodically merge row-based log files into columnar base files to optimize snapshot query read latencies, alongside an archival service that moves older entries from the active timeline into an archived log.   
While these indexing and metadata mechanisms are exceptionally powerful for batch-oriented analytical workloads and multi-petabyte data lakes, their temporal assumptions clash with the reality of DAG-BFT. Iceberg's compaction and snapshot expiration, or Delta Lake's VACUUM, rely heavily on periodic, scheduled maintenance jobs—often executed via Airflow dags or cron jobs running hourly or daily. They operate asynchronously to the primary write path. In a DAG-BFT consensus engine, delaying garbage collection to a batch process creates unbounded memory and disk growth during high-throughput stress, risking out-of-memory crashes and severe I/O stalls.   
5. Hierarchical Storage Management and Tiered Key-Value Stores
Moving closer to the node-level storage engines utilized by blockchain validators, Hierarchical Storage Management (HSM) and tiered key-value stores attempt to optimize data placement across heterogeneous media—such as DRAM, NVMe SSDs, and slower, high-capacity HDDs. These systems bridge the gap between application memory and persistent storage.
Academic and research systems frequently explore intelligent HSM to mitigate the massive cost of keeping all application data in memory or on expensive enterprise SSDs. Systems like ReStore utilize reinforcement learning approaches to automate page migration across multi-tiered storage, dynamically adapting to shifting workload characteristics without human intervention. Other approaches focus heavily on cost-aware data placement, formulating algorithms to ensure that frequently accessed "hot" data strictly resides on fast, expensive storage, while "cold" data is systematically relegated to slower media, optimizing the global cost-to-performance ratio of the server cluster.   
In the context of localized key-value stores, RocksDB's multi-level LSM-tree is inherently a tiered architecture. However, standard LSM compaction suffers from significant performance variability, write amplification caused by data rewriting during compactions, and read amplification resulting from multi-level queries. Because standard RocksDB compaction is entirely driven by physical file sizes and sequential level thresholds rather than application logic, it frequently compacts and buries data that the application might soon need, leading to cache misses and high read latency.   
To counter this, emerging research systems attempt to implement "application-driven" storage tiering. For example, HotRAP is a modified key-value store built atop RocksDB that explicitly tracks the "hotness" of individual keys. Instead of relying solely on background level-based compaction, HotRAP includes specialized pathways to individually promote hot records from slow storage back to fast storage, keeping them there as long as they remain frequently accessed. Other paradigms, such as SANCache or application-driven memory tiering systems like X-Mem, augment disk array caches or heterogeneous memory layers by providing custom APIs that allow the application to dictate Quality of Service (QoS) requirements or explicit data placement based on anticipated usage.   
Despite these advancements in HSM and application-aware KV stores, these systems still operate primarily on statistical access frequency metrics—variants of Least Frequently Used (LFU) or Least Recently Used (LRU) algorithms. They optimize the storage layout based on the heuristic of "what is being read the most," rather than the deterministic logic of "what mathematically must be read to satisfy a distributed protocol constraint." They lack the topological awareness required to map a DAG structure directly onto the storage media.
6. Causal-Value-Aware Tiering versus Ordinary Hot/Cold Tiering
The core of the user's hypothesis rests on clearly distinguishing "causal-value-aware tiering" from ordinary hot/cold tiering models. This distinction is robust, theoretically sound, and addresses a genuine structural gap in current distributed systems storage literature.
In ordinary tiering architectures—whether implemented in an S3 bucket or an LSM-tree—value is derived strictly from temporal proximity (physical age) or statistical popularity (access frequency). In the proposed causal-value-aware tiering model, the "value" of a data object is determined entirely by its topological position within an asynchronous cryptographic graph and its absolute necessity to the progression of the state machine.
6.1. Defining Causal Value
Causal value is not an abstract concept; it can be rigorously defined and quantified by several deterministic, protocol-specific parameters within a DAG-BFT system:
Active Root Status: The most critical determinant of causal value is whether a vertex represents an active root. If a vertex has been proposed but has not yet been included in a committed sub-DAG by the consensus leader, it is an active protocol root. Its causal value is maximal, regardless of its physical age or how rarely it has been accessed in the last few hundred milliseconds. If the storage engine arbitrarily prunes an active root because it crossed a temporal threshold, the validator violates protocol rules and halts its ability to participate in consensus.   
Descendant Fanout: A vertex that serves as a parent to numerous subsequent proposals possesses exceptionally high causal value. Even if the vertex itself is no longer actively read for its internal payload, its cryptographic digest is highly necessary for path validation and traversal during commit waves. In protocols like Bullshark, an anchor vertex in an odd round dictates the commitment of all its causal history. The physical storage layer must recognize this topological weight.   
Missing-Request Frequency: While standard systems measure read frequency, a DAG-BFT storage engine must measure specific missing-request frequency. If a specific block header experiences a sudden, coordinated spike in fetch requests from peers, it does not indicate popularity; it indicates a hole in the network's data availability layer. The causal value of this object spikes instantly, demanding immediate retention in the hottest tier to serve the network and prevent cascading liveness failures.
Validator Lag Distribution: If the local node observes that honest validators are suffering from severe latency or network partitions, the causal value of the specific historical vertices required to bridge their lag window increases. The tiering mechanism must proactively retain this evidence in the fast NVMe tier to facilitate rapid, non-blocking catch-up once connectivity is restored to those specific peers.
Unexecuted Committed Slice: Data that has been ordered by the consensus engine but not yet executed by the state machine retains high causal value. In highly pipelined architectures, such as Aptos's Zaptos or Sui's execution engine, transaction execution is separated from consensus ordering to maximize hardware utilization. This implies that ordered evidence must remain in ultra-fast storage until the execution delta is persistently written; tiering it to a slower medium mid-pipeline would catastrophically degrade throughput.   
Checkpoint/Archive Coverage: Conversely, the causal value of a vertex drops to absolute zero the exact moment it is proven to be covered by an aggregate cryptographic checkpoint or safely ingested by an archival node. In Sui, once an epoch concludes and a checkpoint is generated, the underlying DAG state prior to that checkpoint becomes causally irrelevant to the hot path. The storage engine must use this protocol event as a deterministic trigger to drop the data, rather than waiting for an LRU cache eviction.   
Accountability Window: The causal value of conflicting proposals—equivocations—remains strictly tied to the protocol's unbonding or slashing windows. Equivocation evidence must outlive normal garbage collection boundaries. Once the adversarial validator is successfully penalized via the protocol's governance or slashing mechanism, the evidence's value expires.   
6.2. Novelty Assessment: Beyond Cache Replacement
The framing of causal-value-aware tiering is genuinely novel when applied to the persistence and storage layer of distributed consensus systems. It is critical to distinguish this concept from "semantic caching," a term often used in modern software architecture. Semantic caching typically exists at the application layer, where caches serve data based on query similarity or LLM vector embeddings, matching user queries with previously computed responses based on contextual meaning rather than exact key matches. Semantic caching is a probabilistic read optimization technique.   
Causal-value-aware tiering differs fundamentally from both semantic caching and generic LRU/LFU cache replacement policies across two primary axes:
First, it relies on forward-looking determinism rather than backward-looking heuristics. Traditional cache replacement predicts future access based entirely on past access patterns. Causal-value tiering determines future necessity based on absolute topological determinism. If a vertex is required to complete a commit wave according to the DAG rules, its probability of future access by the consensus engine is exactly 1.0, rendering statistical heuristics entirely moot.
Second, the stakes involve protocol safety versus mere performance. Evicting the wrong object from a traditional cache simply causes a cache miss, resulting in a minor latency penalty as the data is fetched from disk. Evicting the wrong causal evidence in a DAG-BFT system before it is checkpointed or executed can lead to unrecoverable local halts, violation of asynchronous liveness, or the inability to generate fraud proofs regarding Byzantine behavior. Therefore, this concept is not merely cache replacement with different features; it represents the deep integration of Byzantine state-machine semantics directly into the physical placement logic of the storage engine.

8. Critical Assessment: Analogy versus Solution in the Consensus Hot Path
The user's hypothesis—that DAG-BFT systems require a hot-path storage solution distinctly decoupled from flat KV history due to the separation of logical ordering from physical materialization—is structurally sound, theoretically robust, and addresses a documented, critical bottleneck in the field of distributed systems architecture.
Modern production implementations are already violently colliding with the limits of flat KV stores. The development teams behind state-of-the-art DAG-BFT networks are actively engineering around these limitations. For instance, the Anoma network developers explicitly flagged unbounded growth in their Narwhal DAG, batch store, and Mempool, noting that premature garbage collection below the watermark of honest validators effectively acts as a Byzantine fault, highlighting the severe safety implications of improper causal storage management. Furthermore, the designers of the Mysticeti protocol made a highly deliberate engineering choice to avoid key-value stores like RocksDB entirely for their Write-Ahead Log (WAL). They cited the absolute necessity of eliminating the associated structural overhead and the unpredictable I/O penalties of periodic background compaction, opting instead for a custom implementation utilizing memory-mapped files and vectored writes to achieve maximum throughput. This real-world rejection of the industry-standard LSM-tree validates the premise: generic, causally blind storage engines lack the semantic alignment necessary for cutting-edge DAG-BFT.   
However, a critical assessment must distinguish between mechanisms that can be directly imported and those that serve merely as conceptual analogies.
Is Cloud Tiering an Analogy or a Solution?
Cloud tiering and data lake management paradigms serve strictly as an analogy and a conceptual framework, not a direct, drop-in solution for the consensus hot path. The systems are fundamentally mismatched in terms of latency, determinism, and safety assumptions.
What Should Be Borrowed:
The Metadata Abstraction Layer: Just as Apache Iceberg abstracts millions of underlying Parquet files into manageable, hierarchical manifest lists , a causal-value-aware storage engine should abstract raw DAG vertices into structural manifests grouped logically by round or epoch. When a protocol like Mysticeti or Bullshark commits an anchor and advances its deterministic garbage collection round—typically moving the GC frontier to a set number of rounds below the highest committed round —the storage engine should simply deprecate the corresponding round manifest. This instantly releases the underlying payload storage in a single operation, bypassing the need to execute millions of individual key deletion tombstones in a KV store.   
Explicit Storage Classes for Object Components: Cloud systems enforce rigorous separation of storage classes based on utility. In DAG-BFT, this translates to separating the routing layer from the bulk data. Vertex headers, cryptographic signatures, and parent references constitute the critical topological routing layer (which must reside in the highest-performance memory/NVMe tier), while the actual transaction batch payloads constitute the bulk data. By materializing payloads directly into a secondary storage tier upon receipt and keeping only the references in memory, the system naturally resolves hot-path bloat without sacrificing liveness.   
Deterministic State Toggles (The isLive Concept): GCS's versioning model utilizes the isLive boolean state to dictate tiering. In DAG-BFT, a vertex is strictly "live" until it is committed and executed by the state machine. Once the protocol advances past the deterministic pruning threshold, its status toggles atomically, immediately triggering physical transition or deletion without waiting for a background sweep.   
What Should Be Avoided:
Wall-Clock and Access-Frequency Triggers: Time-based rules (e.g., transition if age > 30 days) are toxic to asynchronous consensus state machines. In truly asynchronous networks, a partition could theoretically last indefinitely; data must be retained based on absolute topological progression (e.g., waiting for signatures to form a certificate), never based on arbitrary wall-clock time.   
Background Compaction and Asynchronous Sweeping: The standard LSM compaction model causes unpredictable I/O spikes (write amplification) that compete with the consensus engine for disk bandwidth. Garbage collection and tiering in DAG-BFT must be completely deterministic and executed inline with the consensus commit rule. Outsourcing storage cleanup to external, asynchronous background processes creates dangerous race conditions against the rapidly advancing state machine.   
Agnostic Key-Value Interfaces: Treating DAG evidence as independent, isolated KV pairs completely obscures their fundamental graph relationship. A causal-aware storage system must avoid generic KV APIs. Instead, it must provide an interface natively fluent in the topological semantics of the DAG, understanding "parents," "children," and "weak links," allowing the consensus engine to recursively delete or tier an entire uncommitted sub-DAG via a single, atomic storage operation.
In conclusion, the problem of causal evidence storage in high-throughput DAG-BFT systems cannot be adequately addressed by generic cache replacement algorithms, flat LSM-tree compaction, or age-based cloud tiering mechanisms. Because protocols like Narwhal and Mysticeti structurally decouple the logical ordering of references from the physical materialization of payloads, they generate a highly volatile, graph-based hot path. The proposed framework of "causal-value-aware tiering" represents a necessary architectural evolution. By defining data retention not by age or historical access frequency, but by topological necessity, the storage layer is elevated from a passive data repository to an intelligent, structural extension of the Byzantine fault-tolerant state machine. While hyperscale cloud storage systems and data lake formats provide excellent analogies for metadata grouping and manifest-driven lifecycles, realizing causal-value-aware tiering ultimately requires abandoning the causally blind key-value interface in favor of a bespoke storage engine natively integrated with the topological semantics of the consensus DAG.
高吞吐量DAG-BFT系统的因果证据存储
背景： 高吞吐量的DAG-BFT共识（如Narwhal/Tusk、Bullshark、Shoal等）通过将逻辑排序与实际证据交付（如区块体、事务批次、修复/执行依赖等）解耦来提高吞吐。验证者可能在实际获取对应数据前，只交换其摘要、头、签名证明等，从而引入因果证据管理的“热路径”问题。下面汇总各系统的设计要点，重点关注热路径上的对象、引用关系、物化假设、回收规则等。
一、DAG-BFT协议体系
Narwhal: DAG型Mempool协议。热路径对象：每轮(r)广播一个区块(block)，含交易批次哈希和对前一轮所有区块的引用；以及对应的可用性证明证书（certificate-of-availability，包含至少2f+1个签名）。工作节点还产生事务批次(batches)，主节点在其区块中以哈希形式引用。**前向引用：**区块→其父区块证书列表；区块→批次哈希；证书→对应区块。**反向引用：**未显式维护（可通过扫描所有区块间接获得孩子列表）。**物化假设：**允许先达成区块证书(证明其可用)，再通过从签名者处“拉取”缺失的区块体完成数据。证书存在意味着≥2f+1份副本，可以请求这些签名者提供数据。**修复路径：**收到可用性证书后，缺失区块由证书签名者回答获取。**消费路径：**共识收集区块摘要和证书后才进入排序（Tusk/HotStuff），执行时再取回区块数据。执行层负责将事务写入状态。**保留/回收：**Narwhal使用严格的轮(GC round)机制：在共识协议中达成一致后，可将早期轮的区块证书从主存中移除；这些历史区块可被卸载到离线存储（CDN）。Narwhal论文表明，仅保留当前轮的区块和证书即可保证功能正确；旧轮数据由协议决定何时垃圾收集。**元数据：**主要为当前已同意的GC轮号；无执行/归档标记。Narwhal不跟踪证据是否已被执行或归档，它假设执行后可通过快照/CDN离线复原。
Tusk: 基于Narwhal DAG的异步BFT共识算法。Tusk每个区块还携带门限签名片用于分布式随机硬币，不需额外通信。热路径对象与引用：基本同Narwhal（区块、证书、批次哈希），另有门限签名份额。每轮按波(wave)分三阶段，2f+1链接至某个父块时即提交锚(b)，然后输出该锚的因果片段。引用方式同Narwhal：每区块仅引用上一轮顶级区块（无弱边）。**物化假设：**依赖Narwhal的广播机制，仍可先根据证书排序再获取区块体。**回收：**未具体描述，但可假设沿用Narwhal轮GC。Tusk代码也使用RocksDB持久化结构，但未关注回收细节。**元数据：**门限硬币片、GC轮信息。Tusk不提供执行完成或归档标记，与Narwhal类似。
Bullshark: Narwhal/Tusk的实现改进（Aptos Labs）。**热路径对象：使用Narwhal的顶点(vertex)**表示每轮区块及附加投票信息。每个顶点包含：区块数据、轮号r、发送者ID、对2f+1个前轮顶点的强边（parent references，即上一轮Quorum leaders）；以及快速路径(fallback)的投票信息（f+1个备用票），和正常路径(steady-state)的票。**前向引用：**与Narwhal相同，每顶点引用前一轮的区块；**弱引用：**可能存在快速/备用链路，用于commit fallback。**反向引用：**不显式维护孩子表；链路信息存于顶点自身。**物化：**依赖可靠广播得到区块；排序时检查已收到的所有引用区块后再提交。若顶点数据缺失，可疑未详述（代码实现中已储存和持久化顶点结构）。**消费：**当满足提交条件(≥f+1 fallback或2f+1正常票)时，将提交的领导者顶点压入栈，然后调用order_vertices输出所有该领导者的前驱（因果切片）。此输出包括Leader本身和其所有未交付的祖先。**回收/GC：**Bullshark提出基于时间戳的回收算法（见算法6），每顶点带时间戳(ts)，根据Leader提交的时间戳与各轮内顶点中位数差值，推进GC轮号并清除旧轮DAG。GC规则非简单轮号，而是时间差驱动。**元数据：**每轮顶点的时间戳和当前GC轮号保存在本地；已交付顶点集。Bullshark也持久化使用RocksDB。如保存失误则引发共识分歧（故需一致进行GC）。Bullshark不跟踪执行或归档状态，提交后顶点即被视为交付，不再保留其依赖元数据。
Shoal++: 在Narwhal/Tusk基础上改进，通过轮替领导者和流水线技术降低延迟。实现上仍使用Narwhal的区块+证书结构，只是领导者选取及提交方式不同，对象与引用与Narwhal大体相同（区块、证书、父引用）。Shoal关注高可用和低延迟，没有提出新的存储对象或索引机制，主要改变协议层次（如“锚”（anchor）机制）。其论文指出使用Narwhal和Bullshark所用的RocksDB存储；GC机制沿用Bullshark的时间戳方法。Shoal并未描述额外的保留元数据。
Mysticeti: 一个并发工作（Sonnino等人）的设计，采用未经认证的DAG和改进的Cordial Miners方法。参考资料表明Mysticeti移除了区块可用性证明链路，从而在网络部分同步时更高效。实现上，每个顶点仍带有事务块和父指针，但提交靠多个领导者。物化与修复：因取消可用性证明，块可能只有部分节点可见，若链上缺失块，其他节点需要按需请求（类似IPFS拉取）。Shoal++论文指出“由于缺少可靠广播，链中可能出现部分丢失块的情况”。Mysticeti尚无公开存储细节，但显然不跟踪执行或归档，重点依赖协议层弹性。
Cordial Miners: 提出了“块镶(Blocklace)”结构，用于异步原子广播。每个顶点内含一个交易块和多个指向先前顶点的引用（可能是“强边”），类似DAG-Rider。它没有证书证明，而是通过领导者块（按轮或随机选择）进行提交；在每个“波(wave)”最多一个领导者块被确定。**对象与引用：**和其他DAG-BFT一样，顶点(Blocklace节点)、父指针、可能的门限签名。无可用性证书；传播机制称为“Cordial广播”，减少了可靠广播开销。**保留：**Cordial系列论文没有专门讨论状态存储或GC，主要关注算法性能；可认为它们需要在本地存储全部DAG直至提交完成，不提供更细粒度回收。
**各系统对象与引用汇总：**上述系统主要涉及以下热路径对象：
**区块(Block)/顶点(Vertex)：**包含事务或交易集合，拥有标识和轮号，并引用前一轮或多轮的父区块（强引用）。如Bullshark顶点还可能携带投票、随机数签名等。
**证书(Certificate)：**Narwhal/Tusk引入的区块可用性证明（多签），用于标记区块可用。提交后的证书则成为下轮区块的前向引用。
父引用(Parent References)：通常为指向上一轮Quorum块的指针，在拓扑上构建DAG。DAG-Rider有强边(指上一轮≥2f+1块)和弱边(指更早轮块)；Tusk/Bullshark移除了弱边，只用强边。
**事务批次(Batch Digests/Payloads)：**Narwhal体系下，工作节点生成事务批次，每个批次有哈希被包含在区块中。批次内容独立存储并在需要时拉取。
**提交锚(Committed Anchors)：**如Bullshark或Cordial中领导者块确认后，作为合并输出基准。
**检查点元数据(Checkpoint Metadata)：**Narwhal/Bullshark等并未明确定义检查点机制；一些实现（如Sui等）在更高层加入检查点，但DAG协议本身不直接管理。
**修复等待者(Repair Waiters)：**未在文献中明确提及，但当区块缺失时，各节点会“等待”并从签名者或其他节点请求此块，可视为临时等待状态。
**双花/问责证据(Equivocation Evidence)：**某些协议（如PoS系统）可能跟踪节点双花，此处暂不讨论，大部分DAG-BFT假设签名唯一。
引用关系：所有系统均有前向引用（区块→父区块、证书→区块、区块→事务批次哈希等），但**反向引用(子指针)**通常不显式维护，而是通过搜索或计算获得。例如Narwhal或Bullshark不存储“哪个区块依赖我”，而是依赖DAG结构自行推导。若需要加速查找，例如“谁引用了缺失的区块”，可以为每个区块维护一个子列表，但这会增加写开销且协议论文中未提及。一般而言，反向引用多为“隐式存在”。
物化假设与修复路径：除了Narwhal/Tusk使用可靠广播保证区块最终可达外，其他协议（Bullshark、Shoal、Cordial等）也依赖类似的推送或拉取机制。在这些系统中，允许先达成摘要或证书入共识，然后再下载数据体。例如Narwhal中仅凭可用性证明对块进行排序，再从2f+1签名者处拉取缺失内容。Shoal/Mysticeti通过模拟Narwhal或快捷路径广播；Cordial理论中则需调用“Cordial广播”或直接从网络请求。在消费路径上，共识层排序完成后，相应的区块事务被执行写入状态，这时才真正“消费”这些证据。系统通常不会检查证据是否已执行，而认为提交即可丢弃或归档。
回收/GC规则：大部分协议采用轮/时间驱动的回收策略，与传统区块链历史修剪有所不同。Narwhal规定“共识一致的GC轮”作为界限；Bullshark则用时间戳差值来推进GC；Shoal未明言但可参考Bullshark；Cordial/Mysticeti论文中未见实质GC方案。总体来看，这些协议仅记录轮号或时间戳作为持久元数据，不跟踪“哪些证据仍有活跃依赖”。它们假定在安全条件下可在统一的轮界限前清理，无法基于真正的因果关系精细保留。
**生命周期标记：**这些系统普遍不跟踪诸如“已修复”“已执行”“已归档”“已截图”等信息。Narwhal通过可用性证明隐含“已复制”，但并不保留额外标记。Bullshark代码仅持久化DAG节点及时间戳；当顶点提交输出后，即在内存中标记为已交付，不再依赖。至于是否已被活跃领导者引用、或被归档替代，DAG-BFT共识协议本身没有明确信号，更多依赖于上层执行环境（如状态机检查点）。
二、区块链存储引擎与修剪机制
为了对比，下面分析若干区块链存储系统如何分类存储数据、实施修剪，以及是否考虑因果依赖。特别关注的是它们是否仅按线性时间/状态管理数据，或是否引入额外依赖追踪。
Ethereum 客户端 (Geth、Erigon、Nethermind、Reth 等)：
存储数据类别： 块头(block headers)、区块数据(block bodies)、交易、收据(receipts)、状态Trie节点(或平坦状态)、日志和事件索引等。例如Geth持有完整区块链数据和状态trie节点；Erigon等支持Archive（全节点）或Pruned模式，分别存储全部或有限历史数据。
修剪策略： 通常按区块高度或已确认状态进行修剪。例如比特币节点可删减旧块，只保留UTXO集；Ethereum的Geth默认使用「状态快照」策略，仅保留当前状态和最近128块的Trie节点。Erigon可配置prune.mode，默认仅保留合并后的状态，并清理旧块数据。Finalization 时，老块在节点端被扔掉（Full/Archive模式除外）。有些Beacon链节点按epoch修剪旧状态。
依赖追踪： 这些系统主要关注状态或交易历史值，不维护诸如“哪些旧状态正在被后续区块依赖”的显式元数据。Ethereum状态Trie节点有引用计数来识别仍被活跃区块引用的节点，但这仅用于安全地删除不再需要的枝节点。没有类似2-hop索引来追踪状态或区块之间的因果依赖。
反向引用： 不显式维护。Ethereum中没有反向索引（例如从状态节点倒查属于哪个事务）。部分区块链会有查询索引（比如事物ID到其所在区块），但那是针对历史查询，不用于GC策略。
执行感知： 区块链节点通常“执行一次就算完成”，并不会特别标记已执行或已归档。状态Trie节点引用计数隐含该节点已被用于某区块的状态根；一些节点软件允许导出快照以离线访问历史，但节点本身不跟踪哪个节点执行过。
示例 – Geth： Geth使用快照(snapshot)来确定哪些Trie节点需要保留，其他老节点可清除。它不删区块，只清理过时的状态节点。
示例 – Erigon： Erigon将数据分为多层：chaindata（链头）和执行状态分区。它支持分阶段存储：合并前的合并后状态、日志索引等。通过配置可完全归档或根据需要修剪历史。
示例 – Sui (Move链)：
存储数据类别： 对象（账户/资源）、事务、事件、状态变更日志、检查点(checkpoint)等。
修剪策略： Sui按epoch周期进行修剪。验证者节点只保留最近一个epoch的数据和索引，大约数百GB。历史全节点才持久整个对象历史。对象状态在检查点上被“安全覆盖”，并将归档到云存储快照。
依赖追踪： Sui的存储主要按对象版本设计，对象有严格的线性版本链，不在存储中维持反向链。索引服务器可以查询对象的历史版本和事件，但节点本身不跟踪日常执行依赖。
反向引用： 无。
执行感知： 只有当事务和对象被包含在检查点中时才认为已“消费”。Sui通过检查点bucket在云端保留状态，以供落后节点加载。节点内不跟踪具体事务是否被应用，只有由共识确定的序列。
示例 – Aptos (Move链)：
存储数据类别： 账本(ledger)交易、状态树节点、事件等。文档称默认只保留最近1.5亿笔交易的历史，旧交易会被修剪。物化状态会通过交易再执行产生。
修剪策略： 通过交易数量窗口或Epoch剔除历史（可用--pruning-enabled或配置关闭即可存档全历史）。目前仅对交易历史做轮次修剪；状态Trie则保留当前状态。
依赖追踪： 无专门元数据，仅依赖账本连续性。不会维护“哪些旧交易对当前状态依赖”，因为状态存储已通过根哈希保证正确。
执行感知： 交易执行后即更新状态，不再特别保留痕迹；同理，无归档标志，仅用交易高度界定清理界限。
示例 – Bitcoin (UTXO):
存储数据类别： 区块头、交易、UTXO集合。
修剪策略： 可以开启区块链历史削减模式，仅保留UTXO和最近数百区块（用于重组）。旧区块体被删除，链高度作为裁剪基准。
依赖追踪： UTXO集合自己包含所有未花费输出，未显式追踪交易间依赖，仅隐式通过输入引用UTXO。无附加依赖元数据。
执行感知： “交易执行”在比特币中就是花费UTXO，节点不额外标记已执行或已归档交易（完成即视为可以丢弃旧记录）。
示例 – Celestia (数据可用性层)：
存储数据类别： 数据块(blob)、区块头和数据可用性承诺(DAC)。Celestia不维护交易状态。全节点根据Erasure编码存储交易数据。
修剪策略： 主要保留一定时间的完整区块数据以便验证可用性；过期区块可丢弃。与传统链不同，Celestia的节点不维护经济状态，无需查找依赖。
依赖追踪： 无状态依赖，仅保证可以验证数据可用性。
执行感知： Celestia本身不执行交易，仅负责数据广播。
共性总结： 这些传统区块链系统主要按时间/区块高度切割存储，不维护数据依赖图。它们关心的是“状态是否被固定在某个根”而非“哪些块仍为活跃依赖”。因此它们的修剪规则通常基于高度、时间或窗口，而非数据的因果角色。我们没有找到任何主流区块链系统在热路径存储中引入复杂的因果元数据——它们依赖于链式结构和状态快照来保障一致性。
关键问题对比： 区块链存储引擎与DAG-BFT的差异在于前者是线性有序的历史记录，后者是并行的DAG结构。区块链的修剪更依赖于链高度、最终状态根、检查点（如Epoch）等线性概念，而DAG-BFT需要考虑活跃DAG结构中的各节点依赖。前者对于完成交易已有成熟做法（如UTXO删除、Trie节点引用计数、状态快照），但缺乏对区块间或证据间依赖关系的管理。这意味着，现有区块链系统并未提供“因果依赖意识”的元数据：它们没有专门结构去记“哪些过时数据仍被活跃状态或后续块依赖”。这种差异使得简单照搬线性链的修剪机制到高吞吐DAG-BFT系统时存在不足；DAG系统可能需要更精细的依赖追踪来安全清理数据。
三、云存储分层与生命周期管理
云服务商的存储分层机制主要基于对象访问模式和生命周期，而非协议级的因果依赖。以下总结各家云存储的分层信号及元数据管理：
AWS S3 存储类别与生命周期：
信号： 主要依据对象的访问频率和最近访问时间来自动移动存储层。S3 Intelligent-Tiering监测对象最近30、90、180天无访问情况，自动将数据从“频繁访问层”切换到“低频”或“归档”层；使用生命周期规则可以在指定条件下转移到特定存储类（标准、低频、Glacier等）。对象大小（<128KB）影响监测资格。此外，成本和延迟也是决策因素（例如归档层需要更长恢复时间）。
元数据： S3存储类/生命周期本身是元数据。还有对象标签(Tags)和版本ID可被生命周期策略使用以选择性应用规则。此外，S3提供**清单(Inventory)**功能列出桶中对象以辅助管理。访问日志(Access Logs)和CloudTrail可用于分析访问模式。
不知晓： S3不理解数据内容或协议意义，不会知道“该对象是否正被某DAG活跃引用”或“事务是否执行”。它只关注通用指标如时效、频率。无概念“协议根”“修复根”等。
检索延迟： 标准访问层毫秒级，冷归档层（Glacier）可达小时级。
可用性/一致性： S3保证高可用性和强一致性，无需同步问题。
热路径适用性： S3针对的是离线/近线热度管理，不适合实时共识热路径（其自动分层通常以天为单位）。智能分层更像离线优化工具，而非热路径即时索引。
借鉴与限制： 可借鉴“访问热度作为信号”，但对于DAG-BFT，需要更精细的“因果热度”（如未能按时获取的请求计数）。S3风格的标签和生命周期规则也可借用（给批量对象打标签以规则回收），但实际实现需节点参与判断依赖关系。总体而言，云分层关注频度/年龄，与DAG因果需求并不匹配。
Google Cloud Storage 生命周期：
信号： GCS生命周期规则可根据对象年龄(创建时间)、最后修改时间、版本状态以及对象名称前缀/后缀等条件转移或删除对象。用户定义的版本标签或命名前缀也可作为条件。
元数据： 每个对象有元数据字段 (如storageClass、timeCreated、updated、ACL等) 可被生命周期策略匹配。版本号(noncurrent)也可作为生命周期条件。GCS提供对象清单(inventory)和访问日志。
不知晓： 类似S3，它只看对象属性，无法感知协议相关依赖或执行状态。
热路径适用： GCS生命周期自动性，通常在后台异步运行，对热路径同样太慢。规则主要供运维规划使用。
借鉴： 支持按模式规则选择对象迁移，可用来如同给证据打“存储类”标签；但依然是静态规则，需与协议状态结合使用。
Azure Blob 生命周期管理：
信号： 基于创建时间、最后修改时间(默认)和上次访问时间(可选启用)来决策。规则定义“在x天后冷却”“在y天后删除”之类动作，还可使用路径前缀或Blob索引标签作为过滤条件。对象被访问时可以触发立即重新上移到热层。
元数据： Blob的元数据包括时间戳和用户标签。生命周期策略是托管在存储帐户层面的JSON。Azure会监控规则并分批执行。
检索延迟： 冷层访问需异步“恢复”，有数小时的延迟；热/热频层近实时。
限制： 策略执行最高延迟可达24小时，不适合实时需求。
借鉴： Azure的Blob索引标签类似于自定义标签，用来标记对象类别。规则可实现条件式的分层策略，但本质仍是时间驱动，与DAG活跃度概念不同。
层次存储管理(HSM)研究：
信号： 老式HSM和分层存储系统通常依据访问频率(“冷热”数据)、大小、策略定义进行迁移。部分研究还考虑数据温度模型（热度随时间衰减）。
元数据： 通常需要维护“访问计数”、“上次访问时间”这样的统计；高级方案可能用机器学习来预测迁移。
适用性： HSM理念可借鉴分层和成本考虑，但通常是离线过程。与因果价值相比，HSM更偏向传统的访问统计而无协议语义。
类比与结论： 云存储的分层管理基于年龄和访问频率来决策迁移，利用对象标签、前缀、策略规则等不活跃性指标，并没有因果依赖意识。它们不知道协议中的依赖关系（谁依赖谁、验证路径、补偿请求等）。对于DAG-BFT而言，这些机制启示了使用标签和日志来估计访问需求的思路，但它们的速度和指标并不足以支持热路径：例如，S3的跨层移动通常以天为单位；Azure/GCS也是周期性后台任务。这些都过慢，且缺乏细粒度控制。因此，应借鉴它们“多层存储”与“策略规则”思想，但需要设计专门适合共识系统的快速元数据跟踪和访问频率预估。实质上，因果价值(tiering)与热/冷策略不同：前者可能需要考虑“该区块是否仍未被下游消费？”、“多少个后续块依赖它？”等语义信息，而这些在普通存储系统里没有。
四、图数据库与可达性索引技术
DAG-BFT系统中常见的查询类型包括：“给定某个区块/摘要，有哪些后继或引用它的块？（反向依赖）”“给定一个提交锚，获取其因果前驱的全部顶点（因果切片）”等。这些查询涉及图可达性和局部子图提取。下面评估现有图数据库和索引方案的相关性与可行性。
实际查询需求：
Missing Reference Query： “给定一个丢失的区块摘要，谁在引用它？”需要从DAG中查询其直接子节点。
Causal Slice Query： “给定一个提交锚顶点b，找到它所有未交付的祖先顶点”，即从b向上遍历整个祖先集。
Blocked Descendants： “给定一个修复根（fallback root）r，哪些后续顶点因其丢失而阻塞？”需要查询以r为前驱的部分子DAG。
Active Roots前沿： “给定当前GC或共识前沿，哪些顶点仍被活跃锚引用？”也涉及根据锚或轮号查询相关子图。
高因果值证据： “在存储压力下，哪些区块/顶点因其影响范围大（子代多、依赖多）应优先保留？”这需要评估顶点的下游传播(后代数量)或被多少锚依赖等信息。
是否需要完整图DB：
这些查询大多是局部的可达性查询，不需要图数据库的完整功能。可以通过针对性的辅助索引或遍历来满足。例如，直接维护“每个顶点的父列表”就可迅速查询其前向引用；要找后代可以反向遍历，但可通过辅助索引来加速。真正的图DB功能（复杂的图遍历查询优化、实时交互式图分析等）对共识热路径而言过于笨重。
候选索引分析：
digest -> parents(父引用)： 已内建于每个区块/顶点记录，可快速获得该顶点的所有父节点，无需额外索引。
digest -> reverse_refs (children): 如果节点数巨大，维持完整的反向邻接表会有写放大（每写入一条边要更新儿童列表），而且高并发写时需同步；对热路径查询有利（找谁引用我），但空间开销和维护成本大。可以考虑仅对“热门”或未决区块维护此索引。
digest -> 对象状态/层级: 记录该区块是否已交付、所在存储层（本地内存/RocksDB/归档）等，帮助判断是否需要保留。
anchor/round -> 因果切片: 将提交锚对应的祖先集预先或增量计算；但存储所有可能的切片太大，通常在输出时BFS/DFS计算。可以简单记录“已提交锚列表”，在查询时顺序遍历DAG。
digest -> 活跃锚记录: 跟踪哪些未决的锚或领导者引用该区块，可用以判断是否仍需要该区块。需要更新开销。
payload_digest -> 执行/归档状态: 如果批次(事务组)被执行或归档，可以标记。对DAG-BFT来说，多半无需，因为执行层独立。
digest -> 后代传播估计: 可以维护或估计每区块的后代数量（影响力）。例如，通过级联更新“每个新的区块到其所有祖先的后代计数+1”，但这是全局性的昂贵操作，且易受恶意大扇出攻击。可能用近似方法（如HyperLogLog）统计不同深度的子树大小，但复杂度高。
拓扑/区间标签： 将DAG节点按照某种拓扑顺序赋予标签（例如预序区间），使可达性查询变成区间包含判断。这在静态树或有向无环图中可行（“树覆盖(Tree Cover)”索引）。但DAG每插入节点都需更新标签，非常昂贵。类似GRAIL算法为静态场景设计，动态更新难以实时处理。
近似可达性(Reachability Sketches)： 例如GRAIL、Ferrari等，用多个随机DFS号进行快速判断；只能给出“可能到达/不可能”二分类，需要在线回溯验证。动态维护也复杂。
二跳标签(2-hop labeling)： 每个节点存储两组标签集以回答任意到达查询（A能否到达B只需检查交集）。这是高效查询的理论方案，但构建和更新代价极高，不适合热路径频繁写入的环境。
开销与可行性：
昂贵索引： 上述动态构建的跨越标签(如2-hop)或全量可达率(如传递闭包)对存储和更新要求极高，写入时需要触发全局更新，显然不适合高吞吐的共识环境。行范类的动态索引几乎都不可行于此类工作负载。
副本放大： 维护反向边表（Digest→Children）将使每写入一条边都需额外写入，增加RocksDB压力；在高并发下可能成为瓶颈。
内存占用： 完整子图存储或全域标签会占用大量内存，与减少内存占用、快速应答的目标相悖。
一致性： 并发插入新顶点意味着索引必须可并发更新；复杂索引难以保证一致性（特别是在分布式存储中）。
对抗情况： 恶意节点可以构造大量交叉引用块，迫使索引维护爆炸（例如使每新块与大量旧块建立依赖，导致反向索引暴涨）。
动态DAG可达性： 图论社区对动态可达性问题有研究，但方案多针对静态或半静态场景。例如，DLR等提出分层标签，能应对增量插入，但实现复杂。与DAG-BFT场景不同，一般研究对象不是如此高频插入且需要立即可用的负载。现有动态可达性索引大多性能一般，不适合百万TPS的共识热路径。此外，DAG-BFT的回收和层级快照（Checkpoint）意味着图也在经历删除或归档，这对索引维护构成挑战。
对比：
区块链离线分析： 基于已持久化的DAG大规模图查询如链上分析，可以用Neo4j等图引擎离线运行复杂查询（例如基于Tx DAG的溯源查询）。但这些场景容忍较高延迟，不在共识热路径。
在线图数据库： 例如Neo4j、TigerGraph等支持快速邻接查找和路径查询，但其事务一致性和写入开销大，多数并非为高并发写优化。用于DAG-BFT意味着需要大量存储自定义逻辑，并不轻量。
轻量KV/LSM元数据： 更现实的做法是保留简单辅助索引：父引用已有、可考虑局部维护反向引用表，或使用缓存/布隆过滤来快速判断可能的子节点列表。整体依赖于简化需求：例如，如果目标只是“who needs block X？” 可以检查活跃锚和等待的验证者消息，而不是全图扫描。

Causal Evidence Storage and Dynamic Reachability in High-Throughput DAG-BFT Systems: An Architectural Analysis
1. Introduction: The Evolution of DAG-BFT and the Storage Bottleneck
The landscape of Byzantine Fault-Tolerant (BFT) consensus protocols has undergone a paradigm shift, transitioning from leader-centric architectures to Directed Acyclic Graph (DAG) based systems. Traditional protocols, such as PBFT and HotStuff, inherently couple data dissemination with logical ordering, placing a disproportionate bandwidth and computational burden on the elected leader. This centralization creates significant head-of-line blocking and underutilizes the aggregate network capacity. High-throughput DAG-BFT systems—most notably Narwhal, Bullshark, Shoal++, and Mysticeti—shatter this bottleneck by fundamentally decoupling the physical dissemination of transactional payloads from the logical consensus ordering mechanism.
In these advanced architectures, validators operate continuously and concurrently. They broadcast batches of transactions, acknowledge the batches of their peers, and form cryptographic hash references to prior batches. This continuous, parallel dissemination constructs a causal history graph—either a certified DAG, where vertices require supermajority quorum signatures (as in Narwhal and Bullshark), or an uncertified DAG, where vertices are merely signed by their proposer to minimize latency (as in Mysticeti and Lemonshark). Consensus logic is subsequently executed locally over these identical graph views, deriving a total temporal order with zero or minimal additional message overhead.
While this decoupling unlocks extraordinary throughput—often exceeding 100,000 transactions per second (TPS) with sub-second latencies —it precipitates a profound and novel challenge at the persistent storage layer. Because validators dynamically order and reference metadata (digests, headers, certificates, and parent references) asynchronously, they routinely process topological structures before the corresponding object bodies, payload batches, or execution dependencies are fully materialized or locally verified. Consequently, logical ordering strictly relies on structural precedence—the causal evidence represented by the DAG edges.
The central hypothesis driving this investigation accurately diagnoses the resulting dilemma: high-throughput DAG-BFT systems face a critical hot-path causal evidence storage problem. Causal evidence cannot be managed merely as a flat Key-Value (KV) history utilizing simplistic round-based, epoch-based, or time-based pruning. Correct retention of this evidence necessitates topological awareness, specifically requiring the system to know which delayed nodes depend on specific evidence, whether the evidence is anchored by an active protocol root, whether it has been consumed by execution and catch-up paths, and whether it has been safely substituted by authoritative state snapshots. This report delivers an exhaustive, critical analysis of this hypothesis, evaluating the necessity of graph database paradigms, assessing dynamic reachability algorithms, and defining the optimal lightweight indexing abstractions required to sustain DAG-BFT throughput over Log-Structured Merge-tree (LSM) architectures.
2. Critique of the Hypothesis: Prior Art and Theoretical Framing
The proposition that causal evidence in a DAG-BFT system requires topologically aware retention is technically sound and urgently relevant. However, a critical evaluation requires situating this hypothesis within the broader context of distributed systems, database theory, and runtime memory management. The fundamental problem of preserving causal evidence while pruning unreachable or consumed data is not entirely novel; rather, it is a specialized manifestation of distributed garbage collection operating under an adversarial threat model.
2.1. Prior Art and Counterexamples
To distinguish this research direction from generic storage problems, it is necessary to examine how analogous systems address causal retention.
Content-addressed Merkle DAG systems, such as Git and the InterPlanetary File System (IPFS), provide the most direct structural analogs. In Git, the commit graph forms a Merkle DAG where retention is determined strictly by reachability. When the git gc process is invoked, the system does not simply delete commits older than a certain date; instead, it performs a graph traversal starting from active roots (branches, tags, and the reflog). Any object not transitively reachable from these active roots is deemed "cruft" and isolated for expiration. Similarly, IPFS relies on manual "pinning" to define active roots that protect specific subgraphs from localized garbage collection. Merkle-CRDTs (Conflict-free Replicated Data Types) also leverage content-addressed Merkle-DAGs for causality information, utilizing DAG compaction mechanisms that preserve causal integrity without requiring global consensus on the pruned elements.
Furthermore, the concept of retaining objects based on "active protocol roots" directly mirrors the tracing garbage collection algorithms utilized in managed runtime environments like the.NET Common Language Runtime (CLR) and the Java Virtual Machine (JVM). In these environments, the garbage collector assumes all heap objects are unreachable and constructs a graph of reachable objects originating from active application roots (e.g., current stack frames, static variables). Unreachable objects are subsequently marked for termination.
Therefore, the user's framing is conceptually identical to tracing garbage collection. The novelty and the core research contribution lie not in the concept of active roots, but in executing a distributed, Byzantine-tolerant, persistent tracing GC over an LSM-tree at 100,000 TPS.
2.2. The Failure of Flat KV Watermark Pruning
The hypothesis correctly asserts that ordinary blockchain historical pruning—which typically relies on depth or low-watermark deletion—is catastrophically inadequate for DAG-BFT. Network asynchrony guarantees that honest validators will process information at varying speeds. If a validator aggressively drops older vertices based on a globally observed round watermark, it risks breaking the causal chains required to validate delayed, incoming blocks.
This exact failure mode materialized as a severe safety vulnerability in the Mysticeti-FPC implementation deployed on the Sui blockchain. Mysticeti accelerates consensus by utilizing uncertified DAGs and pipelined rounds. However, the initial garbage collection mechanism was configured to update the GC round immediately upon the commitment of a leader, prematurely pruning uncommitted blocks from older rounds. This hasty DAG-GC destroyed the causal history necessary to validate delayed transactions. Consequently, validators recounted votes based on fractured causal histories, leading to disagreements on transaction acceptance and enabling potential double-spend vulnerabilities.
This incident definitively proves the hypothesis: correct causal retention in DAG-BFT inherently requires structural, graph-aware metadata. However, framing the solution as a search for a "graph database" is too broad. The solution space must be aggressively restricted to lightweight, LSM-compatible metadata serving highly specific query patterns.
3. Formalizing Hot-Path Graph Queries in DAG-BFT
To evaluate the utility of graph indexing concepts, the specific graph queries required by the DAG-BFT hot path must be formally defined. The system architecture dictates that vertices represent proposals or batches, and directed edges represent cryptographic hash references to parent vertices from preceding rounds.
The following queries represent the mandatory hot-path operations required to maintain consensus safety, ensure liveness during asynchrony, and manage storage pressure.
Query 1: Unblocking Pending References (Reverse Reachability)
In a decoupled dissemination model, a validator routinely receives a vertex $V$ that references a parent digest $P$, but $P$ has not yet arrived over the network. The validator must suspend $V$ in a pending state. When the network eventually delivers $P$, the system must instantly determine which pending vertices are unblocked by the materialization of this parent. In graph terminology, this is a single-hop reverse edge lookup (identifying the children of $P$). Without this query, validators would be forced to continuously poll the pending pool, consuming excessive CPU cycles and stalling the execution pipeline.
Query 2: Causal Slice Extraction (Bounded Traversal)
When the asynchronous consensus logic commits an anchor (a designated leader vertex), the protocol must establish a total deterministic order of all historical vertices that contributed to that anchor. This requires the system to extract the complete causal slice—the transitive closure of parents—starting from the anchor and traversing backward until reaching previously committed anchors. This query maps to a bounded backward Breadth-First Search (BFS) or Depth-First Search (DFS) yielding the reachable sub-graph.
Query 3: Blocked Descendant Estimation (Transitive Fanout)
When a validator experiences a transient partition and falls out of sync, it must fetch missing historical vertices (repair roots) from peers. If multiple repair roots are missing, the validator requires a heuristic to prioritize fetching. Estimating the volume of subsequent network traffic dependent on a specific repair root provides this heuristic. This is fundamentally a forward transitive closure cardinality estimation. If a pending root has massive descendant fanout, it represents a critical unblocking path and must be prioritized for fetching.
Query 4: Active Root Tracing (Multi-Source Reachability)
To reclaim storage safely and avoid the Mysticeti vulnerability, the system must identify a Garbage Collection (GC) frontier. However, before a vertex $V$ below the round threshold is deleted, the system must mathematically guarantee that $V$ is not causally required by an uncommitted, active protocol root currently residing in the mempool. This requires answering a multi-source reachability query: does there exist a directed path from any node in the set of Active Roots to the target vertex $V$?
Query 5: Causal Value and Tiering (Topological Annotation)
Under extreme storage pressure, the underlying RocksDB instance will saturate, requiring aggressive flushing of data to cold or archival storage. The system requires a topological heuristic to differentiate high-value structural metadata (which must remain in hot RAM/NVMe for consensus routing) from low-value, fully consumed payload data. This requires querying node degree and traversal metadata annotations to assign a causal value score to each vertex.
4. Architectural Analysis: Graph Databases vs. LSM Stores
Having formalized the hot-path queries, the architectural inquiry must address whether these operations justify the integration of a full graph database, or if they can be satisfied by lightweight DAG-aware metadata over KV/LSM stores.
4.1. The Fundamental Incompatibility of General Graph Databases
General-purpose graph databases such as Neo4j, TigerGraph, JanusGraph, and Dgraph are engineered to execute arbitrarily complex pattern matching, unbounded multi-hop traversals, and deep property graph analytics. They manage data utilizing heavy adjacency lists, property encoding schemas, and complex locking mechanisms to ensure ACID compliance across mutating topological shapes.
Applying a full graph database to the DAG-BFT hot path introduces unacceptable systemic friction.
First, the write amplification is catastrophic. Graph databases operating over Key-Value stores—such as JanusGraph or specialized implementations like Poly-LSM—must map graph properties to distinct column families. Inserting a single DAG vertex with five parent edges may trigger multiple distinct KV insertions: one for the vertex property, five for the forward edges, five for the reverse edges, and several for secondary property indices. In an LSM-tree architecture like RocksDB, this translates to immediate write amplification. At 100,000 TPS, this amplification forces continuous leveled compactions, saturating disk I/O, triggering severe write stalls, and collapsing consensus throughput.
Second, general graph databases optimize for unpredictable query patterns (e.g., "Find all nodes matching property X connected to node Y via relationship Z"). DAG-BFT query patterns, conversely, are strictly deterministic, strictly append-only (until GC sweeps), and temporally localized.
Third, the concurrency overhead is misaligned. Graph DBs enforce strict locking to handle concurrent vertex mutations. In DAG-BFT, vertices are cryptographically immutable once created. The overhead of transaction management in a graph DB is entirely wasted on a workload that is fundamentally an immutable, append-only log.
4.2. Contrast: Offline Blockchain Graph Analytics
It is crucial to distinguish hot-path storage from offline blockchain graph analytics. Systems that perform off-chain analytics utilize ETL (Extract, Transform, Load) pipelines to move finalized blockchain data from the consensus node into heavy-duty analytical engines like Apache Spark or Neo4j. These systems execute heavyweight OLAP queries to analyze address clustering, token flows, and interaction topologies. This analytical model operates entirely outside the consensus critical path. It assumes the total order is already finalized, meaning latency constraints are irrelevant. Attempting to force the hot-path consensus engine to perform the duties of an analytical graph database conflates two fundamentally incompatible system design goals.
4.3. The LSM-Native Metadata Approach
The required queries (bounded reachability, reverse lookups, topological tiering) can be entirely satisfied by narrow secondary indexes built directly into the consensus execution layer, persisting to a highly tuned RocksDB instance. By maintaining specific metadata fields—such as reverse reference pointers and topological generation numbers—the execution engine can perform single-hop lookups directly via RocksDB prefix iterators, and multi-hop traversals via rapid, in-memory caching of active frontiers. A general graph database is an architectural mismatch; narrow, application-aware indexing over an LSM tree is the optimal paradigm.
5. Exhaustive Evaluation of Candidate Indexes
To design the optimal lightweight metadata schema, various algorithmic indexing concepts from graph theory and database engineering must be rigorously evaluated against the adversarial constraints of DAG-BFT.
5.1. Forward and Reverse Edge Indexes
The baseline requirement involves mapping digest -> parents and digest -> reverse_refs.
The digest -> parents index is inherently satisfied by storing the serialized vertex itself, as the vertex cryptographically binds its parent hashes. However, the digest -> reverse_refs index is the critical component for resolving Query 1 (unblocking pending vertices). Because a node's children accumulate dynamically as the network broadcasts asynchronously, updating a single, serialized list of children in a KV store requires a read-modify-write cycle. At high concurrency, this causes massive lock contention and write amplification.
The optimal implementation in an LSM architecture utilizes a dedicated RocksDB Column Family (CF) storing adjacency components as distinct keys. The key format should be constructed as `` with an empty value. RocksDB's prefix seek mechanism can efficiently retrieve all children by seeking the Parent_Digest prefix. This transforms reverse reference tracking into append-only writes, perfectly aligning with LSM mechanics.
5.2. Object Status and Tiering Indexes
The indices mapping digest -> object status/tier/location and payload_digest -> execution/archive status are highly effective for physically separating logical ordering from bulk storage.
Once the execution layer processes a vertex and commits its state transitions, the bulky payload data (the actual transactions) becomes largely irrelevant to the ongoing consensus logic. Maintaining a simple KV status flag allows the system to eagerly prune the massive payload batches while retaining the lightweight, 32-byte digest within the DAG structure. This satisfies future structural validations and causal slice queries without consuming significant disk space.
5.3. Topological Labels and Generation Numbers
The most critical indexing insight applicable to DAG-BFT is the implementation of topological labels, specifically generation numbers.
In Git's commit-graph, traversal speed is radically optimized using a "generation number" (the topological level of a commit). If commit A has a lower generation number than commit B, it is mathematically impossible for A to reach B. Git utilizes this property to short-circuit graph walks.
In DAG-BFT, the consensus protocol already organically enforces a topological level: the Round Number (or wave number). The fundamental rule of the DAG is that every vertex in round $r$ must strictly reference vertices in round $r-1$ (or earlier, depending on the specific protocol variant).
By utilizing the round number as a strict topological label, the execution engine can definitively bound all causal slice traversals (Query 2). If a validator is executing a backward BFS to extract the causal slice of an anchor at Round 10, it can safely and immediately terminate any search branch that hits a previously committed anchor at Round 9. This mechanism prevents the unbounded $O(V+E)$ graph walks that otherwise plague naive implementations and provides extreme resilience against adversarial fanout attacks.
5.4. Ordered Causal Slice Index
The query mapping anchor/round -> ordered causal slice represents a materialization of the traversal output. While the generation number allows fast traversal, computing the causal slice repeatedly during catch-up operations wastes CPU cycles.
Instead of treating the causal slice purely as a runtime computation, the system should treat it as a memoized index. Once an anchor is committed and its causal slice is deterministically ordered by the consensus rules, the resulting sequence of digests should be serialized and stored under the anchor's key. When offline nodes request catch-up data, the serving node simply reads the pre-computed slice index rather than re-traversing the DAG. This significantly reduces the computational overhead of serving synchronization requests.
5.5. Active Root Records
To prevent premature garbage collection, the system requires a digest -> active root records mapping. However, persisting this inverted index on disk incurs severe write amplification, as the set of active roots shifts constantly with every new wave of proposals.
The optimal approach is to maintain an in-memory concurrent hash set of "Active Roots"—representing the frontier of uncommitted, valid vertices currently residing in the validator's mempool. Rather than tracking which roots point to which digests, the GC mechanism operates asynchronously off the critical path, initiating a tracing sweep from the Active Roots down through the DAG. Any vertex not marked by this sweep (and falling below the low-watermark) is flagged for deletion. This mechanism is vital for maintaining GC safety under high network asynchrony.
5.6. Descendant Fanout Estimation
The query mapping digest -> descendant fanout estimate is intended to optimize repair fetching. While computing the exact forward transitive closure cardinality is computationally prohibitive, maintaining an exact index is unnecessary.
A simple heuristic proxy is sufficient: the system can track the raw integer count of direct reverse references (1-hop in-degree). If a vertex is referenced by a massive number of immediate children, it is highly likely to be a critical dependency. This requires only a lightweight atomic counter updated during the prefix-seek reverse reference insertion, avoiding the need for deep graph analytics on the hot path.
5.7. Reachability Oracles (2-Hop, Interval Labeling)
Academic literature proposes extensive indices for static and dynamic reachability, such as TF-Label, which computes 2-hop covers or post-order intervals over the DAG to answer reachability queries in $O(1)$ time.
These oracles are fundamentally incompatible with the DAG-BFT hot path. Reachability oracles require intensive, global pre-computation. While dynamic variants exist, their amortized update costs are too high for a system ingesting 100,000 TPS. Furthermore, interval labeling assumes a relatively static backbone; the highly concurrent, append-only nature of an asynchronous DAG invalidates interval bounds instantly upon the arrival of delayed, concurrent vertices. Recomputing these labels would stall the consensus pipeline.
5.8. Approximate Reachability Sketches
Another prominent theoretical approach involves embedding sequence bloom trees, k-min-wise independent permutations, or random walk estimators (e.g., ARROW) into vertices to probabilistically test reachability.
While Bloom filters have zero false negatives, they inherently suffer from false positives. In a Byzantine consensus context, relying on probabilistic sketches is perilous. A false positive reachability test might lead a validator to erroneously conclude it possesses the causal evidence required to commit a sub-graph, only to fail deterministically during actual state execution. Furthermore, as the DAG deepens, the bit-array of the Bloom filter saturates rapidly unless it is impractically large, degrading its utility for deep historical queries. Precise structural ordering protocols, such as Multi-Round Visibility (MRV), rely on absolute deterministic structural precedence to derive precedence constraints; probabilistic evidence cannot satisfy these rigorous safety proofs.
6. Dynamic DAG Reachability in the BFT Workload
A central aspect of the original hypothesis involves the applicability of dynamic DAG reachability algorithms. Theoretical computer science offers a robust literature on fully dynamic algorithms for maintaining transitive closures under edge insertions and deletions.
6.1. Theoretical Fit vs. Practical Computational Reality
The DAG-BFT workload is highly asymmetrical, divided into two distinct phases:
Insertion Phase (Hot Path): Massive, highly concurrent edge insertions as thousands of vertices are broadcast per second.
Deletion Phase (GC Path): Bulk, sub-graph deletions as causal histories are committed, executed, and aged out of the active root scope.
Maintaining a fully dynamic transitive closure matrix requires $O(n^2)$ space and $O(n^2)$ or $O(m + n \log n)$ amortized update times per edge operation. In a blockchain network committing up to 500,000 vertices per second (as targeted by optimizations like Mysticeti-C), updating an explicit reachability matrix for every new edge is computationally impossible.
Even specialized amortized, insert-only dynamic reachability structures fail when confronted with the deletion phase. Decremental reachability (maintaining reachability properties under edge and vertex deletions) is notoriously complex, often requiring decremental Depth-First Search algorithms or the complete re-evaluation of strongly connected components.
6.2. The RocksDB Deletion Penalty
Translating these dynamic reachability algorithms to persistent storage introduces fatal I/O bottlenecks. In LSM KV stores like RocksDB, "deleting" a vertex or an edge is not an instantaneous operation. RocksDB utilizes Tombstones to mark records as deleted. Executing a point delete for every vertex and edge in a pruned causal history injects millions of tombstones into the MemTable and subsequent SSTables. When iterators attempt to traverse the active graph, they must parse and skip these tombstones, severely degrading read performance.
While RocksDB offers the DeleteRange API to drop sequential keys (which is theoretically highly optimized for bulk data removal), its practical application in distributed systems is perilous. As observed by advanced distributed databases like TiKV, DeleteRange can be unstable under high concurrency, delay actual disk space reclamation, and break snapshot consistency during concurrent reads.
Consequently, dynamic reachability indexes that require constant, fine-grained point deletions to maintain their properties are antithetical to LSM mechanics. The optimal approach to causal retention must sidestep decremental reachability entirely. This is achieved by partitioning DAG data into Epoch-based Column Families or dedicated SSTables. Once the asynchronous GC sweep confirms that all active roots have advanced past a specific epoch boundary, the system can utilize DeleteFilesInRange or drop the entire Column Family, reclaiming storage instantly without tombstone penalties or breaking snapshot isolation.
7. The Adversarial Threat Model and Countermeasures
When designing indexing abstractions for causal evidence, the architecture must account for the Byzantine threat model, which actively exploits vulnerabilities in graph traversal and retention logic.
Adversarial Fanout and Deep Chains: In a benign workload, graph traversals are generally predictable. In a BFT environment, a malicious proposer can intentionally craft vertices that reference extremely old, un-pruned vertices, forcing honest nodes to maintain active links across massive temporal gaps. Alternatively, an adversary can create a deep, linear chain of "garbage" vertices in a single round, circumventing normal width-based heuristics.
If a validator attempts to compute a causal slice (Query 2) or perform active root tracing (Query 4) without strict algorithmic bounds, the adversary forces the validator to load thousands of useless vertices from the LSM disk into memory. This leads to immediate Out-Of-Memory (OOM) crashes, severe CPU starvation, and a complete loss of liveness for the targeted node.
Defensive Indexing Strategy:
To neutralize these vectors, the indexing abstraction must enforce rigid structural boundaries:
Strict Bounding via Generation Numbers: Any DAG traversal must strictly terminate based on the round number of the target anchor. The topological label acts as a cryptographic firewall against deep-chain traversal attacks.
Absolute Evidence Horizons: As implemented in advanced fair-ordering protocols like MRV (Multi-Round Visibility) , the protocol execution layer must enforce a hard limit on the depth of acceptable references. If a newly proposed vertex references an ancestor located beyond the configured "evidence horizon" (e.g., more than $k$ rounds in the past), the vertex is deemed structurally invalid and rejected prior to storage insertion. This structurally limits the maximum possible depth of any graph query to $O(W)$, where $W$ is the horizon width, strictly bounding the worst-case RocksDB read overhead.

9. Architectural Synthesis and Recommendations
The hypothesis under investigation—that high-throughput DAG-BFT systems require distinct, graph-aware mechanisms for causal evidence storage—is fundamentally validated by the engineering realities of systems like Bullshark and Mysticeti. However, the pursuit of generic graph databases, heavy dynamic reachability oracles, or probabilistic sketches to solve this problem represents a misdirection of engineering effort.
The DAG-BFT storage challenge is defined by two brutally opposing forces: the necessity for relentless, extreme-throughput data ingestion, and the necessity for meticulous, structure-aware garbage collection. The system must ensure delayed evidence remains securely available for asynchronous safety protocols, execution dependencies, and Byzantine accountability. Generic graph indices collapse under the weight of the insertion rate, triggering fatal LSM write stalls. Conversely, simplistic KV time-based pruning collapses under the realities of network asynchrony, leading to catastrophic safety failures, such as premature causal history destruction observed in early uncertified DAG deployments.
Strategic Recommendations for Prototype Development
For a research prototype aiming to resolve the causal evidence storage bottleneck in high-throughput DAG-BFT systems, the following architectural synthesis is definitively recommended:
Reject Heavyweight Abstractions: Explicitly avoid integrating Neo4j, JanusGraph, Dgraph, or dynamic transitive closure matrices. Their write-amplification penalties, locking overhead, and decremental deletion complexities will obliterate the precise throughput gains achieved by the DAG-BFT decoupled architecture.
Exploit Native Topological Properties: Emulate the high-performance architecture of Git's commit-graph. The BFT "round number" (or wave number) acts as a mathematically perfect topological generation number. Any graph walk required for extracting a causal slice or tracing dependencies must strictly use the target round number as a hard termination bound. This transforms an unbounded $O(V+E)$ traversal into an $O(k)$ localized, safe search.
Implement Narrow Secondary Indexing over LSM: Utilize RocksDB, but strictly separate metadata from payloads utilizing distinct Column Families (CF) to optimize compaction heuristics.
CF_Payloads: Stores the bulk transaction batches. Marked for aggressive, eager deletion the moment execution logic confirms consumption and state integration.
CF_Metadata: Stores the lightweight 32-byte DAG vertices and their embedded generation numbers.
CF_Reverse_Refs: Stores composite keys `` with empty values. This allows $O(1)$ prefix-seek resolution of unblocked vertices without requiring lock-heavy read-modify-write cycles, perfectly aligning with LSM append-only optimizations.
CF_Causal_Slices: Stores the pre-computed arrays of causal histories upon anchor commitment to accelerate offline node catch-up without re-traversing the graph.
Adopt Active Root Tracing for Garbage Collection: Abandon linear or strict round-based low-watermark pruning entirely. Implement a distributed, asynchronous adaptation of runtime tracing garbage collection. Define "Active Roots" as all uncommitted, valid vertices currently residing in the honest validators' mempools. Garbage collection must be executed asynchronously as a periodic sweep that retains any historical metadata transitively reachable from these active roots. This sweep must be bounded only by a protocol-defined absolute evidence horizon to mitigate Byzantine memory exhaustion attacks.
By framing the storage problem as topologically-bounded distributed memory managementrather than dynamic graph database querying, the storage and indexing layer can successfully match the extraordinary 100,000+ TPS execution capabilities of modern uncertified DAG-BFT protocols.
内容寻址和 DAG-BFT 的对比
内容寻址的 Merkle DAG（如 Git/IPFS）是一种静态数据结构，每个节点通过散列标识内容并引用子节点，实现不可变且可验证的存储。它从叶节点向上构建，一旦节点内容变化，其哈希就变了，所有父节点都需更新。Git 将文件、目录、提交等对象内容哈希后存储，从而实现版本历史的去重与冲突检测。IPFS 则提供分布式哈希表与节点缓存机制，节点会自动缓存数据，并通过垃圾回收删除**未标记（unpinned）的数据。在这些系统中，“原始证据”**是数据对象本身，而替代物则是由内容地址（CID/哈希）指向的相同内容；证明覆盖完整性可使用 Merkle 根或其他散列结构。
相比之下，DAG-BFT 系统的证据 DAG 则具有拜占庭环境下的攻击面和共识含义。每个 DAG 节点通常由验证者签名生成（例如阈值签名），代表了交易或事务的处理或排序结果。例如 IOTA/Sui 等系统中，**交易证书（Transaction Certificate）**是验证者对交易签名聚合的结果；**执行效果证书（Effects Certificate）**则是对交易执行结果（状态变化）的验证者签名聚合；**检查点证书（Checkpoint Certificate）**则是对一系列交易状态的最终确认，由委员会签名。这些证书不仅标识内容，还隐含信誉（>2/3 验证者同意）和时序关系。在此情况下，“原始证据”包括了单笔交易及其执行结果、上链载荷等，而它们的替代物则可能是由多个验证者共同签署的聚合证书或状态快照。
两者的相似之处在于：均使用 DAG 结构和密码哈希确保内容不可篡改，并可通过根哈希或摘要进行完整性验证（Merkle 哈希结构）。但不同之处在于：内容寻址 DAG 不涉及执行语义和权益攻击，即使节点恶意操作也仅影响数据完整性可通过内容哈希直接检测；而 DAG-BFT 的证据 DAG 中，节点（验证者）可能恶意双花或产生冲突版本，因此需要阈值签名和仲裁机制。内容 DAG 的“证明”主要是Merkle 证明：给定根哈希，任何节点的存在性可通过 Merkle 路径验证；而证据 DAG 的“证明”通常是签名聚合或多签证书：例如 Sui 的效果证书和检查点证书通过 2/3 验签来证明交易执行和状态达成一致。此外，内容 DAG 中不存在“节点活跃性”的概念，而 BFT 体系要求满足活性条件（足够验证者在线）。总结：内容 DAG 着眼于数据存储与检索，BFT DAG 则着眼于共识状态与安全保证，两者的设计和信任假设截然不同。
有效替代物
下表列出各类系统/机制下的“原始证据”及其潜在“替代物”，以及需要的关键元数据和可验证性要素：
检查点证书（Checkpoint Certificate）：在 IOTA/Sui 等 DAG-BFT 中，检查点证书由委员会为一批已执行的交易及其效果签名聚合。原始证据如单笔交易证书或执行结果可被检查点证书替代。元数据包括：检查点编号、上一检查点哈希、包含的交易哈希列表和效果哈希列表，以及委员会签名。验证方法是用委员会公钥验证聚合签名，并核对前一个检查点哈希与本地状态一致。检查点证书通常可立即获得验证器共识（已在线）后形成，适用于热路径内的快速状态确认。
状态快照（State Snapshot）：如 Sui 的“正式快照”，它在每个 epoch 结束时将全局活跃对象集导出，并对该对象集做紧凑承诺。它可替代之前的所有交易和检查点数据：只保留后续交易所需的输入对象。元数据包括：快照所对应的 epoch，高层承诺（如 Merkle 根或累加器根）和对象列表或版本范围。快照可从云存储检索，恢复时通过验证协议签署的状态承诺来保证正确性。由于快照以 epoch 为单位生成，它通常是离线操作（节点同步或灾难恢复时才使用），非共识实时路径。
存档对象（Archive Object）：指将区块链历史数据（交易记录、检查点、对象版本等）长期存储在外部系统中的机制。例如 Sui 使用Walrus或 Bigtable 等存储历史检查点和对象；Ethereum 生态的 Archive 节点保留历史状态。原始证据如旧交易和证书可以替换为从存档读取的记录。关键元数据包括：数据的时间范围（区块高度或检查点编号）、每个记录的哈希或索引、查询接口（端点）和数据可用性证明（如包含签名的检查点哈希）。检索延迟取决于存储媒介（如云存储可能较慢），通常用于离线全节点历史同步或审计，不参与热路径共识。
累加器证明（Accumulator Proof）：这类证明以累加器（如 RSA 累加器、Verkle 树等）形式将大量元素（如 UTXO、状态等）聚合为单个承诺，并能产生短的成员资格证明。原始证据如单个交易或状态元素可通过累加器和对应的证明来替代，例如 UTXO 集签名或状态根及 Merkle 证明。元数据包括：累加器的全局承诺值、该证明所证明的元素 ID 和其他成员证明节点。验证依赖累加器的正确性（比如验证Merkle分支或累加器算法）。累加器证明通常为离线/抽样用途，不直接在共识热路径中使用（除非协议专门设计，例如带累加器的账户模型）。
数据可用性（DA）证书：如 Celestia、EigenDA、Avail 等 DA 层提供的数据可用性证明。在 EigenDA 中，DA 证书是操作员基于以太坊锚定共识对上链数据可用性做出的签名证明；在 Celestia 中，区块头包含 NMT 根和 RS 编码行列 Merkle 根，节点通过数据可用性采样验证（并可提供行列哈希承诺）；Avail 使用 KZG 承诺作为有效性证明。这些证书可替代实际数据对可用性的证明：其元数据包括 blob/区块哈希、承诺值（如 KZG 根或 Merkle 根）和签名/证明数据。验证方法视实现而定：EigenDA 依赖以太坊上的共识记录和信任阈值签名；Celestia 依赖通过采样验证整个块；Avail 使用 KZG 验证。DA 证书通常也是离线或链下机制，实际存储数据往往需要离线检索（节点可自行下载完整数据），在热路径中仅用于确认可用性。
Merkle 证明（Merkle Proof）：用于证明某笔交易或对象包含在特定区块/根哈希中。原始证据（交易内容）可通过该交易的 Merkle 路径替代验证其包含性。元数据包括：Merkle 根和分支上的兄弟节点哈希。验证者只需逐层哈希比对根，即可断言元素确实归属该根。Merkle 证明通常由节点离线提供给轻客户端或用于时点验证，不用于共识的主执行路径。
执行效果证书（Effects Certificate）：在 Sui/IOTA 等体系中，效果证书聚合了 >2/3 验证者对交易执行结果（状态变化）的签名。它可替代单独的交易证书和执行结果。其元数据为交易的效果哈希和聚合签名（以及参与的验证者版本）。验证只需检查签名阈值并解码效果结构。效果证书用于快速确认交易最终性，常见于热路径。
替代覆盖完整性的元数据
要确定替代覆盖了哪些原始证据，需要维护特定的元数据：
覆盖范围（Coverage Range）：需指示替代物所涵盖的区块或时间范围。例如，某个检查点证书可能对应第 N 到 M 个检查点或交易序列，状态快照对应某个 epoch 的边界。在 DA 证书中，需包含 blob 的标识或发布块高度范围。
摘要列表（Digest List）：若替代物为批量数据（如检查点），应列出其中所有交易/对象的哈希。例如 Sui 检查点内部记录了该检查点包含的所有交易哈希及其效果哈希，从而明确定义了覆盖的交易集合。
对象版本集（Object Version Set）：对状态快照而言，需要记录哪些对象（ID）及其版本被保留。例如 Sui 正式快照只包含「活跃对象集」的最新版本；使用时需要这些对象的版本号和内容哈希来恢复状态。
上传水印（Upload Watermark）：对存档服务，需要知道已上传到存档（如 Bigtable、Walrus）的最高高度或时间戳。例如 Sui 的存档索引器维护一个 watermark_alt 跟踪处理到哪个检查点，意味着直到该检查点为止的数据都已存入。
检索端点（Retrieval Endpoint）：指出在哪里获取替代数据，如存储桶位置或节点地址。Sui 正式文档指导通过 S3/GCS 存储桶下载快照或使用备档服务的 gRPC 接口。
证明验证数据（Proof/Verif. Data）：任何需要验证的加密证明，例如 Merkle 根、BLS 聚合签名、KZG 承诺等。例如，Sui 快照还需检查对该快照的协议承诺；DA 证书需要相应的签名或链上验证；Merkle 证明需要兄弟节点哈希。
通过维护这些元数据，节点可以判断某个替代物是否完全覆盖所需的原始证据范围，并在检索到替代数据后验证其真实性。
提前修剪的失败模式
如果在替代覆盖完成前过早删除（修剪）原始证据，会导致多种失败：
节点补充失败（Catch-up Failure）：新节点或落后节点无法向前同步，因为缺少要恢复的中间状态或交易数据。例如，若 Sui 节点仅依赖最新快照启动，若间隔检查点未被保留，节点将无法重放这些检查点来重建状态，除非依赖存档。。
重放失败（Replay Failure）：当节点从历史状态重新执行或检索交易时，缺少部分交易或执行结果会导致状态不一致或执行中断。未经保留的交易证书或事件难以验证和执行。
执行失败：如果修剪掉了还未被聚合进检查点或状态快照的对象，如对象版本丢失，后续交易执行时读取这些对象会失败。
问责下降（Accountability Degradation）：缺少历史证据会让审计链遭破坏，难以证明某个验证者曾经签过某笔交易或执行结果，有利于逃避惩罚。拜占庭行为检测需要完整证据链。
无法服务对等节点：节点无法回答历史查询请求（例如区块或交易可用性查询），影响网络上的数据可用性。
审计缺口（Audit Gap）：审计和合规需要完整的链上历史记录才能回溯问题，过早丢弃数据会留下不可追回的历史盲区。
总之，若无充分替代保证就修剪原始数据，会导致状态和可用性丢失。现有区块链系统通常避免直接删除历史：例如 Ethereum 全节点只保留最近状态，旧状态可通过重放恢复；存档节点专门保存全部历史。DAG-BFT 环境下亦需小心：须确保替代措施完成覆盖再修剪。
替代机制：热路径还是离线？
热路径使用：多数替代机制偏离共识的关键路径。比如 Sui 的检查点和快照都是在交易执行最终后才创建，用于节点同步而非影响交易最终性。Sui 文档明确：“检查点不是交易执行或最终性的关键路径”。而执行效果证书则是热路径上的：一旦足够多验证者对事务执行签名，即形成效果证书，保证事务不可撤销。在 IOTA 也是如此，效果证书在达成后保证事务 finality，，而检查点聚合更多结果用于重配置和同步。
数据可用性证书：例如 EigenDA 的 DA 证书是在上链后发布，以太坊合约记录数据可用性，仅用于验证数据可用性，不参与事务本身的共识路径。Celestia 的 KZG/采样机制则是 DA 层的共识本身，但对上层链（如 Rollup）而言数据可用性证明通常不在本地共识热路径中。
延迟容忍：检查点和快照具有较大延迟容忍度（分钟级乃至更高），因为它们用于节点恢复而非实时交易处理。DA 系统通常设计为几十秒到数分钟出证，例如 Avail 块确认约 40 秒；EigenDA 则利用以太坊的区块时间。它们可以在充分缓冲后用于纠错或最终确定。
热路径可用性：效果证书和 DA 证书等可以快速提供最终性保证；而检查点证书和快照基本上脱离共识流程，只在节点需要校验链历史时使用。

研究空白
当前已有的存档、检查点和可用性系统主要关注在历史数据的整体可用性和节点快速同步上，而针对有向证据DAG中的因果依赖并没有专门解决方案。具体而言：
因果依赖跟踪缺失：DAG-BFT 的证据往往相互依赖（如交易引用父事务、执行依赖某些状态）。现有机制通常以时间或区块序号为基础进行修剪，而缺乏自动化方法来跟踪“哪些未确认的后续消息依赖此数据”。换言之，没有专门存储结构记录证据的引用关系以指导精细化回收。
动态根状态识别：在 DAG-BFT 中，哪些证据仍处于“活动链”或“共识分支”的根部并未有明确定义。区块链传统做法是以最长链为依据，但 DAG-BFT 协议往往允许分叉或并行提交。缺乏机制来标记“当前生效的DAG前缀”，因此难判断数据何时可删。
语义覆盖证明：检查点证书或快照能证明达成共识的状态，但它们对未包含数据的覆盖范围是“全或无”。如果要删除个别交易或区块的证据，需要更细粒度的证明（类似累加器或 Merkle 证明能覆盖部分元素）。目前缺乏这类用于部分覆盖验证的通用方案。
与 DA/存储层的整合：现有 DA 系统与 DAG-BFT 共识层大多是独立堆栈，缺乏针对证据回收的协同策略。例如，可用性层可存储数据，但其证书只是证明可用性，而非证明“协议已经以此为基础达成共识”。如果能将“共识证书”与“可用性证书”对齐，或许可以更安全地删除本地数据。
总的来说，DAG-BFT 需要一种因果可验证的存储管理策略：即能够在运行时动态跟踪哪些证据仍被活动分支或协议机制引用，以及哪些已被更高阶对象（检查点、快照、证明）完全覆盖。这可能涉及新的元数据结构（如依赖计数、引用图快照），以及与现有共识机制协同的自动化清理规则。目前相关公开研究甚少，是一个明显的研究空白。当前方案要么粗放（按时间/高度修剪），要么重量级（全部存档），都难以满足超高吞吐下的存储效率和安全保证需求。
Causal Evidence Storage and Pruning Semantics in High-Throughput DAG-BFT Systems
Introduction to the Hot-Path Storage Problem in Directed Acyclic Graph Consensus
High-throughput Directed Acyclic Graph Byzantine Fault Tolerant (DAG-BFT) systems have fundamentally restructured the distributed consensus landscape. By systematically decoupling the logical ordering of transactions from the physical materialization and dissemination of evidence, protocols such as Narwhal, Bullshark, and Mysticeti have shattered the throughput bottlenecks inherent in traditional monolithic blockchains. In these advanced architectures, validators construct a continuously growing, highly parallelized directed acyclic graph composed of transaction batches, cryptographic digests, quorum certificates, and parent references. The consensus logic operates strictly on this lightweight metadata—often referred to as causal evidence—allowing the network to achieve total ordering with near-zero additional message overhead and sustained throughputs exceeding hundreds of thousands of transactions per second.
While this decoupling yields massive scalability gains, it introduces a profound and highly specialized hot-path storage problem. Validators may sequence, order, or reference a multitude of cryptographic digests, block headers, and quorum certificates well before the corresponding physical object bodies, execution payloads, state repair dependencies, or checkpoint metadata are fully materialized, executed, and consumed across the distributed network. Consequently, the causal evidence underpinning the DAG—the unforgeable cryptographic proof that a specific vertex was validly proposed, acknowledged by a quorum, and causally linked to preceding state—must be preserved in high-performance hot storage.
The central hypothesis under investigation posits that causal evidence in storage-backed DAG-BFT systems cannot be managed merely as a flat key-value history subject to arbitrary chronological, epoch-based, or depth-based pruning. Instead, correct retention theoretically requires an intimate, real-time awareness of the system's topological and temporal state. According to this hypothesis, a validator must know precisely which uncommitted vertices depend on specific evidence, whether the evidence remains an active protocol root, whether it has been consumed by secondary protocol paths (such as execution, state catch-up, accountability, or repair mechanisms), and whether it has been safely substituted by a durable cryptographic summary. If a validator prematurely prunes a DAG vertex or its associated payload before these conditions are met, the causal chain breaks, potentially halting local consensus participation, preventing lagging peers from synchronizing, or violating the strict non-equivocation guarantees required to penalize Byzantine actors.
Critical Evaluation of the Hypothesis
While the hypothesis correctly identifies the profound tension between high-throughput data dissemination and bounded memory constraints, a critical examination reveals that the framing is partially over-broad and, in certain protocol designs, assumes a level of required topological tracking that modern DAG-BFT systems have already deliberately circumvented. The idea that DAG-BFT systems struggle with unbounded memory growth due to causal evidence retention is well-documented in the literature and is not inherently novel. The original Narwhal implementation notoriously suffered from garbage collection failures that could exhaust 120 gigabytes of RAM in a matter of minutes, underscoring the severity of the exact hot-path storage problem identified.
However, the hypothesis errs by asserting that correct retention must explicitly require knowing "who depends on this evidence" on a granular, per-validator or per-vertex basis. Tracking exact dependencies across a highly asynchronous, high-throughput network introduces $O(N^2)$ metadata overhead that would cripple the consensus hot path. Prior art, most notably the Bullshark consensus protocol, provides a direct counterexample to the necessity of granular dependency tracking. Bullshark resolves the tension between providing fairness to slow validators and enabling practical garbage collection by implementing a temporal median-based pruning heuristic rather than a strict dependency-aware reference counter.
In Bullshark, validators attach local timestamps to every broadcasted vertex. When a leader anchor vertex is committed, its timestamp is calculated as the median of the timestamps from its causal parents. The protocol then defines a strict pruning threshold using a predefined $\Delta$ window. When the deterministic backward traversal of the causal history reaches a round whose median timestamp is older than the anchor's timestamp by more than $\Delta$, that round and all preceding rounds are indiscriminately garbage collected. Any vertices from slow, honest validators that fall outside this window are pruned before being totally ordered, forcing those validators to re-broadcast their transactions. This explicitly proves that DAG-BFT systems can achieve optimal performance and bounded storage by relaxing asynchronous fairness and accepting the loss of some causal evidence, rather than tracking exact topological dependencies.
Furthermore, the hypothesis slightly conflates the storage requirements of the consensus DAG (the metadata) with the execution DAG (the physical payloads). Mysticeti, the latest evolution of DAG-BFT utilized by the Sui blockchain, explicitly separates these concerns. Mysticeti operates an uncertified DAG where vertices require only 0.5 round-trip times to disseminate, vastly increasing the volume of causal evidence. To manage this, Mysticeti relies on a specialized Write-Ahead Log (WAL) with memory-mapped files and vectored writes, abandoning traditional key-value stores entirely for the consensus hot path. In this environment, garbage collection is simply a deterministic pruning threshold updated implicitly as part of each successful leader block commit. The consensus mechanism does not query whether execution, repair, or accountability paths have consumed the data; it simply drops the metadata once the total order is cryptographically secured, shifting the burden of payload retention entirely to the execution and archival layers.
Therefore, while the core problem of hot-path storage exhaustion is accurate, the framing must be refined: DAG-BFT systems survive not by perfectly tracking the holistic lifecycle of every byte of causal evidence, but by employing aggressive, deterministic pruning heuristics at the consensus layer while relying on robust, asynchronous substitution mechanisms at the execution and archival layers.
Distinguishing the Problem Domain
To properly scope the research direction, it is vital to distinguish the specific hot-path evidence storage problem in DAG-BFT from adjacent, superficially similar paradigms in distributed systems and database architecture.
The DAG-BFT storage problem is fundamentally distinct from ordinary blockchain historical pruning. In traditional Nakamoto consensus or linear BFT systems (such as Tendermint or early Ethereum implementations), the mempool is an unstructured, out-of-band component. Blocks are proposed sequentially, and historical pruning merely involves dropping state or blocks that fall below a finalized depth parameter. The total order is implicit in the block height, and the causal history is strictly linear. In contrast, a DAG-BFT mempool is an integrated, causally entangled data structure where the topological relationship between uncommitted vertices constitutes the very evidence required to achieve consensus. Dropping a block in Bitcoin saves disk space; dropping an uncommitted vertex in Narwhal destroys the protocol's ability to calculate a quorum intersection.
Similarly, this problem cannot be solved using offline blockchain graph analytics or generic graph databases (such as Neo4j). Graph databases are optimized for complex, multi-hop querying and pattern recognition on relatively static or predictably growing datasets. They do not contend with real-time adversarial Byzantine production, sub-second liveness constraints, or the need to continuously truncate the graph root while advancing the tip at hundreds of thousands of operations per second. Applying a generic graph database to a DAG-BFT mempool would introduce intolerable disk I/O latency and serialization overhead, stalling the consensus engine.
Furthermore, ordinary RocksDB or Log-Structured Merge-tree (LSM) database tuning is insufficient for managing causal evidence on the hot path. LSM trees handle high write throughput by buffering data in memory and flushing it to disk in sorted layers, which are periodically compacted. While database compactions reclaim disk space and optimize read performance, the storage engine is entirely agnostic to the semantic validity of the data. RocksDB does not know whether a specific cryptographic digest is required to prove non-equivocation or to fulfill a $2f+1$ quorum requirement. In fact, the write amplification and periodic compaction stalls inherent in RocksDB are so detrimental to high-throughput DAG-BFT liveness that protocols like Mysticeti intentionally bypass them, implementing custom, continuous Write-Ahead Logs tailored specifically to the deterministic garbage collection rules of the consensus algorithm.
Finally, the problem must be distinguished from distributed file storage or Data Availability (DA) storage. DA networks like Celestia, Avail, and EigenDA are optimized for provable publication and eventual retrieval, utilizing erasure coding and cryptographic commitments to guarantee that data was made available to the network. However, their operational latencies range from several seconds to minutes. DAG-BFT consensus requires synchronous, sub-second verification of causal evidence to advance the mempool. DA systems are therefore destinations for cold-tiering and long-term evidence substitution, not solutions for hot-path mempool memory exhaustion.
Comparative Analysis: Content-Addressed Merkle DAGs vs. DAG-BFT Evidence DAGs
The architectural utilization of directed acyclic graphs in BFT consensus systems shares superficial structural characteristics with traditional content-addressed Merkle DAGs, such as Git, InterPlanetary File System (IPFS), and generic Merkle trees. Both paradigms rely heavily on content addressing, wherein objects are identified universally by cryptographic hashes (e.g., SHA-256 or BLAKE3), and relational topology is constructed by embedding the hash of a parent object directly into the payload of a child object. This immutable structure ensures that any modification to a historical vertex irreversibly alters the entire subsequent hash chain, enabling localized, incremental verification. A node can authenticate a specific branch of the DAG without requiring possession of the global state.
However, the operational realities, adversarial threat models, and lifecycle requirements of these two systems diverge profoundly. Content-addressed systems like IPFS and Git operate in largely passive, non-adversarial environments. If two conflicting branches are created in a Git repository, they coexist passively until a human operator manually initiates a merge resolution. There is no strict real-time, algorithmic requirement to determine a globally canonical total order, nor is there a systemic penalty for producing orphaned branches. Garbage collection in these traditional systems is an offline, local disk optimization (e.g., the git gc command) designed merely to reclaim space by removing unreferenced objects. If a node running IPFS drops a chunk of data, that data becomes temporarily unavailable until it is requested and retrieved from another peer; the network itself does not halt.
In stark contrast, DAG-BFT evidence DAGs operate in highly adversarial, Byzantine environments. The vertices are produced by a dynamic, heavily staked validator set that must algorithmically reach a global consensus on a single canonical total order within milliseconds. In this context, causal evidence is not merely a passive content-addressed blob; it is an active cryptographic vote. Every vertex explicitly asserts that the proposing validator has witnessed, verified, and endorsed the causal history of its parent vertices.
Furthermore, these vertices are tightly coupled to execution status and quorum advancement. The transition of a vertex from an uncommitted mempool proposal to an active protocol root, and finally to a totally ordered execution payload, requires traversing strict quorum rules, typically demanding aggregated signatures representing $2f+1$ of the network's stake weight. Consequently, garbage collection in a DAG-BFT system is a critical, consensus-bound safety mechanism, not an offline optimization. As demonstrated by Bullshark, the algorithmic decision to prune a DAG-BFT vertex directly impacts the liveness and fairness of the consensus mechanism itself. If a DAG-BFT validator prematurely drops a vertex before a network quorum has advanced past it, the validator risks equivocating, failing to provide required causal evidence to lagging peers, or permanently diverging from the network's agreed-upon state machine. Thus, the retention parameters in a DAG-BFT system must meticulously track network-wide quorum progression and commit thresholds, a requirement entirely absent in traditional Merkle DAG implementations.
The Lifecycle of Causal Evidence and Failure Modes of Premature Pruning
Given the critical nature of causal evidence in DAG-BFT systems, executing garbage collection before substitute coverage is fully verified exposes the network to a cascade of catastrophic failure modes. Causal evidence transitions through several distinct phases: active proposal, quorum certification, causal anchoring (total ordering), execution, and finally, historical archiving. Understanding the failure modes at each phase is essential for determining safe pruning thresholds.
The most immediate and severe consequence of premature pruning is catch-up failure, which directly impacts the liveness of the consensus network. In asynchronous environments, honest validators routinely fall behind the active network tip due to transient network partitions, CPU throttling, or local disk I/O constraints. When a lagging validator attempts to synchronize, it relies on its peers to provide the missing causal history. If the active peer network has aggressively garbage collected the DAG below the low watermark—without ensuring that a verifiable checkpoint certificate or archive object is available to summarize those specific rounds—the lagging validator cannot cryptographically reconstruct the causal path leading to the current active protocol roots. The validator becomes permanently stalled, unable to vote on new proposals or participate in the total order. This effectively consumes a portion of the network's Byzantine fault tolerance budget, as an honest but stalled node is indistinguishable from a malicious, crashed node.
Execution and replay failures represent a secondary, yet equally fatal, consequence. While the consensus logic only requires the metadata (the DAG of digests) to calculate the total order, the execution engine requires the physical materialization of the transaction payloads. If a validator prunes the physical batch payloads from the mempool storage before the execution engine has successfully applied the corresponding state transitions, the node cannot compute the transaction effects. This creates a desynchronized state where the network agrees on the sequence of events but cannot physically execute them, halting the progression of the blockchain's application layer.
Accountability degradation presents a profound security risk. DAG-BFT protocols rely on strict non-equivocation properties to simplify consensus and penalize Byzantine actors. If a malicious validator broadcasts conflicting vertices to different partitions of the network, honest validators require the physical causal evidence—specifically, the conflicting vertices signed by the malicious actor's private key—to prove the equivocation and trigger slashing protocols. If the system indiscriminately prunes this evidence from hot storage before the equivocation proof is generated, processed, and archived, the Byzantine actor escapes penalty. Consequently, premature garbage collection fundamentally undermines the cryptographic accountability mechanisms of the system.
Finally, premature pruning leads to an inability to serve peers and audit gaps. Seed peers and RPC full nodes are contractually expected to maintain a configurable history window, typically measured in epochs, to serve light clients and lagging nodes. In Sui, this is controlled by the authority-store-pruning-config and num-epochs-to-retain parameters. If a node misconfigures its pruning strategy and drops data too early, it fails its obligation to the peer-to-peer network. If a critical mass of nodes simultaneously adopts aggressive pruning without coordinated substitute coverage, the entire network loses access to recent history, forcing all participants to immediately rely on highly latent, potentially centralized archival fallbacks just to maintain basic operations.
Substitutability: Replacing Hot-Path Evidence
To mitigate the rapid memory exhaustion inherent in DAG-BFT systems without triggering the failure modes detailed above, original hot-path evidence must be safely replaced by a valid substitute object. A valid substitute is a cryptographic primitive or a durable, off-path data structure that provides equivalent verifiability guarantees without requiring the physical presence of the original DAG vertices, raw batches, or uncompressed parent references.
The most fundamental and highly utilized substitute in modern architectures is the checkpoint certificate. A checkpoint represents a contiguous batch of finalized transactions alongside the resulting cryptographic state of the network, secured by an aggregated BLS signature from a quorum of validators. In Sui, validators utilize the minSig BLS mode, which allows independent signatures to be aggregated into a single 48-byte proof, dramatically reducing the size of the substitute object. Once a transaction and its intricate causal history are encapsulated within a checkpoint certificate, the original DAG vertices and Narwhal/Mysticeti batch payloads become strictly redundant for consensus validation. The checkpoint certificate serves as a highly compressed substitute that allows lagging nodes or light clients to efficiently validate the recent state of objects and emitted events without replaying the entire DAG causal history.
State snapshots offer a more aggressive, state-centric form of substitution. Formal snapshots are minimalistic, database-agnostic representations that contain only the end-of-epoch live object set. Because they deliberately omit the complete chronological transaction history and focus entirely on the resulting cryptographic state, formal snapshots drastically reduce the storage footprint required to bootstrap a new node. However, formal snapshots carry severe retention implications; they are only valid substitutes for nodes focused purely on execution and future consensus participation. They do not serve as valid substitutes for RPC nodes, indexers, or block explorers that must answer queries regarding historical transaction execution, event emission, or causal provenance.
Data Availability (DA) certificates, such as those generated by EigenDA or Avail, represent an externalized, off-chain substitute mechanism. When a rollup or decentralized sequencer offloads its bulky causal evidence to a DA layer, the original data is erasure-coded, committed via KZG polynomials, and dispersed to a restaked validator set. The DA disperser returns a BlobStatusReply containing a batch header and a blob verification proof, from which the client constructs a DA Certificate (DACert). This DACert acts as the substitute object on the primary chain. The execution engine evaluates the DACert either pessimistically (as a strict pre-inclusion check before execution) or optimistically (relying on a fraud-proof challenge window), effectively treating the certificate as cryptographic proof that the causal evidence remains retrievable if needed.
Archive objects and execution effects certificates serve as the final tier of historical substitution. When an execution engine processes a totally ordered transaction, it generates an effects certificate detailing the precise object modifications, balance changes, and system events emitted during execution. These effects, along with the raw transaction data, are offloaded to an archival indexer, which transforms the data into highly structured formats suitable for cold storage, such as Google Cloud Bigtable in the Sui archival stack.
Metadata Requirements for Complete Substitute Coverage
For a DAG-BFT system to confidently accept any of these objects as a valid substitute and trigger the physical deletion of the original hot-path evidence, strict coverage metadata must be cryptographically verified. The execution engine, the mempool, and the pruning logic must share synchronized watermarks to ensure atomic transitions.
The required metadata includes:
Coverage Range: The exact epoch boundaries, round sequences, or checkpoint sequence numbers encapsulated by the substitute.
Digest Lists: Cryptographic maps linking the original transactions or DAG vertices to the substitute object, proving inclusion.
Object Version Sets: Monotonically increasing version numbers that prove the exact state of an object authenticated by a snapshot.
Upload Watermarks: Explicit database flags or receipts confirming that the causal data has been safely and redundantly persisted in the remote retrieval endpoint (e.g., the watermark_alt table in Bigtable) before local disk deletion is permitted.
Proof Verification Data: The aggregated BLS signatures, Merkle paths, or KZG polynomial commitments necessary for any node fetching the substitute to independently authenticate it against the protocol's known epoch configuration.
Data Availability Systems: Mechanisms and Latencies
Given the strict mathematical and physical limitations on hot-path storage within the consensus validator set, the broader modular blockchain ecosystem has developed externalized Data Availability (DA) systems to manage causal evidence and transaction payloads. However, these systems vary drastically in their retrieval latencies and retention guarantees, dictating exactly when and how they can be used as substitutes.
Celestia approaches data availability through the use of Namespaced Merkle Trees (NMTs) and Data Availability Sampling (DAS). When data blobs are submitted to Celestia, they are split into chunks using 2D Reed-Solomon erasure coding, ensuring that even if a large portion of the block is withheld by malicious actors, the full data can be reconstructed. Light nodes perform DAS by randomly sampling a tiny fraction of the encoded matrix to probabilistically verify availability. Crucially, Celestia enforces a strict 7-day sampling and pruning window for its light nodes. Data blobs older than the 7-day recency window are completely pruned from the consensus network to maintain efficiency. After this window, the data is relegated to professional archival nodes.
EigenDA leverages the restaked security of the Ethereum validator set but operates as a purely off-chain middleware. It utilizes KZG polynomial commitments rather than standard Merkle proofs to cryptographically bind the erasure-coded data. The EigenDA disperser handles the encoding, dispersal, and signature collection asynchronously. Notably, EigenDA treats data availability as a highly temporary guarantee, specifically designed to support short-term rollup challenge periods and bridge finalizations. The retention window for EigenDA is typically configured to approximately 4 hours.
Avail similarly utilizes 2D matrices, erasure coding, and KZG commitments to guarantee availability to light clients. Like Celestia and EigenDA, Avail is fundamentally designed to publish and order data, not to store it perpetually. The pruning time for Avail makes data available in the network for approximately 60 hours, after which archival solutions must take over retention responsibilities.
Because these DA systems explicitly drop data after a defined temporal window (ranging from 4 hours to 7 days), they are highly suitable as temporary substitutes for hot-path mempool data during the active execution and optimistic challenge phases. However, they are entirely unsuitable as permanent historical archives. If a DAG-BFT system offloads its causal evidence to EigenDA and relies on it for long-term catch-up, any node attempting to synchronize after the 4-hour retention window has expired will suffer a permanent derivation halt, unable to reconstruct the necessary state. Furthermore, the retrieval latency from DA systems—often measured in seconds to minutes depending on the disperser queue and signature gathering—renders them too slow to serve as synchronous hot-path substitutes for an active DAG-BFT mempool that operates on sub-second commit cycles.
Archival Fallbacks and Cold Storage Mechanisms
To bridge the gap between aggressive hot-path pruning and the temporary retention windows of external DA layers, high-throughput systems rely on sophisticated, highly scalable Archival Fallback mechanisms. When a full node or lagging validator attempts to synchronize its state, it initially connects to its active peers via standard gRPC protocols. However, seed peers are configured to only retain a limited history locally. If a node lags beyond the epochs retained by its peers—dictated by configurations such as num-epochs-to-retain—standard peer-to-peer synchronization fails.
To recover, the node triggers an archival fallback protocol, executing configurations like state-archive-read-config to fetch required checkpoint data directly from massive, external object stores, such as Amazon S3 or Google Cloud Storage. This ensures that the network can dynamically recover from deep partitions without demanding that every validator purchase petabytes of NVMe storage.
The Sui archival stack represents the industry standard for cold-tiering DAG-BFT causal evidence. The architecture is designed to handle the immense data footprint generated by 100k+ TPS networks. It consists of a highly scalable Google Cloud Bigtable deployment backed by dedicated indexers (sui-kvstore-alt) and a gRPC server (sui-kv-rpc) that exposes standard node RPC endpoints. The indexers asynchronously ingest checkpoints from the active network or from Requester Pays cloud buckets, structuring the complex causal history across 11 specialized tables. These tables include transactions, objects, checkpoints, checkpoints_by_digest, and epochs.
The physical scale of this archival data underscores the impossibility of retaining causal evidence on the consensus hot path. A complete mainnet backfill of the Sui archival stack exceeds 13.3 Terabytes, with the transaction data alone consuming over 8.1 TB and the object state consuming 4.4 TB. The Bigtable cluster requires robust SSD provisioning and dynamic gRPC connection pooling to handle the ingestion rate without buckling.
Crucially, the latency profile of these archival mechanisms dictates their operational placement. Lookups against Bigtable or large-scale cloud buckets operate on a scale of hundreds of milliseconds to several seconds. They are strictly out-of-band and tolerate significant synchronization lag. They are utilized exclusively for full-node historical synchronization, auditing, and analytical data serving (e.g., GraphQL RPC endpoints), and can never be used to advance an active DAG-BFT consensus root.

Conclusion: The Research Gap in DAG-BFT Causal Storage
When addressing the central question—when can original hot-path evidence be safely pruned, cold-tiered, or replaced by another object?—the research demonstrates that pruning can only occur safely when the causal evidence has transitioned from an active protocol root into a cryptographically verified substitute, and only when that substitute is physically accessible by the mechanisms that require it. For the consensus engine, evidence can be pruned the millisecond a Bullshark $\Delta$ threshold is crossed or a Mysticeti leader block is totally ordered. For the execution engine, payloads can be pruned only after a checkpoint certificate is generated and signed by a $2f+1$ quorum. For the broader network, historical evidence can be deleted locally only when a durable archive object has been safely watermarked in cold storage.
Despite the rapid maturation of DAG-BFT consensus protocols and the concurrent explosion of modular Data Availability architectures, a profound research gap remains at the intersection of causal evidence tiering and hot-path memory management. Current DAG-BFT systems rely on rigid, macro-level heuristics to bridge the gap between active consensus and cold storage. Mechanisms like Bullshark’s $\Delta$ timestamp garbage collection or Sui’s epoch-based authority-store-pruning-config enforce deterministic, chronological cut-offs. They treat all evidence within a temporal window identically. They do not possess the granular awareness required to dynamically offload or prune specific vertices based on precise, per-object causal consumption without introducing unacceptable $O(N^2)$ metadata overhead.
What high-throughput DAG-BFT systems require—and what current DA or archival systems fail to natively provide—is a dynamic, dependency-aware evidence tiering protocol that operates strictly within the sub-second latency bounds of the consensus hot-path. Archival systems like Bigtable and DA layers like EigenDA are fundamentally out-of-band; they are designed for post-execution state dumping or optimistic challenge resolution. Their retrieval latencies are completely incompatible with a stalled validator attempting to reconstruct a live protocol root in real-time.
The unresolved research frontier lies in designing an intermediate, hot-path substitute structure specifically for DAG-BFT systems—a highly compressed, localized accumulator mechanism that allows a validator to definitively cryptographically compress a subgraph of uncommitted causal evidence the exact microsecond it proves that no lagging honest peer will require those specific bytes for a quorum intersection. Until such an intra-consensus evidence compression scheme is developed, high-throughput DAG-BFT systems will remain ultimately bounded by the physical memory limits of individual nodes, forcing a persistent architectural trade-off between the retention required to guarantee asynchronous network liveness and the aggressive pruning required to prevent out-of-memory cascading failures.
替代证据与覆盖
高吞吐 DAG-BFT 系统中，原始共识证据（如完整的交易数据或提案负载）可以被多种可验证替代物覆盖和取代。这些替代物包括：检查点证书（checkpoint certificate）、状态快照、归档对象、累加器证明、DA 证明/证书、Merkle 证明、执行结果证书等。例如，Sui 的检查点证书包含到目前为止所有事务和其执行结果的哈希列表，并由 ≥2/3 验证人签名；因此一个检查点证书可以替代原始事务/执行数据用以验证历史。常见的替代覆盖示例如下：
检查点证书（Checkpoint Certificate）：包含所有已提交事务的摘要及效果摘要列表，带有 BLS 聚合签名。只要全节点获取连续的检查点，就能凭此验证并重演历史，不需单独存储每笔交易。
状态快照（State Snapshot）：例如区块链全状态的存储快照（含状态根或账户集）。获得快照后，新节点可跳过执行历史直接加载状态并验证哈希。以太坊新版本节点可通过 Snapshots 快速同步，但需要可靠的快照链根证明支持。
归档对象（Archive Object）：如存储在远程网关或分布式存储（IPFS/Walrus/S3）中的区块或检查点数据。这些归档通常是内容寻址的二进制 blob，可离线下载。
例如，Sui 的 Walrus 归档应用会持续获取链上检查点数据，将其打包为标准 blob 并上传至 IPFS/Walrus；节点在缺失历史时可从该归档检索。
累加器证明（Accumulator Proof）：密码学累加器可对一组元素生成简短承诺并提供成员证明。若区块链维护对所有已确认交易或状态的累加器根，则可用其中的证明替代逐一查找历史。不过目前主流链还少用此方案。
DA 证明/证书（Data Availability Attestation）：如 EigenDA 生成的可用性证明。EigenDA 对上传的数据 blob 输出一份可用性证书，证明该 blob 已被网络多个验证方承诺可检索。此证书可作为数据可用性的替代物，无需下载原始 blob 即可确信其可检索。
Merkle 证明（Merkle Proof）：利用 Merkle 树结构提供的包含证明。例如 Celestia 采用 Namespaced Merkle Trees，可证明某个命名空间下的数据被包含在已公布的块中。Merkle 证明允许轻节点在无需全量数据的情况下验证包含性。
执行结果证书（Execution Effects Certificate）：证明事务已正确执行并生成相应状态变化的证书。例如 Sui 的检查点中记录了每笔交易的事务影响哈希（effects），可用于验证执行结果与共识一致。
这些替代物各自携带的元数据决定了覆盖的完整性。例如，检查点证书通常含有事务/效果摘要列表以及该检查点的顺序编号；归档对象需要时间范围标记或最高已归档高度；DA 证明需要包含blob 范围或哈希及聚合签名；Merkle 证明需要包含分支的哈希路径。保证完整覆盖通常需要记录：覆盖范围（例如第 X 到第 Y 个块或事务）、包含的摘要或对象清单、版本集（状态或对象版本）、检索节点地址，以及能够验证证明正确性的验证数据（如 KZG 聚合多项式或 BLS 公钥）。例如 Avail 使用多维 RS 编码和 KZG 承诺，节点只需采样几个数据片段并验证对应 KZG 证明即可高概率确认全数据可用。
提前清除的失效后果
如果在替代覆盖齐备之前就提前删除（Prune）原始证据，会引发多种故障风险：
重放失败（Replay Failure）：缺少必要的事务或前置状态使节点无法重演历史。若提交已确认，但缺少事务主体（payload），节点无法执行出块，造成“已提交但无法执行”的失效情况。
同步故障（Catch-up Failure）：新节点或滞后节点无法从同步点追回链上历史，因为原始区块数据已被清除且备份不可用。
修复/补全失败：出现证据缺失时，节点无法发起成功的修复请求（recovery），导致无法补全 DAG 的空缺分支。
问责证据缺失（Accountability Degradation）：缺少历史记录可能使得在出现分叉或故障时无法完整溯源各验证者的投票或提议。
服务能力下降：节点无法对等提供历史数据，对外节点查询也会失败。
审计缺口（Audit Gap）：审计或一致性检查时，缺少证据会导致不可验证链的完整性。
存储成本上升：若不知道何时安全删除，系统可能被迫保留过多冗余数据，增大存储需求。
例如，**数据隐匿攻击（Data Withholding Attack）**下，恶意区块生产者发布新区块头但隐匿其数据，若多数诚实节点不检测则块会被不正确地添加，最终导致全节点执行异常。类似地，在 DAG-BFT 中若攻击者故意延迟或拒绝广播批次内容（digest-first, body-late），其他节点只能投票确认摘要引用，而缺失的依赖会导致后续执行/重放失败。
替代机制的热路径使用
大多数替代机制属于离线恢复/归档层面，通常不参与共识的热路径：
检查点/状态快照：通常由协议周期性生成，用于新节点或恢复节点进行一次性同步；不会参与日常共识消息流。
归档对象：存放在第三方存储（如 IPFS、云桶）中，仅在节点严重滞后时才访问；不可用于在线共识修复。
DA 证明/证书：由专门的数据可用性层（如 Celestia、EigenDA）提供，并不会直接影响链内共识决策，通常用于外部验证数据是否可用，属于通信层面的保障。
Merkle 证明：主要用于轻节点或链上合约验证数据包含性，可在运行时使用，但仍算查询层面，不影响共识主链的快速决定。
也就是说，除了某些基于委托证明（如 EigenDA “随时可用承诺”）可以在理论上用于共识决策外，大多数替代手段只能作为“冷启动”或历史查询使用，不适用于共识本身的延迟修复路径。因此在共识热路径上依然需要原始或足够新鲜的证据，直到其被安全签发或封存为替代物。
Merkle DAG 与 DAG-BFT 证据图对比
内容寻址的Merkle DAG 系统（如 Git、IPFS）与 DAG-BFT 证据 DAG 有相似之处：二者都是无环结构并利用哈希标识。然而也有本质区别：
内容可验证性：Merkle DAG 中每个节点的标识（CID）即为其内容及其所有后代内容的哈希。一个节点一旦创建，其内容和哈希不可更改；同一哈希必定对应同一数据。这保证了去重与完整性。相反，DAG-BFT 的节点（提案、证书）可能包含签名或随机戳，即便对内容不同也可能拥有不同哈希，不具备那样严格的不可变性。攻击者可以有选择地生成不同的拓扑结构来试图破坏系统，而内容地址无法直接防范此类结构性攻击。
引用意义：Merkle DAG 引用父节点指向内容的指针；验证可通过 Merkle 证明完成。DAG-BFT 引用则是对前提提案或证书的签名引用，并伴随投票或 QC。这种引用不仅仅是数据完整性的证明，更包含拜占庭参与者达成协议的安全保证。换言之，Merkle 链偏向数据结构完整性，DAG-BFT 偏向协议安全性。
对等节点行为：在内容网络中，节点“有即为真”；而在 BFT 系统中，即使多数副本有一个证据节点，也不能证明其他副本也有；而且有分叉的合规 DAG 结构常常被协议认为无害。
用途差异：内容 DAG 多用于协同编辑、文件存储等，需要证明“所有人都看到同样的数据”。DAG-BFT 证据图则用于并行提案和确认，高吞吐但需要附加的共识层次。
综上，Merkle DAG 系统强调数据去重和内容一致性验证；DAG-BFT 强调在对抗性环境下也能最终共识。对 DAG-BFT 来说，节点的拷贝状态、执行情况、分叉信息等都很重要，而这超出了简单 Merkle DAG 能提供的保证。

对抗模式与安全属性
对手能力
合理的拜占庭对手可能具备能力：
设计化的 DAG 构造：故意生成满足协议验证要求但结构“特殊”的顶点。例如创建一个节点被许多后继引用（高出度）、大量引用尚未提供的祖先等。
网络延迟控制：操纵部分消息顺序，使有效负载（transaction payload）慢于消息标头（digest/header）。即先广播摘要再迟迟不发数据，迫使他人等待或频繁请求。
分散传播负载：利用网络分区，部分节点收到不同内容的版本，导致全局看似一致但各自有缺失。
存储压力：充填节点存储满载。恶意节点评价不多次检索已丢弃或垃圾回收的旧证据，制造 I/O 压力。
选择性修复延迟：对节点提出修复请求时故意慢响应或只针对部分请求者响应，延迟其同步。
归档/检查点滞后：验证者拖延发布检查点或归档数据，使替代覆盖出现盲区。
验证者带宽/算力偏斜：故意使一些诚实验证者难以跟上（如网络噪声），导致他们落后于某些“活跃”视图，难以获得最近证据。
这些能力均在常见假设下可实现，只要拜占庭节点精心控制消息发送顺序和选择性发送。
容易触发的攻击模式
拜占庭节点在合法协议行为范围内，可以制造存储繁重或修复困难的模式：
缺失祖先放大：一个节点投票形成的 QC 引出一个新顶点，但故意不附带先前节点的全部祖先数据。其他节点必须发起多次获取请求，网络带宽和 I/O 负载大增。
多重后代引用单一丢失数据：许多不同的合法顶点都以同一延迟数据为父（digest-first 模式），使得修复请求集中对同一个丢失区块，形成请求“雪崩”。
垃圾传播（Digest-first/Body-late）：对手生产大量只有摘要（或证书）的块，再延迟内容广播。协议允许先批准摘要，一旦太多无实体块积压，系统会饱和。
检查点滞后：在协议需要生成检查点或状态快照的场景，对手故意使部分节点错过快照时间，导致部分节点所依赖的替代物不可用。
问责证据冷存储：协议将老历史转移到冷存储（归档节点），对手利用冷节点响应慢、丢块率高造成“归档不可用”假象。
验证者滞后攻击：对手针对单个验证者制造高延迟，使其 DAG 视图落后，从而加大其获取历史数据的难度，可能错过重要证据。
潜在后果
这些模式可能导致：
修复失败：大量丢失依赖无法及时修复，整个网络的状态同步被拖住。
节点被排除：处于滞后状态的节点无法及时产生证据，可能被误判为作恶节点而被孤立。
「已提交但无法执行」：协商的序列中含有缺失数据，对应事务虽然被签发证书但执行节点无法执行，从而在共识层面造成不一致。
归档/回放缺口：历史数据缺失可能导致无法完整回放、审计或滚回链，影响链的可验证性。
审计困难：证据不齐备意味着无法检验各节点是否全程诚实执行协议，问责能力受损。
可用性与活跃性降低：系统频繁等待数据修复，整体出块/共识效率下降。
存储开销增加：如果放弃分层存储策略而保留所有热点证据，则系统需要更大的快速存储投入。
需要注意，这些影响主要偏向一致性可用性层面，而非直接导致代币盗窃等安全破坏。但也可能间接影响共识终结性（如误认为某些节点不在线而触发新委员会），或触发降级保护（引入中央化成分）。
与传统拒绝服务的区别
与传统 DoS（如简单网络洪泛）不同，证据存储攻击利用了协议层面和存储语义的不匹配：它不依赖大量资源消耗，而是利用系统预期的数据可用性和修复流程。例如，对手并非发送异常多的消息，而是发送合法的摘要引用并选择性隐匿载荷，迫使节点正常遵循协议中的修复步骤，从而在合法行为范围内拖垮系统。这种攻击更像是对协议“信任边界”的探索，而非纯粹的流量洪泛。它利用了系统对因果依赖满足的假定：协议假定一旦共识达成，相关数据最终会到达；攻击则是违反这一定义，造成后端链路异常。
安全属性定义
针对上述风险，我们可以考虑定义诸如**“因果可证存储安全”的属性，即保证对任何被共识确认的事务，其所有因果依赖都能在有限延迟内通过原始数据或可验证替代物恢复。如：“任一提交事务的所有先验父节点和执行依赖，要么可由活跃节点原始提供，要么由有效检查点/归档覆盖证实。” 或称 “证据可恢复性”（evidence recoverability）：确保在拜占庭少数限制内，任何必要证据最终可取回，不会因网络或惰性而永久丢失。类似地，可以定义“修复完备性”**：给定数据可用性的合理延迟界限，任何缺失依赖必能被及时补全。总体目标是使得节点即使在部分数据缺失时，也能保证最终能利用替代物无损地恢复完整链状态。
相关工作
已有研究多集中于数据可用性攻击与存储压力问题：
数据可用性攻击（Data Availability Attack）：Celestia 定义的“数据隐匿攻击”说明了在区块提议时故意隐藏交易数据可对全节点造成影响。DAG-BFT 的问题类似于对手在分布式 DAG 协议中隐藏或延迟发布批数据。
存储拒绝服务（Storage DoS）：少有直接文献，但一些区块链设计讨论了“状态膨胀”（state bloat）和“存储压力”，例如讨论在持久化存储下保存所有旧状态的成本。
修剪安全性（Pruning Safety）：BFT 协议如 Tendermint/PBFT 等有检查点委托模型，关于何时安全地删除旧日志的工作更靠近系统设计（比如对 PBFT 检查点的论文），而针对 DAG-BFT 的专门分析尚未见到。
共识可用性在资源受限下：一些论文讨论了在链增长和网络延迟下的 liveness 保证，例如部分同步假设下的最优延迟（参考 Mysticeti 等工作）。但这些多关注消息复杂度和延迟界限，而非存储层面的可用性。
交易池 DoS：Mempool 抗垃圾攻击策略已有研究（过滤、门槛等），但与存储无关。
归档节点失效：业界讨论了归档节点（Archive Node）在各链中的作用，例如 Chainstack 指出 Sui 需归档节点支持；Celestia 建议用档案节点保存超出采样窗口的数据。但这些一般指导实践，不涉及恶意激发。
总体而言，目前少有针对基于 DAG 的 BFT 协议中因果证据保存的正式安全研究。现有工作分散在数据可用性、存储优化和共识设计领域，但没有直接提出需要因果依赖跟踪的安全属性。

Causal Evidence Indexing and Retention for Storage-Backed DAG-BFT
Critical Evaluation of the Hypothesis and State of the Art
The architectural evolution of Directed Acyclic Graph Byzantine Fault Tolerant (DAG-BFT) systems represents a fundamental shift in how decentralized networks achieve high throughput and low latency. Early consensus mechanisms coupled the logical ordering of transactions tightly with their physical execution and materialization, creating severe bottlenecks. Modern DAG-BFT systems, including Narwhal, Tusk, Bullshark, and Mysticeti, achieve unprecedented scalability—often exceeding 200,000 transactions per second—by explicitly decoupling these processes. In these architectures, validators advance consensus rapidly through the exchange of lightweight metadata, such as headers, digests, and parent references, while the heavy lifting of processing payload bodies, executing smart contracts, and repairing sub-DAG dependencies is pushed to asynchronous, background pipelines.
The hypothesis presented—that this decoupling creates a hot-path storage problem where causal evidence cannot be safely managed as flat key-value history using purely round, epoch, or checkpoint-based pruning—is not only structurally sound but addresses a critical, underexplored vulnerability in modern blockchain engineering. A rigorous analysis of recent production systems validates this premise definitively. The most compelling evidence supporting this hypothesis is the 2026 safety vulnerability discovered in the Mysticeti-FPC (Fast Path Commit) protocol deployed on the Sui network.
In this specific deployment, the protocol designers attempted to manage memory and storage through two distinct, desynchronized garbage collection (GC) procedures. The first, DAG-GC, inherited from the base Mysticeti-C protocol, pruned committed blocks from the local DAG based strictly on advancing consensus rounds. The second, Vote-GC, was responsible for pruning the transaction finalization caches and descendant graphs, but only operated after sub-DAGs were fully finalized via execution. Because transaction execution and finalization naturally lag behind the lightning-fast block commit rate of the DAG, a functional desynchronization emerged. Under asynchronous network conditions lasting merely 12 seconds, the round-based DAG-GC advanced ahead of the Vote-GC. This caused the system to prematurely delete blocks containing explicit invalid votes before the fast-path tallying logic could process them. In Mysticeti-FPC, valid votes are implicit; thus, the premature erasure of invalid causal evidence led the system to misinterpret their absence as implicit valid votes, creating an over-optimistic tally. The ultimate result was a catastrophic safety violation: honest validators diverged on the ledger state, enabling double-spending attacks. This real-world failure directly justifies the core claim of the hypothesis: execution-oblivious pruning in decoupled DAG-BFT systems fundamentally breaks consensus safety.
However, while the hypothesis correctly identifies the vulnerability, it is slightly overbroad in its implied categorization of potential solutions. It correctly asserts that safe retention requires knowing whether an object is a causal dependency or an active protocol root, but it must strictly distinguish this requirement from the capabilities of generic graph databases. Applying dynamic graph database reachability indexes, such as GRAIL or GRIPP, is mathematically unviable for the hot path of a high-throughput blockchain. Dynamic reachability queries in graph databases, particularly under edge deletion scenarios (which simulate garbage collection), exhibit severe update penalties. For instance, maintaining strongly connected components or interval labels typically incurs update costs ranging from $\mathcal{O}(n)$ to $\mathcal{O}(n \log n)$, or even $\mathcal{O}(n + m)$. Techniques like DAGGER, which rely on relaxed interval labeling on equivalent DAGs, face extremely expensive re-computations when a deleted edge causes a connected component to split. Imposing these $\mathcal{O}(n)$ penalties on a storage engine attempting to sustain hundreds of thousands of state transitions per second would immediately destroy the throughput advantages of the DAG-BFT architecture.
Furthermore, the hypothesis must distinguish between full causal reachability and the much narrower concept of monotonic consumption tracking. A DAG-BFT node does not need to answer arbitrary queries about whether Node A historically referenced Node B. It only needs to know a binary condition: has the "digestion frontier" of all asynchronous protocols (including execution, repair, and archiving) safely passed a specific causal root?
This distinction becomes paramount when examining the state of the art in blockchain storage engines. Modern storage engines optimized for validators, such as Mysten Labs' Tidehunter, are engineered specifically to eliminate the severe write amplification—often 10x to 30x—caused by Log-Structured Merge-Tree (LSM-tree) compaction in databases like RocksDB or Erigon's MDBX. Tidehunter achieves 830K writes per second by treating the Write-Ahead Log (WAL) as permanent storage and bypassing compaction entirely, relying instead on lazily flushed indexes and epoch-based pruning to reclaim space without blocking writes. Yet, even this highly optimized engine relies on epoch or round-based pruning heuristics, rendering it structurally oblivious to the granular causal dependencies that trigger vulnerabilities like the Mysticeti-FPC desynchronization.
Similarly, in the Ethereum ecosystem, the Erigon client utilizes MDBX and experimental FlatTrie architectures to reduce archive storage from 35TB to 10TB. However, Erigon's pruning mechanisms—such as state migration and batch pruning—rely on deleting historical data based on fixed block distance heuristics (e.g., --prune.htc for 90,000 blocks). This flat, linear approach to pruning is entirely incompatible with the multi-dimensional, concurrent nature of DAGs, where blocks from the same round may execute at vastly different speeds due to payload complexity or object contention.
Other systems address state pruning effectively but fail to address transient consensus evidence. The Jellyfish Merkle Tree (JMT), used in systems like Diem and Aptos, utilizes version-based node keys and explicitly tracks "stale node indices" to identify orphaned nodes that can be safely pruned from the state tree without breaking cryptographic proofs. While JMT brilliantly solves execution-layer state pruning, it operates strictly post-execution. It offers no mechanism to index or protect the transient consensus evidence (uncommitted DAG paths, raw payload bodies, explicit invalid votes) that reside in the hot-path storage prior to state commitment.
Finally, Data Availability (DA) networks, such as Celestia, approach data retention through entirely execution-agnostic lenses. Celestia's recent upgrades (CIP-34 and CIP-36) reduced the trusting period and minimum data pruning window to 7 days plus a 1-hour buffer. This rolling time window is highly effective for light node sampling and broad namespace availability, but it fundamentally ignores whether the data has been consumed. DA nodes will drop data when the TTL expires, regardless of whether a delayed rollup execution pipeline still requires that data to compute its state root. For a primary DAG-BFT validator, relying on time-based TTLs guarantees liveness failures during prolonged network degradation.
The synthesis of these findings reveals a crucial intersection that defines the proposed research direction. DAG-BFT systems require the raw I/O performance of append-only, compaction-free storage (like Tidehunter) paired with a highly specialized, ultra-low-overhead index that tracks the asynchronous consumption of causal evidence rather than simple block age or static graph reachability. The hypothesis is strong, but it must be refined from a generic "graph indexing" problem into a targeted "monotonic consumption tracking" problem.

Finally, authenticated data structures like the Jellyfish Merkle Tree offer execution awareness through orphaned node identification, but their scope is strictly limited to post-execution state, offering no protection for the transient consensus objects that formulate the DAG itself.
Identification of the Strongest Research Gap
The most significant and actionable research gap identified through this synthesis is the fundamental conflict between Execution-Oblivious Consensus Garbage Collection and Asynchronous Causal Materialization.
Current high-throughput DAG-BFT architectures push the boundaries of performance by splitting the protocol into multiple asynchronous pipelines. There is a worker layer for data dissemination, a primary DAG for metadata ordering, a fast-path for immediate transaction finality, and a state machine for execution. However, despite this multi-dimensional processing architecture, the underlying storage engine relies on a one-dimensional threshold for lifecycle management—specifically the consensus round or the epoch. When a configuration like consensus_gc_depth = 60 is applied, the storage engine blindly purges any object older than 60 rounds from the hot path.
This creates a severe vulnerability under conditions of network asynchrony. Straggler nodes, delayed fast-path pipelines, or background accountability modules may still require historical causal evidence to validate a vote, complete an equivocation proof, or finalize a long-running transaction sequence. Because the storage engine only monitors the advancing consensus round, it deletes the required historical objects before the asynchronous pipelines have physically consumed them.
The research gap is the absence of a lightweight, storage-engine-native dependency index that synchronizes the retention of physical evidence with these multi-dimensional, asynchronous consumption protocols. It must achieve this synchronization without incurring the fatal $\mathcal{O}(n)$ update penalties of traditional graph databases. This gap is highly specific—it can be isolated entirely within the storage and garbage collection logic of a DAG-BFT validator—yet it is broad enough to address critical safety failures, liveness stalls, and unbounded storage bloat across multiple protocols, making it an ideal foundation for academic publication.

Formal Definition of Core Abstractions
To address this problem systematically, the ad-hoc terminology surrounding DAGs and storage must be unified into a formal set of core abstractions. These definitions provide the theoretical framework required to build the proposed indexes.
The lifecycle of data in a DAG-BFT system begins with Evidence, defined as the physical byte representation of a protocol object. This encompasses payload bodies, block headers, explicit invalid vote declarations, and cross-epoch certificates. Evidence is the raw material that must be protected. The logical structure is anchored by a Causal Root, which is a topologically terminal node in the local DAG view that establishes a cryptographic ordering over a specific subgraph of evidence.
Because modern DAG-BFT systems decouple execution from ordering, the presence of a Causal Root does not imply that its subgraph is finalized. Therefore, we introduce the Active Root Record (ARR). The ARR is a localized metadata structure bound to a Causal Root that maintains a bitmap or counter representing the asynchronous protocol paths (e.g., execution engine, vote tallying, archive handoff) that have yet to consume the subgraph. The ARR is the primary vehicle for tracking pending obligations.
The physical lifecycle of evidence relies on two discrete events. Materialization is the transition of an object from a logical reference (a cryptographic hash embedded in a parent link) to a physically accessible, verified byte sequence resting in the local storage engine (e.g., Tidehunter's WAL). Consumption is the subsequent event where an asynchronous subsystem reads the materialized evidence, processes it, advances its internal state machine, and explicitly signals to the storage engine that its dependency on the object has been extinguished.
As evidence ages, retaining raw bytes indefinitely is impossible. Substitute Coverage defines a cryptographic state transition—such as a Jellyfish Merkle Tree state root or an authenticated checkpoint summary—that logically supersedes the raw historical evidence. Once substitute coverage is generated and verified, the raw evidence transitions from being consensus-critical to merely historically interesting, allowing it to be delegated to archive storage.
To govern the retention of evidence holistically, we define the Evidence Digestion Frontier (EDF). The EDF is a multi-vector logical clock tracking the lowest consumption watermarks across all asynchronous pipelines. It represents the strict lower bound of active causal obligations. Finally, Retention Safety is the formal invariant the system must uphold: for any piece of evidence $E$, if $E$ is reachable by an Active Root Record $ARR$, or if $E$ is required to mathematically advance the $EDF$, $E$ cannot be evicted from local hot-path storage, regardless of its age, epoch, or consensus round.

Proposed Retention Policies for Comparison
Evaluating the efficacy of the proposed causal-aware architecture requires benchmarking it rigorously against the prevailing industry standards. The analysis will compare three distinct retention policies.
1. Baseline: Round/Checkpoint-Only Retention (The "Sui / Mysticeti" Model) This baseline relies on hardcoded spatial thresholds. As seen in Sui's architecture, the configuration consensus_gc_depth determines the exact number of rounds to retain in the hot path. When the consensus leader commits round $R$, the storage engine unconditionally prunes all DAG blocks and associated evidence from round $R - 60$ and older. Hypothesized Behavior: Under synchronous, ideal network conditions, this policy achieves maximum storage efficiency with perfectly predictable disk usage. However, under network partitions or targeted Denial of Service attacks that exceed the temporal equivalent of the round depth, this policy will trigger severe safety and liveness failures. Execution pipelines will stall as they attempt to fetch payloads that have been cleanly deleted by the GC, and fast-path voting tallies will break as invalid votes are purged.
2. Baseline: Age/LRU Tiering (The "Celestia / Cloud" Model) This baseline represents the Data Availability and cloud storage approach, retaining evidence based on physical ingestion time or Least Recently Used (LRU) semantics. Similar to Celestia's network upgrades, data is retained for a rolling window, such as a 7-day trusting period, after which it is pruned or evicted to cold storage. Hypothesized Behavior: This policy is highly suboptimal for the hot path of a DAG-BFT validator. Time elapsed correlates poorly with protocol consumption. A straggling node requiring 8-day-old repair data to process an intensely complex smart contract state transition will stall indefinitely because the DA policy purged the data. Furthermore, LRU metrics will thrash unpredictably due to background state sync processes pulling old data into the hot cache, causing premature eviction of critical new consensus evidence.
3. Proposed: Causal-Aware Retention
The proposed policy abandons rounds and time in favor of causal determinism. Eviction is gated strictly by the boolean intersection of the Digestion Frontier Index ($DFI$) and the Pending Consumer Reference Table ($PCRT$). Evidence is retained until it is sequentially bypassed by the lowest watermark of all consumption pipelines and its reference count in the PCRT reaches zero.
Hypothesized Behavior: Under normal synchronous operations, this policy mimics the bounded memory footprint of the round-based baseline. The storage overhead is identical because consumption closely tracks ordering. However, under asynchronous conditions or targeted partitions, the storage footprint will dynamically and elastically expand to absorb the "causal overhang." The system gracefully degrades into a large straggler-tolerant buffer, preserving perfect safety. Once the partition resolves and consumption resumes, the SCM triggers a bulk eviction, rapidly returning the node to its baseline memory footprint.
Formal Definition of Failure Modes
If a storage-backed DAG-BFT system fails to index and retain causal evidence correctly, it will encounter distinct, theoretically verifiable failure modes. Each mode represents a different vector of protocol degradation.
1. GC-Before-Repair (Liveness Break) This occurs when a straggling validator attempts to request missing parent evidence to complete a local sub-DAG view, but the honest majority has already advanced the consensus round and executed round-based GC. The straggler can observe the cryptographic hashes but cannot acquire the physical payload bytes required to validate the parent links. The result is a permanent liveness stall for that node; it can never participate in consensus again without an out-of-band, full-state synchronization procedure.
2. Committed-but-Unexecutable (Liveness and Safety Degradation) In heavily decoupled architectures, a block may be logically committed via consensus, while its underlying transaction payloads are routed to separate worker storage. If the worker's storage engine utilizes an aggressive TTL or LRU policy, the payload may be purged before the execution pipeline schedules its materialization. The entire network may agree that transaction $X$ occurred at index $Y$, but because the physical payload is lost, the state machine cannot compute the transition.
3. Fast-Path Vote Desynchronization (Safety Break) This is the precise mechanism responsible for the Mysticeti-FPC vulnerability. Asynchronous fast-path finalization protocols frequently optimize performance by relying on implicit votes—deducting explicit invalid votes from a total assumed valid set. If the storage engine prunes a block containing an explicit invalid vote before the transaction finalization logic computes the tally, the system registers a false positive consensus. This leads immediately to diverging state roots among honest validators and the realization of double-spending attacks.
4. Accountability Evidence Degradation (Byzantine Enabling)
To slash a Byzantine validator for equivocation or malicious behavior, honest nodes must generate a cryptographic proof containing conflicting signatures. If the evidence of the first malicious signature is aggressively pruned due to rapid epoch advancement before the equivocation is fully detected and broadcast across the network, the protocol loses its deterrent mechanism. The Byzantine actor can operate without economic penalty, breaking the game-theoretic assumptions of Proof-of-Stake systems.
5. Archive Handoff Gap (Data Availability Failure) This failure emerges when hot-path storage prunes data under the assumption that an archive node or DA layer has successfully captured it. If the archive ingestion pipeline is delayed due to I/O bottlenecks or network latency, a temporal blind spot is created. Historical evidence is permanently lost between the hot path and cold storage, violating data availability guarantees and severely hindering off-chain analytics, block explorers, and tax-reporting infrastructure.
6. High-Fanout Ancestor Loss (Performance Collapse)
In DAG architectures, certain highly connected ancestor nodes provide crucial routing for reachability proofs and historical verification. If a storage index drops a high-fanout node due to age-based eviction without recognizing its high degree of pending causal references (which a PCRT would detect), any subsequent graph traversal initiated by a late-arriving node will incur massive network re-fetch penalties. This manifests as sudden, unexplained spikes in bandwidth and CPU load across the validator network.
Comprehensive Evaluation Plan
To empirically validate the causal-aware storage architecture, a rigorous systems evaluation plan must be executed.
Prototype Scope: The prototype will instrument an existing, open-source production-grade DAG-BFT node, such as Sui's implementation of Mysticeti. The node will be backed by an append-only, compaction-free storage engine modeling Tidehunter's design. We will modify the engine to intercept all read/write paths, implementing the in-memory DFI, PCRT, and SCM indexes while disabling the hardcoded consensus_gc_depth configurations entirely.
Workloads:
Sustained Maximum Throughput: Simulating 200,000+ TPS with uniform 1KB payload sizes to benchmark the raw I/O overhead of the index structures against the baseline Tidehunter performance.
Highly Skewed DAG Topologies: Inducing deep, asymmetric sub-DAGs by manipulating network latency among virtual nodes to stress the reference counting (PCRT) mechanisms and force deep causal overhangs.
Fast-Path Intensive Workloads: Generating workloads dominated by owned-object transactions that require rapid vote tallying but delayed execution, directly stressing the vulnerability surface identified in Mysticeti-FPC.
Delay and Backpressure Injection:
To simulate the adversarial and asynchronous conditions that trigger the identified failure modes, specific injections are required:
Execution Throttling: Artificially delay the CPU cycles allocated to the execution pipeline, creating a rapidly growing delta between the committed DAG round and the physical state root.
Network Partitioning: Isolate a subset of $f$ nodes for 15 to 45 seconds. This forces them into a straggler state to test GC-before-repair and fast-path desynchronization under realistic partition conditions.
Byzantine Equivocation Spam: Inject deliberate double-votes from simulated malicious nodes, delayed slightly by network latency, to stress the retention of accountability evidence against a rapidly advancing epoch counter.
Metrics to Capture:
Write Amplification and IOPS: Measuring the exact overhead of atomic PCRT increments against flat WAL writes.
Causal Overhang Size (MB): Tracking the physical memory and disk space required to sustain retention safety during artificial partitions.
Time-to-Recovery: The duration required for a partition-isolated straggler to re-sync and resume consensus participation.
Safety Violation Count: Automated monitoring for state root divergence or successful double spends across the test cluster.
Expected Results:
The causal-aware system is expected to exhibit negligible throughput degradation (less than 5% impact on sustained IOPS) under normal synchronous operations compared to the raw baseline. Under network partition conditions, the baseline configurations will experience quantifiable safety violations (state divergence) and permanent liveness stalls. In contrast, the causal-aware system will dynamically expand its storage footprint to absorb the causal overhang, maintaining strict safety properties and recovering immediately once the network partition is resolved.
What NOT to Implement: The evaluation will not implement a full, generalized graph database backend (e.g., Neo4j or GRAIL) , as their $\mathcal{O}(n)$ performance failure is already established in literature. Furthermore, we will not implement content-addressed erasure coding networks mirroring Celestia's DA layer. The focus must remain strictly bounded to local, node-level index optimization for physical DAG-BFT retention.
Top 12 Reviewer Objections and Honest Responses
Anticipating the critique of systems and security reviewers is critical for shaping the narrative. The following represents the most likely objections, including an acknowledgment of their valid components and the strategic academic rebuttal based on current systems data.
Objection 1: "This is just distributed garbage collection, which was extensively solved in the 1990s."Response: Valid in theory, invalid in current blockchain practice. While reference counting and tracing GC are classical computer science concepts, modern DAG-BFT systems explicitly avoid tracing GC because arbitrary graph traversals break constant-time guarantees on the critical execution path. The contribution of this research is not the invention of garbage collection, but the specific abstraction of adapting causal tracking to $\mathcal{O}(1)$ updates over flat, append-only WALs without requiring the read-modify-write loops that plague traditional implementations.
Objection 2: "If network asynchrony persists indefinitely, your causal-aware retention will lead to Out-Of-Memory (OOM) crashes, whereas round-based GC prevents node death by dropping data."Response: This is a highly valid objection. Bounded memory is a core liveness requirement for any Byzantine system. However, crashing a node via OOM is strictly preferable to silently breaking consensus safety and allowing a double spend to process. To mitigate the OOM risk, our protocol includes an "Emergency Substitute Coverage" mechanism. If the causal overhang hits a critical, predefined memory threshold (e.g., 90% RAM utilization), the system automatically triggers a localized, synchronous state snapshot to truncate the DAG. This explicitly trades temporary availability (halting consensus participation) to preserve long-term safety and bounded memory.
Objection 3: "Fast-path desynchronization (like the Mysticeti-FPC bug) is merely a protocol implementation bug, not a fundamental storage problem."Response: This argument mischaracterizes the trajectory of modern decoupled systems. The fundamental design of high-throughput blockchains pushes computation entirely out of the consensus hot-path. As long as execution, payload materialization, and voting remain asynchronous to logical ordering, the logical lifecycle of an object will always be decoupled from its physical consumption. Pushing the burden of tracking this multi-dimensional lifecycle onto the application layer leads to brittle, error-prone code. A causally-aware storage engine solves this fundamentally at the data persistence layer, eliminating an entire class of implementation bugs.
Objection 4: "Why not just use an LSM tree with custom compaction filters, like RocksDB or Erigon's MDBX?"Response: High write amplification. As demonstrated extensively by the development of Tidehunter, LSM compaction under uniformly distributed cryptographic keys incurs 10x to 30x write amplification, heavily bottlenecking modern NVMe drives during peak blockchain loads. Furthermore, LSM compaction filters operate strictly on linear key values or age-based TTLs. They lack the architectural context to query a DAG dependency structure safely without locking the database.
Objection 5: "Maintaining reference counts (via the PCRT) on every block ingestion will destroy the system's write throughput."Response: This is a valid concern. If implemented naively with synchronous disk writes for every parent reference, the I/O overhead would be fatal. Our architecture mitigates this by maintaining the PCRT exclusively in memory. The active causal overhang (the delta between logical ordering and physical execution) is typically very small in temporal terms—usually seconds to minutes. Only Substitute Coverage Maps (the final checkpoints) and the flat, append-only WAL payloads are flushed to persistent disk, insulating the I/O layer from the indexing overhead.
Objection 6: "General DAG reachability indexes like GRAIL or GRIPP already exist. Why invent new indexes?"Response: Generic reachability indexes are designed to support arbitrary edge insertion and deletion, resulting in update complexities of $\mathcal{O}(n + m)$ or $\mathcal{O}(n)$ depending on the algorithm. Blockchain DAGs, however, are strictly append-only. We do not need to query "Is Node A reachable from Node B?" at random across historical data. We only need to compute the trailing, monotonic digestion frontier. The DFI and PCRT are $\mathcal{O}(1)$ specialized structures that exploit the monotonically advancing nature of BFT protocols to achieve speeds impossible in general-purpose graph databases.
Objection 7: "This framing confuses Data Availability (DA) with local state storage."Response: DA networks like Celestia focus on whether data was successfully published to the network, utilizing a rolling window (e.g., a 7-day trusting period) for light node sampling. They do not care if the data was actually executed. Primary BFT validators, conversely, must maintain state machine replication. If a validator drops data because a time window expired, but execution was stalled due to high CPU load, the validator can never compute the resulting state root. The report explicitly distinguishes execution-oblivious DA retention from execution-aware hot-path retention.
Objection 8: "The storage overhead of keeping invalid votes or orphaned DAG paths is trivial. Just buy larger NVMe SSDs."Response: Hardware scalability cannot solve asymptotic data bloat. While raw storage media is cheap, unbounded state growth leads to severe secondary systemic failures, such as unmanageable node bootstrapping times and excessive I/O latencies during state sync processes. As seen in Erigon's multi-terabyte archive nodes, pruning is strictly necessary for operational viability ; therefore, the pruning mechanism must be mathematically safe under all fault conditions.
Objection 9: "Byzantine nodes could deliberately spam invalid votes or un-executable payloads to bloat the causal overhang and perform resource exhaustion attacks."Response: Valid attack vector. If retention is strictly bound to execution, a payload that intentionally halts or infinite-loops the execution engine could prevent the PCRT from ever decrementing, freezing garbage collection. This is mitigated by integrating gas-based bounded execution limits into the DFI. If a block exceeds its execution budget or time allocation, it is marked as "consumed by failure." The PCRT is decremented, and the block is purged, neutralizing the resource exhaustion attack.
Objection 10: "Substitute Coverage relies on state checkpoints. If checkpoint generation stalls, your entire retention mechanism stalls."
Response: This is correct. However, checkpoint generation is intrinsically tied to the advancement of consensus itself. If consensus stalls completely (e.g., due to a massive network failure lacking a $2f+1$ quorum), there are no new blocks being added to the DAG anyway. Therefore, hot-path storage growth naturally halts in tandem with the checkpoint stall. The system achieves a stable equilibrium without risking safety.
Objection 11: "Erigon already does block history pruning successfully without a complex DAG index."Response: Erigon operates on a linear blockchain (Ethereum), not a DAG. In a linear chain, block $N$ implicitly subsumes all history before $N$. In a DAG architecture, block A and block B can be committed simultaneously in the same consensus round but executed at entirely different speeds depending on payload complexity or object contention. The linear pruning logic of Erigon (e.g., batch state migration ) cannot map onto the concurrent, multi-dimensional execution state of a DAG.
Objection 12: "Your evaluation plan relies heavily on synthetic workloads. Real-world smart contract execution patterns are highly unpredictable and skewed."
Response: This is a valid critique of baseline systems research. While synthetic uniform payloads are necessary to isolate pure I/O metrics and establish a theoretical ceiling, they fail to capture state contention. The evaluation will be expanded to replay historical transaction traces from a production network (e.g., Sui Mainnet or Aptos) to measure the actual DFI and PCRT memory overhead under highly skewed, realistic state-access distributions and smart contract execution latencies.
